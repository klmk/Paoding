# Phase 2: 规划设计

## 概述

Phase 2 是连接需求与实现的桥梁。其核心使命是将 Phase 1 产出的 PRD（需求规格文档）转化为两个关键成果：**技术架构**（决定"怎么构建"）和**开发计划**（决定"怎么执行"）。

**核心理念：** 没有经过设计评审的架构不值得实现，没有经过自审的计划不值得执行。本阶段的核心价值不在于产出文档的厚度，而在于思考的深度和决策的严谨性。

**适用场景：**
- Phase 1 PRD 已通过需求对齐确认，准备进入技术设计
- 项目技术方向需要明确选型和架构决策
- 需要将需求拆解为可执行、可验证的开发任务
- 任何涉及架构变更或重大功能新增的场景

---

## 阶段目标

| 目标 | 描述 | 验证方式 |
|------|------|---------|
| 技术选型完成 | 基于PRD约束和使用者技术栈偏好，确定前端/后端/数据库/部署方案 | 技术选型报告，含候选方案对比和选择理由 |
| 架构设计完成 | 系统分层、模块划分、接口定义清晰无歧义 | 架构文档，含架构图、模块说明、接口定义 |
| 数据设计完成 | 数据模型、存储方案、数据流向明确 | 数据设计文档，含ER图、字段定义、存储方案 |
| 开发计划完成 | 任务拆解粒度合适、无占位符、每个任务可验证 | 开发计划文档，含任务列表、依赖关系、验收标准 |
| 设计评审通过 | 使用者确认架构和计划满足需求且可执行 | 设计评审记录，含评审意见和确认签名 |

## 入口条件

在进入 Phase 2 之前，以下条件**必须全部满足**：

- [ ] Phase 1 的全部出口条件已满足（Gate 1 通过）
- [ ] PRD（需求规格文档）已获得使用者显式确认
- [ ] 每条核心需求有唯一标识、描述、验收标准
- [ ] 需求优先级已与使用者确认
- [ ] 使用者画像和项目画像已存在（Phase 0 产出）

**入口检查：** AI 在进入本阶段前，必须逐一验证上述条件。任何一条不满足，必须回退到对应阶段修复。

## 铁律（Iron Laws）

### 铁律 1：设计未批准不得写代码

```
设计评审未通过前，禁止任何形式的代码实现。
包括但不限于：编写生产代码、搭建项目骨架、配置开发环境中的业务逻辑。
```

设计评审是 Phase 2 的硬性门控。未通过评审就写代码，等同于没有图纸就施工——结果必然是返工。

### 铁律 2：计划不允许占位符

```
开发计划中的每个任务描述必须具体、明确、可执行。
禁止使用以下占位符表述：
- "TODO"、"待定"、"后续补充"
- "根据实际情况调整"
- "适当处理"、"合理设计"
- 任何用省略号代替具体内容的描述
```

占位符是计划的毒药。一个包含占位符的计划，本质上是一个未完成的计划。

### 铁律 3：每个任务必须可验证

```
开发计划中的每个任务必须包含：
1. 具体的操作描述（做什么）
2. 涉及的文件路径（在哪里做）
3. 预期的输出结果（做成什么样）
4. 对应的测试命令（怎么验证）
```

不可验证的任务等于不可完成的任务。如果无法定义"完成"的标准，说明任务本身还需要进一步拆解。

### 铁律 4：架构必须覆盖全部核心需求

```
架构设计必须逐条覆盖PRD中的每一条核心需求。
任何一条核心需求在架构中找不到对应的实现方案，架构即为不完整。
```

架构是需求的映射。遗漏需求的架构，等于埋下了返工的种子。

### 铁律 5：技术选型必须有理由

```
每一项技术选择都必须记录：
1. 候选方案（至少2个）
2. 评估标准
3. 选择理由
4. 潜在风险
```

"熟悉"不是选型理由，"流行"也不是。选型必须基于需求特征和客观评估。

### 铁律 6：设计变更必须追溯

```
Phase 2 中的任何设计变更必须：
1. 记录变更原因
2. 评估变更影响（涉及哪些模块、哪些任务）
3. 更新所有受影响的文档
4. 通知使用者确认
```

设计变更不追溯，等于在架构中埋下不一致的隐患。

## 执行流程

### 步骤 1：技术选型

**目标：** 基于 PRD 约束和使用者技术栈偏好，确定项目的技术方案。

**执行要点：**

1. **提取选型约束**：从 PRD 中提取技术相关约束（性能要求、部署环境、兼容性要求等）
2. **获取使用者偏好**：参考 Phase 0 使用者画像中的技术栈偏好和经验背景
3. **候选方案评估**：每个技术维度至少评估 2-3 个候选方案
4. **多维对比**：从以下维度进行对比评估
   - 功能匹配度（是否满足PRD需求）
   - 性能表现（是否满足非功能性需求）
   - 学习曲线（使用者的上手成本）
   - 生态成熟度（社区、文档、第三方库）
   - 可维护性（代码组织、调试便利性）
   - 部署复杂度（CI/CD、运维成本）
5. **记录决策**：使用架构决策记录（ADR）格式记录每个选型决策

**评估维度清单：**

| 维度 | 前端 | 后端 | 数据库 | 部署 |
|------|------|------|--------|------|
| 框架/语言 | | | | |
| 状态管理 | | | | |
| UI 组件库 | | ORM/查询 | | |
| 构建工具 | | 测试框架 | | |
| 包管理器 | | API 框架 | | |

**输出：** 技术选型报告（填入 `architecture-template.md` 的技术选型章节）

---

### 步骤 2：架构设计

**目标：** 设计系统分层、模块划分和接口定义。

**执行要点：**

1. **系统分层**：定义系统的层次结构（如表现层、业务层、数据层、基础设施层）
2. **模块划分**：基于 PRD 的功能需求，将系统拆分为独立的模块
   - 每个模块有明确的单一职责
   - 模块间通过定义良好的接口通信
   - 模块内部高内聚，模块之间低耦合
3. **接口定义**：定义模块间接口和外部接口
   - 接口契约：输入、输出、错误码
   - 接口版本策略
   - 接口文档规范
4. **架构图绘制**：使用文本格式（Mermaid 或 ASCII）绘制
   - 系统架构图（模块关系、数据流向）
   - 部署架构图（如需要）
   - 核心流程时序图（如需要）
5. **非功能性设计**：
   - 安全方案（认证、授权、数据加密）
   - 性能方案（缓存策略、负载均衡、数据库优化）
   - 可用性方案（容错、降级、监控）
   - 可扩展性方案（水平扩展、插件机制）

**输出：** 技术架构文档（填入 `architecture-template.md`）

---

### 步骤 3：数据设计

**目标：** 设计数据模型、存储方案和数据流向。

**执行要点：**

1. **实体识别**：从 PRD 中提取核心业务实体
2. **关系定义**：明确实体间的关系（一对一、一对多、多对多）
3. **字段定义**：为每个实体定义字段
   - 字段名称、类型、约束（必填/唯一/默认值）
   - 索引策略
   - 数据校验规则
4. **存储方案**：
   - 主数据库选型及理由
   - 缓存方案（如需要）
   - 文件存储方案（如需要）
   - 搜索方案（如需要）
5. **数据流向**：定义数据在系统中的流转路径
6. **数据迁移策略**：如需要，定义数据迁移方案

**输出：** 数据设计文档（填入 `architecture-template.md` 的数据模型章节）

---

### 步骤 4：开发计划拆解

**目标：** 将架构设计转化为可执行、可验证的开发任务列表。

**执行要点：**

1. **任务拆解原则**（参考 Superpowers Writing Plans 技能）：
   - **每步 2-5 分钟**：每个任务应能在 2-5 分钟内完成（指 AI 执行时间）
   - **一个动作**：每个任务只包含一个明确的动作
   - **无占位符**：任务描述中不允许有任何模糊或待定的内容
   - **可验证**：每个任务有明确的完成标准和验证方式
2. **任务粒度标准**：
   - 太粗（>10分钟）：继续拆分
   - 太细（<30秒）：合并到相邻任务
   - 合适（2-5分钟）：保持
3. **任务依赖分析**：
   - 标注每个任务的前置依赖
   - 识别关键路径
   - 标注可并行的任务
4. **风险标注**：
   - 标注技术风险点（如第三方依赖、性能瓶颈）
   - 标注不确定性（如需要使用者确认的决策点）
5. **验收标准定义**：
   - 每个任务有具体的预期输出
   - 每个任务有对应的测试命令
   - 整体计划有阶段性的验收里程碑

**任务描述规范：**

```
正确示例：
"在 src/modules/user/user.service.ts 中创建 UserService 类，
包含 createUser(email: string, password: string): Promise<User> 方法，
方法内部调用 userRepository.save() 保存用户并返回。
运行 npm test -- user.service.test.ts 验证，
预期输出：1 test passed。"

错误示例：
"实现用户模块"                    ← 太粗，不可验证
"创建用户相关的代码"              ← 太模糊
"根据需求实现用户注册功能"        ← 有占位符（"根据需求"）
"搭建后端基础架构"                ← 不可验证，没有具体输出
```

**输出：** 开发计划文档（填入 `plan-template.md`）

---

### 步骤 5：计划自审

**目标：** 在提交评审前，对架构和计划进行系统性自检。

**执行要点：**

1. **规格覆盖率检查**：
   - 逐条对照 PRD，确认每条核心需求在架构中有对应方案
   - 逐条对照 PRD，确认每条核心需求在计划中有对应任务
   - 标注任何未覆盖的需求，补充对应设计
2. **占位符扫描**：
   - 全文搜索以下关键词：TODO、待定、后续、TBD、适当、合理、根据实际情况
   - 发现任何占位符，必须替换为具体内容或标注为阻塞项
3. **依赖检查**：
   - 检查任务依赖关系是否有循环依赖
   - 检查关键路径上的任务是否合理
   - 检查是否有孤立任务（无依赖也无被依赖）
4. **一致性检查**：
   - 架构中的模块与计划中的任务是否对应
   - 接口定义与数据模型是否一致
   - 技术选型与架构设计是否匹配
5. **可行性检查**：
   - 每个任务是否在 AI 的能力范围内
   - 是否有需要使用者提供的信息或资源
   - 是否有外部依赖（第三方服务、API 密钥等）

**自审输出：** 自审报告，包含发现的问题和修复记录

---

### 步骤 6：设计评审

**目标：** 与使用者确认架构和计划，确保双方对技术方案和执行计划达成共识。

**执行要点：**

1. **评审准备**：
   - 整理评审材料：架构文档、开发计划、自审报告
   - 准备关键决策的背景说明
   - 标注需要使用者决策的开放问题
2. **评审执行**：
   - 使用 `design-review-checklist.md` 逐项评审
   - 重点向使用者说明：技术选型理由、架构设计思路、任务拆解逻辑
   - 对使用者提出的问题逐一回应
   - 记录使用者的意见和决策
3. **问题处理**：
   - 评审中发现的问题分为三级：
     - **阻塞问题**：必须修复后才能通过评审
     - **建议改进**：建议修复，但不阻塞评审
     - **记录备查**：记录但不影响当前评审
4. **评审结论**：
   - 通过：架构和计划满足需求，可以进入 Phase 3
   - 有条件通过：修复指定问题后即可进入 Phase 3
   - 不通过：重大问题需要重新设计，回到对应步骤

**输出：** 设计评审记录（含评审清单、问题清单、使用者确认）

## 红旗列表（Red Flags）

以下是 AI 在 Phase 2 中可能出现的"合理化借口"及对应反驳：

| # | AI 的借口（红旗） | 反驳 |
|---|------------------|------|
| 1 | "架构很简单，不需要画图" | 简单的架构更需要图。图的目的是消除歧义，不是展示复杂度。没有图意味着你对架构的理解还不够清晰。 |
| 2 | "这个技术栈我最熟悉，不需要评估其他方案" | 熟悉不等于合适。选型必须基于需求特征，而非个人偏好。至少评估 2 个候选方案是底线。 |
| 3 | "计划写得太细了，执行时灵活调整就行" | "灵活调整"是计划失败的常见原因。计划的价值在于提供明确的执行路径，而非留出模糊空间。 |
| 4 | "先搭个骨架写点代码验证一下设计" | 这违反了铁律 1。验证设计的方式是评审和推演，不是写代码。如果设计无法通过推演验证，说明设计本身有问题。 |
| 5 | "数据模型后面再细化，先把接口定好" | 接口定义依赖数据模型。没有数据模型的接口定义是空中楼阁。数据模型是架构设计的基础，不能延后。 |
| 6 | "这个需求在PRD里优先级低，架构里可以先不考虑" | 架构必须覆盖全部核心需求。低优先级不等于不需要设计。至少要有预留方案，避免后期架构大改。 |
| 7 | "使用者不是技术专家，评审时简单过一下就行" | 评审的目的不仅是技术审查，更是期望对齐。使用者需要理解技术方案的影响和取舍，而非技术细节。用通俗语言说明关键决策。 |
| 8 | "计划中的TODO后面执行时再补" | 这违反了铁律 2。计划中的占位符必须在提交评审前全部消除。如果信息不足，标注为阻塞项并明确获取信息的途径。 |
| 9 | "这个模块跟之前项目一样，直接复用之前的架构" | 相似不等于相同。必须验证当前需求与历史需求的差异，并记录具体的适配方案。 |
| 10 | "评审通过了就可以开始写代码了" | 评审通过只是进入 Phase 3 的条件。进入 Phase 3 后仍需遵循 TDD 流程（IL-2 测试铁律），先写测试再写代码。 |

**如果出现以上任何一种情况，立即停止，回到对应步骤重新执行。**

## 硬性门控（Hard Gate）

### Gate 2：设计评审门控

**门控规则：** 设计评审通过前，禁止任何实现动作。

**通过条件（必须全部满足）：**

- [ ] 技术选型报告已完成，每项选型有候选方案对比和选择理由
- [ ] 架构设计文档已完成，包含架构图、模块说明、接口定义
- [ ] 数据模型设计已完成，包含实体定义、关系、字段、存储方案
- [ ] 开发计划已完成，每个任务可执行、可验证、无占位符
- [ ] PRD 中的每条核心需求在架构中有对应方案
- [ ] PRD 中的每条核心需求在计划中有对应任务
- [ ] 计划自审已完成，发现的问题已全部修复
- [ ] 设计评审已完成，阻塞问题已全部解决
- [ ] 使用者已显式确认架构和计划

**违反门控的后果：**

1. 立即停止任何实现动作
2. 回退到 Phase 2 对应步骤
3. 完成缺失的设计工作
4. 重新通过设计评审后方可继续

## 与其他阶段的关系

```
Phase 0（双向画像）
    │
    │  提供使用者技术栈偏好、项目约束
    ▼
Phase 1（需求分析）── PRD ──→ Phase 2（规划设计）
                                │
                                │  输出：技术架构 + 开发计划
                                ▼
                          Phase 3（开发实施）
```

**Phase 2 对 Phase 0 的依赖：**
- 使用者画像中的技术栈偏好指导技术选型
- 项目画像中的约束条件限制架构选择
- 对齐策略中的风险点需要在设计中特别关注

**Phase 2 对 Phase 1 的依赖：**
- PRD 是架构设计的输入，每条需求必须有对应的技术方案
- 需求优先级指导开发计划的执行顺序
- 验收标准指导任务的验证方式

**Phase 2 对 Phase 3 的约束：**
- 架构文档是 Phase 3 编码的蓝图
- 开发计划是 Phase 3 执行的路线图
- 未在架构中定义的模块不得在 Phase 3 中实现
- 未在计划中列出的任务不得在 Phase 3 中执行（除非经过变更流程）

---

## 注意事项

- 架构设计是迭代的。在评审过程中可能需要多次修改，这是正常的
- 技术选型应优先考虑使用者的技术栈偏好，降低上手成本
- 开发计划的粒度应以 AI 可独立执行为标准，不依赖使用者的实时指导
- 如果在 Phase 2 过程中发现 PRD 有遗漏或歧义，必须回退到 Phase 1 修复，而非在设计中自行假设
- 所有架构决策必须使用 ADR（Architecture Decision Record）格式记录，确保决策的可追溯性
