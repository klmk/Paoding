# workflow-overview.md — 全流程总览

> **定位**：本文件定义 AI-Dev-Framework 从 Phase 0 到 Phase 5 的完整工作流，包括每个阶段的入口条件、核心活动、必输出物、出口条件及阶段间的门控规则。
> **适用范围**：所有使用本框架进行的软件开发项目。

---

## 流程全景图

```
Phase 0          Phase 1          AIS           Phase 2          Phase 3          Phase 4          Phase 5
双向画像  ──→  需求分析  ──→  实施规格  ──→  规划设计  ──→  开发实施  ──→  知识沉淀  ──→  交付标准
  │                │                │                │                │                │                │
  ▼                ▼                ▼                ▼                ▼                ▼                ▼
[Gate 0]         [Gate 1]       [Gate 1.5]      [Gate 2]         [Gate 3]         [Gate 4]         [Gate 5]
硬性门控         硬性门控         硬性门控         硬性门控         硬性门控         软性门控         硬性门控
```

**核心原则：** 每个阶段有明确的入口条件和出口条件。只有满足出口条件（通过门控检查）后，方可进入下一阶段。

---

## Phase 0：双向画像

### 目标

建立对用户和项目的双向理解，确保 AI 的行为与用户期望对齐。

### 入口条件

- 用户提出开发需求或项目意向。
- 尚未进行任何形式的需求分析或设计。

### 核心活动

| # | 活动 | 描述 |
|---|------|------|
| 0.1 | 用户画像构建 | 分析用户的技术背景、角色定位、沟通偏好、决策风格、风险承受度 |
| 0.2 | 项目画像构建 | 分析项目的业务领域、技术栈、规模预估、时间约束、质量要求 |
| 0.3 | 期望对齐 | 确认用户对交付物、沟通方式、质量标准的期望 |
| 0.4 | 约束识别 | 识别技术约束、业务约束、资源约束、时间约束 |

### 必输出物

| 输出物 | 格式 | 最低要求 |
|--------|------|---------|
| 用户画像 | 结构化文档 | 不少于 5 个关键维度 |
| 项目画像 | 结构化文档 | 不少于 5 个关键维度 |
| 期望对齐记录 | 文档 | 双方确认的期望清单 |
| 约束清单 | 文档 | 已识别的全部约束及影响评估 |

### 出口条件（Gate 0）

- [ ] 用户画像已构建并获得用户确认
- [ ] 项目画像已构建并获得用户确认
- [ ] 双方期望已对齐并有记录
- [ ] 已识别的约束已记录并评估影响

### 门控类型

**硬性门控（Hard Gate）** — 必须全部满足，不可绕过。

## Phase 1：需求分析

### 目标

将用户的模糊需求转化为结构化、可验证、可追溯的需求规格。

### 入口条件

- Phase 0 的全部出口条件已满足。
- 用户已确认画像和期望对齐。

### 核心活动

| # | 活动 | 描述 |
|---|------|------|
| 1.1 | 需求收集 | 通过结构化访谈、场景分析、原型讨论等方式收集需求 |
| 1.2 | 需求分类 | 将需求分为功能需求、非功能需求、约束条件 |
| 1.3 | 需求优先级排序 | 与用户协商确定需求的优先级（MoSCoW 或数值排序） |
| 1.4 | 验收标准定义 | 为每条需求定义明确的、可测试的验收标准 |
| 1.5 | 需求评审 | 与用户逐条确认需求，消除歧义 |

### 必输出物

| 输出物 | 格式 | 最低要求 |
|--------|------|---------|
| 需求规格文档 | 结构化文档 | 每条需求有唯一标识、描述、分类、优先级、验收标准 |
| 用户故事/场景 | 文档 | 覆盖核心用户场景 |
| 需求追溯矩阵 | 表格 | 需求与用户场景的映射关系 |
| 需求评审记录 | 文档 | 用户确认记录、变更记录 |

### 出口条件（Gate 1）

- [ ] 所有核心需求已收集并文档化
- [ ] 每条需求有明确的验收标准
- [ ] 需求优先级已与用户确认
- [ ] 需求追溯矩阵已建立
- [ ] 用户已确认需求规格文档

### 门控类型

**硬性门控（Hard Gate）** — 需求规格文档必须经用户签字确认（显式同意）。

## Phase AIS：AI 实施规格书

### 目标

将 PRD（自然语言）翻译为 AI 可精确执行的结构化规格书（AIS），消除 PRD 到代码之间的理解偏差。

### 入口条件

- Phase 1 的全部出口条件已满足（Gate 1 通过）
- PRD 已获得使用者显式确认

### 核心活动

| # | 活动 | 描述 |
|---|------|------|
| AIS.1 | AIS 自动生成 | AI 基于 PRD 自动生成 AIS，包含数据契约、业务规则表、状态机、异常处理矩阵、具体示例、环境约束 |
| AIS.2 | 产品经理审核 | 产品经理逐项审核 AIS，确认 AI 对 PRD 的理解是否正确 |
| AIS.3 | 理解偏差纠正 | 对 AIS 中与产品经理理解不一致的部分进行修正 |
| AIS.4 | AIS 确认 | 产品经理确认 AIS 后，AIS 作为 Phase 2/3 的直接输入 |

### 必输出物

| 输出物 | 格式 | 最低要求 |
|--------|------|---------|
| AIS 文档 | 结构化文档 | 使用 `ais-template.md` 模板，所有章节已填写 |
| AIS 审核记录 | 文档 | 产品经理确认记录、偏差纠正记录 |

### 出口条件（Gate 1.5）

- [ ] AIS 已基于 PRD 自动生成
- [ ] 产品经理已逐项审核并确认 AIS
- [ ] AI 对 PRD 的理解偏差已全部纠正
- [ ] AIS 与 PRD 的映射关系已建立

### 门控类型

**硬性门控（Hard Gate）** — AIS 必须经产品经理显式确认。

---

## Phase 2：规划设计

### 目标

基于需求规格，制定技术方案和实施计划。

### 入口条件

- Phase AIS 的全部出口条件已满足（Gate 1.5 通过）。
- AIS 已获产品经理确认。

### 核心活动

| # | 活动 | 描述 |
|---|------|------|
| 2.1 | 技术选型 | 评估候选技术方案，基于明确标准选择技术栈 |
| 2.2 | 架构设计 | 设计系统架构、模块划分、接口定义 |
| 2.3 | 数据模型设计 | 设计数据结构、存储方案、数据流 |
| 2.4 | 接口设计 | 定义模块间接口、外部接口、API 契约 |
| 2.5 | 实施计划制定 | 制定开发计划、里程碑、资源分配 |
| 2.6 | 风险评估 | 识别技术风险、进度风险、质量风险及应对措施 |
| 2.7 | 设计评审 | 对设计方案进行审查，确保满足需求且可行 |

### 必输出物

| 输出物 | 格式 | 最低要求 |
|--------|------|---------|
| 技术设计文档 | 结构化文档 | 架构图、模块说明、接口定义、数据模型 |
| 技术选型报告 | 文档 | 候选方案对比、选择理由 |
| 实施计划 | 文档/表格 | 里程碑、任务分解、时间估算 |
| 风险登记册 | 表格 | 已识别风险、影响评估、应对措施 |
| 设计评审记录 | 文档 | 评审意见、处理结果 |

### 出口条件（Gate 2）

- [ ] 技术设计文档覆盖全部核心需求
- [ ] 技术选型有明确的评估标准和选择理由
- [ ] 接口定义完整且无歧义
- [ ] 实施计划合理且经用户确认
- [ ] 已识别风险有应对措施
- [ ] 设计评审已完成且问题已解决

### 门控类型

**硬性门控（Hard Gate）** — 设计文档必须通过评审，实施计划必须获用户确认。

---

## Phase 3：开发实施

### 目标

按照设计文档和实施计划，严格遵循 TDD 流程完成编码实现。

### 入口条件

- Phase 2 的全部出口条件已满足。
- 技术设计文档已通过评审。
- 实施计划已获用户确认。

### 核心活动

| # | 活动 | 描述 |
|---|------|------|
| 3.1 | 任务分解 | 将实施计划分解为可执行的开发任务 |
| 3.2 | TDD 开发循环 | 对每个任务执行：红（写失败测试）→ 绿（写生产代码）→ 重构 |
| 3.3 | 代码审查 | 每个实现完成后进行自审和交叉审查 |
| 3.4 | 持续集成 | 频繁集成代码，确保构建和测试通过 |
| 3.5 | 进度跟踪 | 按里程碑跟踪进度，及时报告偏差 |
| 3.6 | 变更管理 | 需求或设计变更时，评估影响并更新相关文档 |

### 必输出物

| 输出物 | 格式 | 最低要求 |
|--------|------|---------|
| 源代码 | 代码仓库 | 遵循编码规范，有注释和追溯标识 |
| 测试代码 | 代码仓库 | 覆盖核心路径、边界条件、异常路径 |
| 测试报告 | 报告 | 测试覆盖率、通过/失败统计 |
| 代码审查记录 | 文档 | 审查意见、处理结果 |
| 变更日志 | 文档 | 所有需求/设计变更的记录 |
| 进度报告 | 文档 | 里程碑完成情况、偏差分析 |

### 出口条件（Gate 3）

- [ ] 所有计划内的开发任务已完成
- [ ] 所有测试通过（无跳过、无忽略）
- [ ] 测试覆盖率满足预设标准
- [ ] 所有代码审查已完成且问题已解决
- [ ] 所有变更已记录并更新相关文档
- [ ] 代码可追溯到需求（追溯矩阵完整）

### 门控类型

**硬性门控（Hard Gate）** — 必须有新鲜的测试通过证据，不得有未解决的审查问题。

---

## Phase 4：知识沉淀

### 目标

将开发过程中的经验、决策和教训沉淀为可复用的知识资产。

### 入口条件

- Phase 3 的全部出口条件已满足。
- 所有代码已通过审查和测试。

### 核心活动

| # | 活动 | 描述 |
|---|------|------|
| 4.1 | 决策记录整理 | 回顾关键设计决策，记录决策背景、选项、理由、后果 |
| 4.2 | 经验教训总结 | 总结开发过程中的成功经验和失败教训 |
| 4.3 | 代码文档完善 | 补充 API 文档、架构说明、使用指南 |
| 4.4 | 知识库更新 | 将新知识整合到项目知识库中 |
| 4.5 | 改进建议 | 提出流程改进、工具改进、技术改进建议 |

### 必输出物

| 输出物 | 格式 | 最低要求 |
|--------|------|---------|
| 决策记录（ADR） | 结构化文档 | 每个关键决策有背景、选项、结论 |
| 经验教训报告 | 文档 | 至少 3 条经验、3 条教训 |
| API 文档 | 文档 | 所有公开接口有说明、参数、返回值、示例 |
| 知识库更新 | 文档/数据库 | 新增知识条目已入库 |
| 改进建议清单 | 表格 | 按优先级排序的改进建议 |

### 出口条件（Gate 4）

- [ ] 所有关键决策已记录
- [ ] 经验教训已总结
- [ ] API 文档完整
- [ ] 知识库已更新
- [ ] 改进建议已记录

### 门控类型

**软性门控（Soft Gate）** — 知识沉淀的完整性由 AI 自评，但必须产出全部必输出物。

---

## Phase 5：交付标准

### 目标

确保交付物满足全部质量标准，完成正式交付。

### 入口条件

- Phase 4 的全部出口条件已满足。
- 知识沉淀已完成。

### 核心活动

| # | 活动 | 描述 |
|---|------|------|
| 5.1 | 交付检查清单验证 | 逐项检查交付物是否满足标准 |
| 5.2 | 全量回归测试 | 运行完整测试套件，确保无回归 |
| 5.3 | 需求符合性验证 | 逐条验证需求是否满足 |
| 5.4 | 交付物打包 | 整理代码、文档、配置为可交付包 |
| 5.5 | 验收演示 | 向用户演示交付物，展示验证证据 |
| 5.6 | 交付签字 | 用户确认交付物满足要求 |

### 必输出物

| 输出物 | 格式 | 最低要求 |
|--------|------|---------|
| 交付检查清单 | 表格 | 逐项验证结果（通过/不通过/不适用） |
| 回归测试报告 | 报告 | 全量测试结果 |
| 需求符合性报告 | 表格 | 每条需求的验证状态 |
| 交付包 | 压缩包/仓库 | 包含全部代码、文档、配置 |
| 验收记录 | 文档 | 用户确认记录 |

### 出口条件（Gate 5）

- [ ] 交付检查清单全部通过（无不通过项）
- [ ] 全量回归测试通过
- [ ] 所有需求已验证满足
- [ ] 交付包完整
- [ ] 用户已确认验收

### 门控类型

**硬性门控（Hard Gate）** — 必须有用户显式验收确认。

## 阶段间依赖关系

```
Phase 0 ──[Gate 0]──→ Phase 1 ──[Gate 1]──→ Phase AIS ──[Gate 1.5]──→ Phase 2 ──[Gate 2]──→ Phase 3 ──[Gate 3]──→ Phase 4 ──[Gate 4]──→ Phase 5 ──[Gate 5]
                                                                                                                        ↑
                                                                                                                        │
                                                                                                                可选：回到 Phase 3
                                                                                                                （修复发现的问题）
```

### 依赖规则

1. **严格顺序**：Phase 0 → 1 → AIS → 2 → 3 → 4 → 5，不可跳过任何阶段。
2. **回退机制**：后续阶段发现前置阶段的问题时，必须回退到问题所在阶段修复，修复后重新通过门控。
3. **迭代支持**：整个流程支持迭代。每次迭代从 Phase 1 开始（Phase 0 只需在首次执行），经过完整流程交付增量。

### 回退规则

| 发现问题的阶段 | 回退目标 | 回退条件 |
|--------------|---------|---------|
| Phase AIS（实施规格阶段） | Phase 1（需求阶段） | 发现需求遗漏或歧义 |
| Phase 2（设计阶段） | Phase AIS（实施规格阶段） | 发现 AIS 遗漏或理解偏差 |
| Phase 2（设计阶段） | Phase 1（需求阶段） | 发现需求遗漏或歧义 |
| Phase 3（开发阶段） | Phase 2（设计阶段） | 发现设计缺陷或不可行 |
| Phase 3（开发阶段） | Phase 1（需求阶段） | 发现需求变更 |
| Phase 4（知识沉淀） | Phase 3（开发阶段） | 发现代码质量问题 |
| Phase 5（交付标准） | Phase 3（开发阶段） | 验证发现功能缺陷 |
| Phase 5（交付标准） | Phase 1（需求阶段） | 需求符合性验证失败 |

## 硬性门控（Hard Gate）定义

硬性门控是不可绕过的质量检查点。未通过硬性门控，不得进入下一阶段。

### 硬性门控清单

| 门控 | 阶段 | 检查内容 | 通过标准 |
|------|------|---------|---------|
| Gate 0 | Phase 0 → 1 | 画像完整性 | 用户画像和项目画像已确认 |
| Gate 1 | Phase 1 → AIS | 需求规格质量 | 需求文档获用户显式确认 |
| Gate 1.5 | Phase AIS → 2 | AIS 质量 | AIS 获产品经理显式确认 |
| Gate 2 | Phase 2 → 3 | 设计方案质量 | 设计文档通过评审，计划获确认 |
| Gate 3 | Phase 3 → 4 | 代码质量 | 测试全通过，审查全完成，追溯完整 |
| Gate 5 | Phase 5 → 交付 | 交付质量 | 用户显式验收确认 |

### 硬性门控的约束

- **不可降级**：不得以"时间紧迫"为由降低门控标准。
- **不可跳过**：不得以"改动很小"为由跳过门控。
- **不可合并**：不得将多个门控合并为一次检查。
- **必须记录**：每次门控检查必须有记录，包括检查时间、检查结果、检查人。

### 软性门控（Soft Gate）

软性门控建议通过但不强制阻塞。未通过时，AI 必须记录原因和风险，经用户确认后可继续。

| 门控 | 阶段 | 说明 |
|------|------|------|
| Gate 4 | Phase 4 → 5 | 知识沉淀完整性。建议完整，但允许在交付后补充。 |
