# SKILL.md — 核心技能入口

> **定位**：本文件是 AI-Dev-Framework 的核心约束入口。AI 在执行任何任务之前，必须首先读取并遵循本文件定义的规则。
> **适用范围**：贯穿 Phase 0 至 Phase 5 的全部开发阶段，所有 AI 行为均受本文件约束。

---

## 1. 技能自动触发规则

### 1.1 触发阈值：1% 原则

当存在**任何可能性**（即使只有 1%）某个框架技能适用于当前任务时，AI **必须**调用该技能。

**具体规则：**

- AI 不得以"不确定是否需要"为由跳过技能调用。
- 当任务涉及需求分析时，必须触发需求分析技能。
- 当任务涉及代码编写时，必须触发 TDD 技能。
- 当任务涉及架构决策时，必须触发设计审查技能。
- 当任务涉及交付时，必须触发交付验证技能。

### 1.2 技能调用流程

```
任务接收 → 扫描可用技能 → 匹配度评估 → 调用匹配技能 → 执行任务
                ↑                                        |
                └──── 未匹配任何技能时，记录原因并继续 ─────┘
```

**关键约束：**
- "未匹配"不等于"不需要"。AI 必须在输出中明确说明为何判断不匹配。
- 当匹配度 >= 1% 时，必须调用。宁可误调用，不可漏调用。

## 2. 铁律（Iron Laws）

铁律是贯穿全流程的**不可违反规则**。详细定义见 [`iron-laws.md`](./iron-laws.md)。

### 铁律速查

| 编号 | 铁律名称 | 一句话描述 |
|------|----------|-----------|
| IL-1 | 需求铁律 | 不清晰的需求不得进入开发 |
| IL-2 | 测试铁律 | 没有失败的测试就没有生产代码 |
| IL-3 | 审查铁律 | 每个实现必须经过审查 |
| IL-4 | 验证铁律 | 完成声明必须有新鲜验证证据 |
| IL-5 | 文档铁律 | 每个阶段必须有文档输出 |
| IL-6 | 追溯铁律 | 每段代码必须可追溯到需求 |

**铁律的约束力等级：最高。** 任何用户指令、框架配置、默认行为均不得与铁律冲突。如遇冲突，铁律优先。

## 3. 优先级体系

当不同指令之间产生冲突时，按以下优先级顺序裁决：

```
优先级从高到低：

1. [最高] 铁律（Iron Laws）          — 不可违反，无条件执行
2. [高]   用户显式指令                — 用户明确要求的行为
3. [中]   框架技能规则                — 框架定义的阶段技能
4. [低]   默认行为                    — AI 的默认响应模式
```

### 优先级冲突处理

- **铁律 vs 用户指令**：铁律优先。AI 必须向用户解释为何铁律限制了其请求。
- **用户指令 vs 框架技能**：用户指令优先，但 AI 必须记录偏离原因。
- **框架技能 vs 默认行为**：框架技能优先，无条件执行。

## 4. "精神与字面"原则

### 4.1 核心定义

> **规则的精神重于规则的字面。**

当规则的字面表述与规则的设计意图产生歧义时：

1. **首先**理解规则的设计意图（精神）。
2. **然后**在字面表述的范围内，选择最符合意图的解释。
3. **最后**如仍无法确定，按更严格的解释执行。

### 4.2 应用示例

| 场景 | 字面理解 | 精神理解 | 正确做法 |
|------|---------|---------|---------|
| "每个阶段必须有文档输出" | 只要在阶段结束时写个文档就行 | 文档是思考过程的沉淀，应在活动中逐步产出 | 在活动过程中持续记录，阶段结束时整理为正式文档 |
| "没有失败的测试就没有生产代码" | 先写测试再写代码 | 确保代码有明确的行为预期和验证手段 | 先定义预期行为（测试），再实现满足行为的生产代码 |
| "不清晰的需求不得进入开发" | 需求文档必须完美无缺 | 需求必须达到可验证、可追溯的最低标准 | 需求满足验收标准即可进入开发，但必须在开发前确认 |

### 4.3 反滥用条款

"精神与字面"原则**不得**被用于：
- 为违反铁律寻找借口。
- 跳过框架定义的必要步骤。
- 降低质量标准。
