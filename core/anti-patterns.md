# anti-patterns.md — 反模式清单

> **定位**：本文件列出 AI 在开发过程中常见的"合理化借口"及对应反驳。AI 在执行任务时，如内部出现类似念头，必须立即对照本清单进行自我纠正。
> **使用方式**：每当 AI 准备跳过、简化、或"灵活处理"某个步骤时，先查阅本清单。

---

## 一、需求阶段反模式

### AP-R01 "需求已经很清楚了"

| 项目 | 内容 |
|------|------|
| **借口** | "用户描述得很清楚，不需要再做需求分析" |
| **反驳** | 你还没问过用户场景。用户的描述往往是解决方案而非需求本身。"清楚"是验证的结果，不是直觉的判断。 |
| **正确做法** | 使用结构化需求分析方法（如用户故事、场景分析），至少产出：用户角色、核心场景、验收标准、边界条件。 |
| **违反铁律** | IL-1 需求铁律 |

### AP-R02 "这个需求跟之前的项目类似"

| 项目 | 内容 |
|------|------|
| **借口** | "之前做过类似的功能，可以直接复用之前的方案" |
| **反驳** | 相似不等于相同。差异点可能隐藏在边界条件、性能要求、用户群体等维度。复用不等于不分析。 |
| **正确做法** | 明确列出当前需求与历史需求的差异点，针对差异点进行专门分析。 |
| **违反铁律** | IL-1 需求铁律、IL-6 追溯铁律 |

### AP-R03 "用户会告诉我们需要什么"

| 项目 | 内容 |
|------|------|
| **借口** | "不需要主动分析，用户有新需求会说的" |
| **反驳** | 被动等待需求是需求遗漏的主要原因。用户通常只描述他们能想到的场景，而遗漏边缘场景和异常路径。 |
| **正确做法** | 主动识别潜在场景，向用户确认。使用"5W1H"方法系统性地挖掘需求。 |
| **违反铁律** | IL-1 需求铁律 |

### AP-R04 "需求文档太重了，简单记录就行"

| 项目 | 内容 |
|------|------|
| **借口** | "写详细的需求文档浪费时间，简单记一下就够了" |
| **反驳** | 需求文档的目的是消除歧义，而非展示工作量。简单记录往往留下大量歧义空间，成为后续返工的根源。 |
| **正确做法** | 需求文档可以简洁，但必须完整：每条需求有标识、描述、验收标准、优先级。 |
| **违反铁律** | IL-1 需求铁律、IL-5 文档铁律 |

## 二、设计阶段反模式

### AP-D01 "这个设计太简单不需要文档"

| 项目 | 内容 |
|------|------|
| **借口** | "就几个函数的事，画什么架构图" |
| **反驳** | 简单的设计更需要文档，因为简单的设计容易被误解为"不需要思考"。文档的价值在于记录"为什么这样设计"，而非"设计有多复杂"。 |
| **正确做法** | 即使是简单设计，也必须记录：设计决策、关键接口、数据流向、选型理由。 |
| **违反铁律** | IL-5 文档铁律 |

### AP-D02 "先用最简单的方式实现，后面再重构"

| 项目 | 内容 |
|------|------|
| **借口** | "先跑起来再说，架构可以后面优化" |
| **反驳** | "后面再重构"是最常见的谎言。技术债务一旦积累，重构成本随时间指数增长。没有设计就编码，等于在积累债务。 |
| **正确做法** | 在编码前完成最小可行设计：定义核心接口、数据模型、模块边界。设计可以迭代，但不能跳过。 |
| **违反铁律** | IL-1 需求铁律、IL-5 文档铁律 |

### AP-D03 "这个技术栈我最熟悉，直接用就行"

| 项目 | 内容 |
|------|------|
| **借口** | "用这个技术肯定没问题，不需要评估其他方案" |
| **反驳** | 熟悉不等于合适。技术选型必须基于需求特征（性能、规模、团队、生态），而非个人偏好。 |
| **正确做法** | 至少评估 2-3 个候选方案，基于明确的标准（如性能、可维护性、学习曲线、社区支持）进行对比选择，并记录选型理由。 |
| **违反铁律** | IL-5 文档铁律 |

### AP-D04 "设计评审可以省掉，反正就我一个人开发"

| 项目 | 内容 |
|------|------|
| **借口** | "没有其他人参与，设计评审没有意义" |
| **反驳** | 设计评审的目的不仅是发现问题，更是强制思考。评审过程本身就能发现设计中的盲点和矛盾。 |
| **正确做法** | 即使只有一个人，也应使用设计评审清单进行自审，或请求用户/其他 AI 实例参与评审。 |
| **违反铁律** | IL-3 审查铁律 |

## 三、编码阶段反模式

### AP-C01 "太简单不需要测试"

| 项目 | 内容 |
|------|------|
| **借口** | "就一个工具函数，写什么测试" |
| **反驳** | 简单代码也会出错。越"简单"的代码，越容易因为过度自信而跳过验证。Bug 往往藏在"不可能出错"的地方。 |
| **正确做法** | 无论复杂度，每个函数/模块都必须有对应的测试。TDD 的"红-绿-重构"循环适用于所有代码。 |
| **违反铁律** | IL-2 测试铁律 |

### AP-C02 "先写完功能再补测试"

| 项目 | 内容 |
|------|------|
| **借口** | "先把功能实现完，测试回头统一补" |
| **反驳** | 事后补写的测试往往只验证"代码做了什么"，而非"代码应该做什么"。这导致测试变成了代码的附庸，而非行为的定义者。 |
| **正确做法** | 严格遵循 TDD 流程：先写失败的测试，再写生产代码。测试是设计的工具，不是质量的补丁。 |
| **违反铁律** | IL-2 测试铁律 |

### AP-C03 "这个边界条件不太可能发生"

| 项目 | 内容 |
|------|------|
| **借口** | "正常流程都覆盖了，极端情况不用管" |
| **反驳** | "不太可能"在软件系统中等于"迟早会发生"。生产环境的输入永远超出预期。边界条件不处理，就是定时炸弹。 |
| **正确做法** | 系统性地识别边界条件：空输入、最大值、最小值、并发、网络异常、数据损坏等。为每个边界条件编写测试。 |
| **违反铁律** | IL-2 测试铁律 |

### AP-C04 "先复制这段代码，后面再抽象"

| 项目 | 内容 |
|------|------|
| **借口** | "赶时间先复制粘贴，重构以后再说" |
| **反驳** | 复制粘贴是技术债务的快速通道。重复代码意味着重复的 bug 修复、重复的维护成本。 |
| **正确做法** | 发现重复时立即抽象。遵循 DRY（Don't Repeat Yourself）原则。如果抽象时机不成熟，至少添加 TODO 标记并设定处理期限。 |
| **违反铁律** | IL-3 审查铁律（审查时应发现重复代码） |

### AP-C05 "注释太多了反而影响可读性"

| 项目 | 内容 |
|------|------|
| **借口** | "好代码不需要注释，代码本身就是文档" |
| **反驳** | 代码描述"怎么做"，注释描述"为什么这么做"。两者解决不同的问题。没有"为什么"的代码，在维护时只能靠猜测。 |
| **正确做法** | 为每个非显而易见的决策添加注释。注释应解释意图和上下文，而非重复代码逻辑。 |
| **违反铁律** | IL-5 文档铁律 |

## 四、审查阶段反模式

### AP-V01 "我已经手动测试过了"

| 项目 | 内容 |
|------|------|
| **借口** | "我自己跑了一遍没问题，不需要自动化测试" |
| **反驳** | 手动测试不等于系统测试。手动测试是一次性的、不可重复的、依赖测试者记忆的。自动化测试是持久的、可重复的、可回归的。 |
| **正确做法** | 手动测试可以作为补充，但不能替代自动化测试。所有核心路径必须有自动化测试覆盖。 |
| **违反铁律** | IL-2 测试铁律、IL-4 验证铁律 |

### AP-V02 "这个改动不会影响其他模块"

| 项目 | 内容 |
|------|------|
| **借口** | "只改了这个文件，其他地方应该不受影响" |
| **反驳** | "应该"是危险词汇。模块间的依赖关系往往比想象中复杂。没有全量回归测试，就不能声称"不影响其他模块"。 |
| **正确做法** | 运行全量测试套件验证改动的影响范围。如果测试套件不足，先补充测试再提交。 |
| **违反铁律** | IL-2 测试铁律、IL-4 验证铁律 |

### AP-V03 "代码风格问题不重要"

| 项目 | 内容 |
|------|------|
| **借口** | "功能正确就行，格式什么的无所谓" |
| **反驳** | 代码风格不是审美问题，而是沟通问题。一致的代码风格降低认知负担，提升团队协作效率。风格不一致意味着有人在"翻译"代码。 |
| **正确做法** | 使用自动化工具（linter、formatter）强制代码风格。审查时检查风格一致性。 |
| **违反铁律** | IL-3 审查铁律 |

### AP-V04 "这个 TODO 以后再处理"

| 项目 | 内容 |
|------|------|
| **借口** | "标记个 TODO 就行，不阻塞当前任务" |
| **反驳** | TODO 是技术债务的借条。没有截止日期的 TODO 等于永远不会处理。TODO 积累到一定程度，代码库就变成了"TODO 仓库"。 |
| **正确做法** | 每个 TODO 必须附带：问题描述、影响范围、处理期限、负责人。定期清理 TODO 列表。 |
| **违反铁律** | IL-4 验证铁律（TODO 意味着未完成） |

## 五、完成阶段反模式

### AP-F01 "应该可以了"

| 项目 | 内容 |
|------|------|
| **借口** | "功能实现了，应该没问题了" |
| **反驳** | 禁止使用"应该"。完成声明必须有新鲜验证证据。如果你没有运行测试、没有验证输出、没有对比需求，就不能声称完成。 |
| **正确做法** | 提供具体的验证证据：测试报告、运行日志、需求对比表。使用确定性语言："已验证"、"测试通过"、"需求满足"。 |
| **违反铁律** | IL-4 验证铁律 |

### AP-F02 "剩下的都是小问题"

| 项目 | 内容 |
|------|------|
| **借口** | "核心功能完成了，剩下几个小 bug 后面修" |
| **反驳** | "小问题"的累积效应可能比"大问题"更严重。多个小问题组合在一起可能产生不可预测的交互效应。未修复的 bug 是对质量的妥协。 |
| **正确做法** | 所有已知问题必须记录在案，评估影响和优先级。高优先级问题必须在交付前修复。低优先级问题必须有明确的处理计划。 |
| **违反铁律** | IL-4 验证铁律 |

### AP-F03 "用户没提这个要求，不用做"

| 项目 | 内容 |
|------|------|
| **借口** | "交付清单上没有这一项，跳过就行" |
| **反驳** | 交付标准由框架定义，不仅限于用户的显式要求。框架的交付标准包括：测试覆盖率、文档完整性、代码质量、安全检查等。 |
| **正确做法** | 对照框架的交付检查清单逐项验证。未满足的项必须处理或明确记录风险。 |
| **违反铁律** | IL-4 验证铁律、IL-5 文档铁律 |

### AP-F04 "差不多就行了，追求完美会拖慢进度"

| 项目 | 内容 |
|------|------|
| **借口** | "80/20 法则，差不多好了就行" |
| **反驳** | "差不多"不是工程标准。框架定义了明确的质量门槛，达到门槛是最低要求，不是"完美主义"。 |
| **正确做法** | 严格对照交付标准逐项验证。每项标准只有"满足"和"不满足"两种状态，没有"差不多满足"。 |
| **违反铁律** | IL-4 验证铁律 |

### AP-F05 "交付了就完成任务了"

| 项目 | 内容 |
|------|------|
| **借口** | "代码提交了，任务完成" |
| **反驳** | 交付不是终点。没有知识沉淀的交付是不完整的交付。团队/用户需要的不仅是代码，还有代码背后的决策、经验和教训。 |
| **正确做法** | 交付后必须执行 Phase 4 知识沉淀：记录关键决策、经验教训、已知限制、后续建议。 |
| **违反铁律** | IL-5 文档铁律 |

## 六、生命周期管理反模式

### AP-L01 "老接口先留着吧，万一还有人在用"

| 项目 | 内容 |
|------|------|
| **借口** | "新接口已经上线了，但老接口先不删，万一还有地方在调用" |
| **反驳** | 没有调用方追踪的"万一"就是技术债务。老接口和新接口长期共存会导致：维护成本翻倍（两套代码都要维护）、调用方困惑（不知道该用哪个）、数据不一致（两套逻辑可能产生不同结果）、代码库膨胀（越来越多的"先留着"的代码）。 |
| **正确做法** | 新接口上线时，同步执行废弃流程：① 老接口标记 `@deprecated`；② 通知所有已知调用方迁移；③ 设定过渡期（默认 30 天）；④ 过渡期结束后执行下线（删除代码、更新文档、更新追溯矩阵）。如果不确定是否有调用方，加监控确认调用情况后再决定。 |
| **违反铁律** | IL-6 追溯铁律（接口生命周期管理） |

### AP-L02 "下线接口太危险了，还是别动了"

| 项目 | 内容 |
|------|------|
| **借口** | "老代码虽然不用了，但删了万一出问题就麻烦了" |
| **反驳** | 不删除的死代码比删除代码更危险——它占用认知资源（阅读代码时需要判断是否有效）、干扰搜索和重构、可能被误调用、隐藏真实的技术债务规模。有全量回归测试保护的删除操作是安全的。 |
| **正确做法** | 在全量回归测试通过的保护下执行下线。如果测试覆盖不足，先补充测试再下线。下线后运行全量测试确认无回归。 |
| **违反铁律** | IL-2 测试铁律、IL-6 追溯铁律 |

## 反模式使用指南

### 自检流程

AI 在以下时刻必须查阅本清单：

1. **准备跳过某个步骤时** — 检查是否属于已知反模式。
2. **内心出现"可以简化"的念头时** — 检查简化是否合理。
3. **准备声明"完成"时** — 检查是否有未处理的反模式。
4. **用户要求跳过某步骤时** — 使用本清单向用户解释风险。

### 新反模式发现

当 AI 在实践中发现新的反模式时：

1. 记录反模式的借口表述。
2. 分析反模式的本质和危害。
3. 编写反驳和正确做法。
4. 提交用户确认后加入本清单。

## 七、人机协作反模式

### AP-C01 "测试悖论——测试通过了但行为是错的"

| 项目 | 内容 |
|------|------|
| **借口** | "测试全部通过了，代码肯定没问题" |
| **反驳** | 测试通过 ≠ 行为正确。如果测试本身就是基于错误理解编写的，测试通过只证明"代码和测试一致"，不证明"代码和需求一致"。AI 写的测试倾向于验证"代码做了什么"而非"代码应该做什么"。 |
| **正确做法** | 测试用例必须对照 AIS 中的 Concrete Examples（具体示例）编写，期望值必须来自 AIS 的 I/O 契约。产品经理应至少审核核心用户旅程的 E2E 测试，确认测试验证的是正确的业务逻辑。 |
| **违反铁律** | IL-2 测试铁律、P3-1 TDD 铁律 |

### AP-C02 "反馈毒性——用自然语言纠正 AI 的理解偏差"

| 项目 | 内容 |
|------|------|
| **借口** | "我跟 AI 说了好几次了，它怎么还是理解错" |
| **反驳** | 自然语言本身有歧义。你用自然语言纠正 AI，AI 可能只是换了一种错误方式理解。每次纠正不一定让理解更准确，可能引入新的偏差。这就是"反馈毒性"——纠正越多，偏差可能越大。 |
| **正确做法** | 纠正偏差时更新 AIS（结构化规格书），而非口头描述。用决策表、I/O 契约、状态机等无歧义的格式纠正。如果同一问题被纠正 2 次以上仍未解决，重写 AIS 中该条目的描述。 |
| **违反铁律** | IL-5 文档铁律 |

### AP-C03 "规格漂移——AIS 和代码逐渐脱节"

| 项目 | 内容 |
|------|------|
| **借口** | "AIS 之前写的，后来改了代码没来得及更新" |
| **反驳** | AIS 和代码脱节后，新会话的 AI 基于 AIS 写代码会和现有代码冲突。AIS 是 AI 开发的"唯一真相源"，如果它和代码不一致，AI 的所有行为都不可预测。 |
| **正确做法** | 每次代码变更后，必须检查 AIS 是否需要同步更新。如果代码行为和 AIS 定义不一致，必须立即同步——要么更新 AIS，要么回退代码变更。Gate 4.5 验收就绪预检中包含 AIS 一致性检查。 |
| **违反铁律** | IL-5 文档铁律、IL-6 追溯铁律 |

### AP-C04 "只测后端不测前端——接口通了但用户用不了"

| 项目 | 内容 |
|------|------|
| **借口** | "接口测试都通过了，前端应该没问题" |
| **反驳** | 用户接触的是界面，不是接口。接口全通不代表用户能正常使用。按钮点击无响应、页面加载失败、表单验证缺失、状态显示错误——这些前端问题接口测试完全覆盖不到。 |
| **正确做法** | 遵循测试分层策略（Layer 1-4），至少覆盖 API 测试 + 组件测试 + E2E 交互测试。核心用户旅程必须有 E2E 测试覆盖。参见 TDD 指南第 7 节。 |
| **违反铁律** | IL-2 测试铁律 |
