# 需求收集指南

> **定位**：本文件是 Phase 1 需求分析的实操指南，定义 AI 在需求收集过程中的提问策略、挖掘维度、分类框架和优先级评估方法。
> **适用范围**：Phase 1 步骤 1（需求收集）和步骤 2（潜在需求挖掘）。
> **前置依赖**：Phase 0 使用者画像（`phase-0-profiling/user-profile-template.md`）。

---

## 一、苏格拉底式提问技巧

苏格拉底式提问的核心不是"问问题"，而是通过系统性追问引导使用者从模糊想法走向清晰需求。AI 应扮演"好奇的专家"角色——既懂技术又尊重使用者的领域知识。

### 1.1 提问的四个层次

需求收集应按以下四个层次逐步深入，不可跳层：

```
层次 1：现象层  →  "发生了什么？" "现在是怎么做的？"
层次 2：原因层  →  "为什么会这样？" "痛点在哪里？"
层次 3：目标层  →  "理想状态是什么？" "成功的标准是什么？"
层次 4：方案层  →  "你设想过怎么解决？" "有没有参考的产品？"
```

**原则：先理解问题，再讨论方案。** 使用者往往直接描述解决方案而非需求本身。AI 必须穿透方案描述，找到背后的真实需求。

### 1.2 基于使用者能力的差异化提问策略

根据 Phase 0 使用者画像中的需求表达能力等级（A/B/C/D），采用不同的提问策略：

#### A 级（精确描述）—— 协作式提问

使用者能给出详细的功能描述和验收标准，AI 的角色是补充遗漏和验证完整性。

| 策略 | 说明 | 示例 |
|------|------|------|
| 补充确认 | 对使用者未提及的维度进行确认 | "你提到了正常流程，那如果网络中断了，系统应该怎么处理？" |
| 边界追问 | 追问边界条件和极端场景 | "如果数据量达到 10 万条，这个查询还能在 3 秒内返回吗？" |
| 一致性检查 | 检查需求之间是否有矛盾 | "你要求实时同步，但也要求离线可用，这两个需求在技术上存在冲突，你更看重哪个？" |

#### B 级（场景描述）—— 引导式提问

使用者能描述使用场景和期望行为，但缺乏技术细节。AI 需要从场景中提取需求。

| 策略 | 说明 | 示例 |
|------|------|------|
| 场景细化 | 将笼统场景拆解为具体步骤 | "你说'登录后能看到数据'，具体是哪些数据？以什么形式展示？需要支持筛选和排序吗？" |
| 角色切换 | 从不同角色的视角追问 | "如果是一个新用户第一次使用，他会看到什么？他需要什么引导？" |
| 量化追问 | 将定性描述转化为定量标准 | "你说'响应要快'，具体是多快？1 秒？3 秒？5 秒？" |

#### C 级（目标描述）—— 探索式提问

使用者能说清楚想要什么效果，但不确定具体怎么做。AI 需要帮助使用者明确需求。

| 策略 | 说明 | 示例 |
|------|------|------|
| 参考锚定 | 用已知产品作为讨论锚点 | "你用过 Notion 吗？你想要的功能和 Notion 的文档管理类似吗？有哪些不同？" |
| 场景模拟 | 模拟具体使用场景来启发需求 | "假设你现在是销售小王，早上打开系统，你第一步会做什么？" |
| 选项引导 | 给出选项让使用者选择 | "数据展示你更倾向于：A) 表格列表 B) 图表可视化 C) 两者都要？" |

#### D 级（模糊描述）—— 启发式提问

使用者只有一个大致想法，需要大量引导。AI 需要从零开始构建需求理解。

| 策略 | 说明 | 示例 |
|------|------|------|
| 问题诊断 | 先理解使用者想解决什么问题 | "你能描述一下现在遇到的最大困难是什么吗？这个问题给你造成了什么影响？" |
| 最小场景 | 从最简单的使用场景开始 | "如果这个系统只做一件事，你希望它做什么？" |
| 渐进展开 | 从核心场景逐步扩展 | "好的，核心功能是 X。那除了 X，还有什么是你觉得需要的？" |
| 示例驱动 | 用具体示例代替抽象描述 | "比如你管理客户信息，现在是用 Excel 还是纸质的？大概有多少条数据？" |

### 1.3 关键提问句式

以下是 AI 在需求收集过程中应使用的关键提问句式：

**挖掘真实需求：**
- "你提到想要 [X]，能说说为什么需要 [X] 吗？"（区分需求与方案）
- "如果没有 [X]，你的工作/生活会有什么不同？"（评估需求重要性）
- "你之前是怎么解决这个问题的？"（了解现状和痛点）

**消除歧义：**
- "当你说 [X]，具体是指什么？能举个例子吗？"
- "在什么情况下你会用到这个功能？能描述一个具体场景吗？"
- "如果 [条件 A] 和 [条件 B] 同时发生，你期望系统怎么处理？"

**挖掘边界条件：**
- "如果数据量是现在的 10 倍，这个功能还能正常工作吗？"
- "如果用户在操作到一半时网络断开了，应该怎么处理？"
- "如果两个用户同时修改同一条数据，应该怎么处理？"

**确认理解：**
- "我理解你的意思是 [复述需求]，对吗？"
- "根据你说的，我总结出以下 3 个核心需求：[列表]。有没有遗漏或不对的？"

## 二、潜在需求挖掘清单

以下维度是 AI **必须主动检查**的潜在需求领域。即使使用者未提及，AI 也应逐一评估其相关性，并对相关维度向使用者追问。

### 2.1 权限与角色管理

| # | 检查项 | 追问示例 |
|---|--------|---------|
| 1 | 系统是否有不同角色的用户？ | "使用这个系统的有哪些不同角色？他们的权限有什么不同？" |
| 2 | 是否需要角色权限控制？ | "普通用户能看到所有数据，还是只能看到自己的？" |
| 3 | 是否需要管理员角色？ | "谁负责管理系统配置和用户管理？" |
| 4 | 是否需要操作审计？ | "是否需要记录谁在什么时间做了什么操作？" |
| 5 | 是否需要数据隔离？ | "不同部门/团队的数据是否需要互相隔离？" |

### 2.2 数据安全与隐私

| # | 检查项 | 追问示例 |
|---|--------|---------|
| 1 | 数据是否涉及个人隐私？ | "系统中是否存储用户的个人敏感信息（身份证、手机号等）？" |
| 2 | 数据是否需要加密？ | "数据在传输和存储时是否需要加密？" |
| 3 | 是否有数据合规要求？ | "是否需要符合 GDPR、个人信息保护法等法规要求？" |
| 4 | 数据备份策略？ | "数据丢失的容忍度是多少？需要多频繁的备份？" |
| 5 | 数据导出/删除？ | "用户是否有权导出或删除自己的数据？" |

### 2.3 异常场景处理

| # | 检查项 | 追问示例 |
|---|--------|---------|
| 1 | 网络异常 | "网络不稳定时，系统应该怎么处理？是否需要离线模式？" |
| 2 | 并发冲突 | "多人同时操作同一条数据时，怎么处理冲突？" |
| 3 | 数据异常 | "如果用户输入了不合法的数据，系统应该怎么提示？" |
| 4 | 系统故障 | "系统出现故障时，用户应该看到什么？是否有降级方案？" |
| 5 | 操作失败 | "如果某个操作失败了（如支付失败），用户应该看到什么提示？" |
| 6 | 数据量增长 | "如果数据量远超预期（10 倍、100 倍），系统还能正常工作吗？" |

### 2.4 性能要求

| # | 检查项 | 追问示例 |
|---|--------|---------|
| 1 | 响应时间 | "页面加载和操作响应，你能接受的最大等待时间是多少？" |
| 2 | 并发用户数 | "预计同时使用系统的最多有多少人？" |
| 3 | 数据量级 | "预计系统需要处理多少数据？增长速度如何？" |
| 4 | 处理时效 | "数据是否需要实时处理？延迟容忍度是多少？" |
| 5 | 资源限制 | "部署环境有什么资源限制（CPU、内存、带宽）？" |

### 2.5 兼容性要求

| # | 检查项 | 追问示例 |
|---|--------|---------|
| 1 | 浏览器兼容 | "需要支持哪些浏览器？最低版本要求？" |
| 2 | 设备兼容 | "需要在手机、平板、电脑上都使用吗？" |
| 3 | 操作系统 | "需要支持哪些操作系统？" |
| 4 | 数据格式 | "是否需要与现有系统对接？数据格式是什么？" |
| 5 | 第三方集成 | "是否需要与第三方服务集成（支付、短信、邮件等）？" |

### 2.6 可用性要求

| # | 检查项 | 追问示例 |
|---|--------|---------|
| 1 | 用户引导 | "新用户第一次使用时，是否需要引导流程？" |
| 2 | 错误提示 | "系统出错时，你希望用户看到什么样的提示？" |
| 3 | 帮助文档 | "是否需要提供使用帮助或 FAQ？" |
| 4 | 多语言 | "系统是否需要支持多语言？" |
| 5 | 无障碍 | "是否需要考虑无障碍访问（如屏幕阅读器支持）？" |

### 2.7 可维护性要求

| # | 检查项 | 追问示例 |
|---|--------|---------|
| 1 | 日志需求 | "是否需要详细的操作日志和系统日志？" |
| 2 | 监控告警 | "系统是否需要运行状态监控和异常告警？" |
| 3 | 配置管理 | "哪些参数需要支持动态配置，不需要重启系统？" |
| 4 | 部署方式 | "系统部署在哪里？本地服务器还是云端？" |

### 2.8 可扩展性要求

| # | 检查项 | 追问示例 |
|---|--------|---------|
| 1 | 功能扩展 | "未来可能新增哪些功能？需要预留扩展点吗？" |
| 2 | 用户增长 | "预计用户规模会增长到多少？系统需要支持水平扩展吗？" |
| 3 | 多租户 | "未来是否需要支持多租户（不同客户独立使用）？" |
| 4 | API 开放 | "未来是否需要提供 API 给第三方系统调用？" |

### 2.9 合规性要求

| # | 检查项 | 追问示例 |
|---|--------|---------|
| 1 | 行业法规 | "项目所属行业是否有特殊的法规要求？" |
| 2 | 数据留存 | "数据需要保留多长时间？是否有法规要求的最低留存期？" |
| 3 | 审计要求 | "是否需要满足外部审计要求？" |
| 4 | 知识产权 | "是否有开源协议限制？使用的第三方组件是否有许可证要求？" |

### 使用指引

- 对每个维度，AI 应先判断是否与当前项目相关
- 相关的维度，向使用者提出 1-2 个关键问题
- 使用者确认不需要的维度，记录"已评估，不适用"及判断依据
- 使用者未明确回答的维度，AI 应给出建议并标注"AI 建议项"

## 三、需求分类框架

### 3.1 四象限分类法

将需求按"使用者显式程度"和"功能重要性"分为四个象限：

```
                    使用者显式提出
                         │
         增强需求        │        核心需求
      （使用者提到但      │     （使用者明确提出
       非核心功能）       │      且是核心功能）
                         │
    ─────────────────────┼─────────────────────
                         │
         未来需求        │        隐性需求
    （使用者提到但      │     （使用者未提到
     当前不需要）       │      但系统必须具备）
                         │
                    使用者未提及
```

### 3.2 各类别定义

| 类别 | 定义 | 示例 | 处理方式 |
|------|------|------|---------|
| **核心需求** | 使用者明确提出且是系统核心价值的需求 | "用户能注册登录并管理客户信息" | 必须实现，优先级最高 |
| **增强需求** | 使用者明确提出但非核心价值的需求 | "支持数据导出为 Excel" | 评估后纳入，优先级次之 |
| **隐性需求** | 使用者未提及但系统正常运行所必需的需求 | 数据备份、错误处理、权限控制 | AI 必须主动挖掘并纳入 |
| **未来需求** | 使用者提到但当前阶段不需要的需求 | "以后要支持移动端" | 记录但不纳入当前范围 |

### 3.3 分类标注规则

在 PRD 文档中，每条需求必须标注其类别：

```
REQ-001 [核心] 用户注册登录功能
REQ-002 [核心] 客户信息 CRUD
REQ-003 [增强] 数据导出为 Excel
REQ-004 [隐性] 数据自动备份（每日）
REQ-005 [隐性] 操作审计日志
REQ-006 [未来] 移动端适配
```

### 3.4 隐性需求的识别原则

隐性需求是需求分析中最容易被遗漏的部分。AI 应遵循以下原则识别隐性需求：

1. **系统生存性原则**：如果缺少这个需求，系统是否能正常运行？如果不能，它是隐性需求。
2. **数据安全原则**：涉及用户数据的功能，是否考虑了安全保护？数据安全永远是隐性需求。
3. **异常必然原则**：任何操作都可能失败。失败时的处理方式是隐性需求。
4. **用户体验原则**：用户第一次使用时能否顺利上手？上手引导是隐性需求。
5. **运维可行原则**：系统部署后能否被有效运维？日志和监控是隐性需求。

## 四、需求优先级评估方法

### 4.1 优先级等级定义

| 优先级 | 标识 | 定义 | 处理原则 |
|--------|------|------|---------|
| **P0 - 必须有** | Must Have | 系统核心价值所在，没有则产品不可用 | 必须在第一版实现，不可协商 |
| **P1 - 应该有** | Should Have | 重要但非核心，缺少会显著影响用户体验 | 应在第一版实现，资源不足时可推迟 |
| **P2 - 可以有** | Could Have | 锦上添花的功能，缺少不影响核心流程 | 资源允许时实现，优先级最低 |
| **P3 - 未来有** | Won't Have (this time) | 当前版本不需要，但未来可能需要 | 记录需求，纳入未来规划 |

### 4.2 优先级评估矩阵

从两个维度评估需求优先级：

```
          业务价值高
               │
    P0 必须有  │  P1 应该有
               │
  ─────────────┼─────────────
               │
    P2 可以有  │  P3 未来有
               │
          业务价值低

  实现难度低 ←──┼──→ 实现难度高
```

**评估规则：**
- 业务价值高 + 实现难度低 = P0（优先实现）
- 业务价值高 + 实现难度高 = P1（应该实现）
- 业务价值低 + 实现难度低 = P2（可以顺手做）
- 业务价值低 + 实现难度高 = P3（未来再做）

### 4.3 优先级评估步骤

1. **列出所有需求**：将功能需求和隐性需求全部列出
2. **评估业务价值**：每条需求对使用者业务的价值（高/中/低）
3. **评估实现难度**：每条需求的技术实现难度（高/中/低）
4. **标注优先级**：根据评估矩阵标注 P0/P1/P2/P3
5. **与使用者确认**：将优先级排序结果展示给使用者，协商调整
6. **处理冲突**：当 AI 与使用者的优先级判断不一致时，以使用者判断为准，但 AI 必须记录分歧和理由

### 4.4 优先级确认原则

- **P0 需求的数量控制**：P0 需求建议不超过总需求数量的 30%。如果 P0 过多，说明范围定义不够精确，需要重新聚焦
- **隐性需求的优先级**：安全和数据保护类的隐性需求通常应归为 P0 或 P1
- **增强需求的优先级**：增强需求通常为 P1 或 P2，不应挤占 P0 资源
- **未来需求的处理**：P3 需求不纳入当前版本，但必须记录在 PRD 中，作为未来迭代的输入

### 4.5 优先级变更管理

需求优先级在分析过程中可能发生变化。变更时必须记录：

| 字段 | 内容 |
|------|------|
| 需求标识 | REQ-XXX |
| 原优先级 | P0/P1/P2/P3 |
| 新优先级 | P0/P1/P2/P3 |
| 变更原因 | 为什么调整优先级 |
| 变更提出方 | 使用者 / AI |
| 变更时间 | YYYY-MM-DD |
| 影响评估 | 优先级变更对其他需求的影响 |

## 五、需求收集的注意事项

### 5.1 常见陷阱

| 陷阱 | 表现 | 应对 |
|------|------|------|
| **方案即需求** | 使用者描述的是解决方案而非需求 | 追问"为什么需要这个"，找到真实需求 |
| **遗漏用户角色** | 只考虑了一种用户角色 | 主动追问"还有谁会使用这个系统" |
| **忽略异常路径** | 只讨论正常流程 | 主动追问"如果出错了怎么办" |
| **需求过于宽泛** | "做一个管理系统" | 逐步聚焦到具体功能和场景 |
| **需求过于细节** | 陷入 UI 按钮颜色等细节 | 先确认功能需求，UI 细节留到设计阶段 |

### 5.2 需求收集的质量标准

一次高质量的需求收集应满足：

- [ ] 使用者的核心诉求已被完整记录
- [ ] 每条需求都能追溯到使用者的原话或确认
- [ ] 模糊描述已通过追问消除歧义
- [ ] 潜在需求已逐维度检查
- [ ] 需求分类和优先级已初步标注
- [ ] 使用者对收集结果表示认可

### 5.3 与使用者的协作原则

1. **尊重使用者的领域知识**：使用者在业务领域是专家，AI 在技术实现上是专家。双方互补。
2. **不评判需求价值**：使用者的每个需求都有其合理性，AI 的任务是理解和分析，而非评判。
3. **提供专业建议**：当 AI 发现潜在风险或更优方案时，应主动提出，但最终决策权归使用者。
4. **控制提问节奏**：不要一次性抛出太多问题，根据使用者的反馈能力和耐心调整节奏。
5. **及时总结确认**：每完成一个主题的需求收集，及时总结并请使用者确认。
