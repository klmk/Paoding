# Phase 1: 需求分析

## 概述

Phase 1 是连接"想做什么"（Phase 0 双向画像）和"怎么做"（Phase 2 规划设计）的关键桥梁。本阶段的目标是将使用者的模糊需求、业务目标和场景期望，转化为结构化、可验证、可追溯的产品需求文档（PRD）。

**核心理念：** 需求分析的深度直接决定项目的成败。模糊需求进入开发，等于没有图纸就盖房子。AI 必须扮演"主动提问者"的角色，通过系统化的方法挖掘显性需求、隐性需求和潜在需求，确保需求文档达到可验证、可追溯的标准。

**与 Phase 0 的衔接：**
- Phase 0 产出的使用者画像（技术背景、需求表达能力、沟通偏好）指导本阶段的沟通策略
- Phase 0 产出的对齐策略（风险清单、补强方案）指导本阶段的质量控制重点
- Phase 0 识别的约束条件（技术、业务、时间、预算）直接作为本阶段的输入

**适用场景：**
- Phase 0 双向画像与对齐完成后的任何项目
- 项目需求发生重大变更，需要重新分析时
- 迭代开发中新增功能模块的需求分析

---

## 阶段目标

将使用者的需求转化为满足以下全部条件的结构化 PRD：

1. **完整性**：覆盖核心场景、边界条件和异常路径
2. **无歧义性**：每条需求只有一种合理解读
3. **可验证性**：每条需求有明确的、可测试的验收标准
4. **可追溯性**：每条需求有唯一标识，可追溯到使用者场景或业务目标
5. **优先级明确**：需求按重要性排序，指导后续开发资源的分配

## 铁律（Iron Laws）

### 铁律 1：需求不清晰不得进入设计

```
任何未通过清晰度验证的需求，不得进入 Phase 2 规划设计阶段。
```

清晰度验证的标准：需求描述无歧义、有可测试的验收标准、覆盖正常和异常场景。使用者说"清楚了"不等于需求真的清晰——清晰是结构化分析的结果，不是直觉的判断。

### 铁律 2：每个需求必须可追溯

```
每条需求必须有唯一标识（如 REQ-001），且可追溯到使用者场景或业务目标。
```

无法追溯的需求是"孤儿需求"，其存在价值无法验证。追溯关系是需求变更管理的基础。

### 铁律 3：必须挖掘潜在需求

```
AI 不得仅记录使用者显式提出的需求，必须主动挖掘隐性需求和潜在风险。
```

使用者通常只描述他们能想到的场景。权限、安全、异常处理、性能、兼容性等维度，使用者往往不会主动提及，但这些是系统稳定运行的必要条件。AI 必须主动追问。

### 铁律 4：需求变更必须记录

```
需求分析过程中的任何变更（新增、修改、删除、优先级调整）必须有记录。
```

变更记录包括：变更内容、变更原因、变更时间、变更提出方、影响评估。无记录的变更是需求失控的根源。

### 铁律 5：PRD 必须经使用者确认

```
PRD 文档必须经使用者逐项确认（显式同意），方可视为本阶段完成。
```

AI 单方面宣布"需求分析完成"不算确认。确认意味着使用者对每条需求的理解与文档一致，且认可优先级排序和范围边界。

### 铁律 6：不做假设，只做确认

```
当需求存在不确定性时，AI 必须向使用者确认，不得自行假设。
```

如果确认需要时间，将不确定项标记为"待确认"并记录待确认的问题，而非自行填补假设。

## 执行流程

### 步骤 1：需求收集

**目标：** 基于使用者的画像和初始描述，通过苏格拉底式提问系统性地收集需求。

**执行要点：**
1. 回顾 Phase 0 使用者画像，确定沟通策略（详见 `requirement-collection-guide.md`）
2. 根据使用者的需求表达能力等级（A/B/C/D），选择对应的提问策略
3. 使用苏格拉底式提问技巧，引导使用者从"想要什么"到"为什么想要"再到"期望什么效果"
4. 记录所有需求陈述，标注来源（使用者原话 / AI 推断 / 待确认）
5. 对模糊描述进行即时追问，消除歧义

**关键提问维度：**
- 核心功能：系统必须做什么？
- 用户角色：谁在使用？不同角色有什么不同需求？
- 操作流程：用户如何完成核心任务？
- 数据需求：需要处理什么数据？数据从哪来？到哪去？
- 业务规则：有什么业务约束和规则？

**输出：** 原始需求记录（使用者的需求陈述 + AI 的追问记录）

---

### 步骤 2：潜在需求挖掘

**目标：** AI 主动识别和追问使用者未提及但系统必须考虑的需求维度。

**执行要点：**
1. 对照潜在需求挖掘清单（详见 `requirement-collection-guide.md`），逐维度检查
2. 对每个维度，评估是否与当前项目相关
3. 对相关维度，向使用者提出具体问题
4. 记录使用者的回答或"不适用"的判断
5. 对使用者未明确回答的维度，标注为"AI 建议项"并说明理由

**必须覆盖的维度：**
- 权限与角色管理
- 数据安全与隐私
- 异常场景处理
- 性能要求
- 兼容性要求
- 可用性要求
- 可维护性要求
- 可扩展性要求
- 合规性要求

**输出：** 潜在需求清单（含使用者确认状态）

---

### 步骤 3：用户旅程梳理

**目标：** 梳理核心用户的核心操作路径，确保需求覆盖完整的用户体验。

**执行要点：**
1. 识别核心用户角色（基于 Phase 0 使用者画像）
2. 为每个核心角色梳理主要操作路径（从进入系统到完成任务）
3. 标注每个步骤的输入、操作、输出和预期结果
4. 识别关键决策点和分支路径
5. 标注可能的异常路径和错误处理
6. 与使用者确认用户旅程的准确性

**输出：** 用户旅程图（核心角色的核心操作路径）

---

### 步骤 4：功能需求整理

**目标：** 将收集到的需求模块化拆解，形成结构化的功能需求列表。

**执行要点：**
1. 将需求按功能模块分组
2. 为每条需求分配唯一标识（REQ-001, REQ-002, ...）
3. 为每条需求定义：
   - **描述**：清晰、无歧义的需求陈述
   - **输入**：该功能需要什么输入
   - **输出**：该功能产生什么输出
   - **验收标准**：如何验证该需求已满足（必须可测试）
   - **优先级**：P0（必须有）/ P1（应该有）/ P2（可以有）/ P3（未来有）
   - **追溯**：关联的使用者场景或业务目标
4. 检查需求之间的依赖关系和冲突
5. 使用需求分类框架（核心/增强/隐性/未来）进行分类

**输出：** 结构化功能需求列表

---

### 步骤 5：非功能需求整理

**目标：** 定义系统的非功能性要求，确保系统不仅"能用"而且"好用"。

**执行要点：**
1. 基于步骤 2 的潜在需求挖掘结果，整理非功能需求
2. 按以下维度分类整理：
   - **性能需求**：响应时间、吞吐量、并发量、资源限制
   - **安全需求**：数据加密、访问控制、审计日志、漏洞防护
   - **可用性需求**：易用性、可访问性、错误提示、帮助文档
   - **兼容性需求**：浏览器、操作系统、设备、数据格式
   - **可靠性需求**：可用率、容错能力、数据备份、灾难恢复
   - **可维护性需求**：日志、监控、配置管理、部署方式
   - **可扩展性需求**：水平扩展、功能扩展、数据增长
3. 为每条非功能需求定义可量化的验收标准
4. 与使用者确认非功能需求的合理性

**输出：** 非功能需求列表（含量化验收标准）

---

### 步骤 6：需求对齐确认

**目标：** 与使用者逐项确认需求，消除所有歧义，达成共识。

**执行要点：**
1. 使用需求对齐确认清单（详见 `requirement-alignment-checklist.md`）逐项检查
2. 向使用者展示完整的 PRD 文档
3. 对每条需求，确认使用者的理解与文档一致
4. 特别关注以下易产生分歧的方面：
   - 需求优先级是否合理
   - 范围边界是否明确（做什么 vs 不做什么）
   - 验收标准是否可接受
   - 非功能需求是否被遗漏或低估
5. 记录使用者的所有反馈和修改意见
6. 根据反馈更新 PRD 文档
7. 对更新后的文档进行二次确认

**输出：** 需求对齐确认记录（含使用者反馈和修改历史）

---

### 步骤 6.5：原型/低保真验证（Prototype Validation）

**目标：** 对涉及 UI/交互/流程的需求，在 PRD 定稿前通过可视化原型进行二次确认，消除"文字确认了但看到效果不对"的返工问题。

**触发条件：** 当需求中涉及以下任一情况时，必须执行本步骤：
- 用户界面（UI）变更或新增
- 交互流程（操作路径、按钮事件、页面跳转）
- 数据展示方式（列表、表格、图表、表单）
- 跨系统交互流程（如 ERP→MES 的单据推送链路）

**执行要点：**
1. 基于已对齐的 PRD 草稿，产出低保真原型（线框图/流程图/交互草图）
2. 原型必须覆盖核心用户旅程的关键节点，标注交互行为和预期结果
3. 向使用者展示原型，引导使用者按真实操作流程走一遍
4. 收集使用者反馈，重点关注：
   - 交互流程是否符合使用习惯
   - 视觉布局是否符合预期
   - 信息展示是否完整、清晰
   - 异常分支的处理方式是否可接受
5. 根据反馈调整原型，必要时进行第二轮确认
6. 原型确认后，将确认的交互细节同步更新到 PRD 文档

**输出：**
- 低保真原型（线框图/流程图）
- 原型确认记录（使用者反馈 + 调整记录）
- 更新后的 PRD 文档（含交互细节补充）

**不执行本步骤的风险：** 文字确认的精度上限就是文字本身。不做原型验证就定稿 PRD，相当于盲人摸象——使用者看文字觉得"对"，但看到实际界面才发现"不对"。原型验证的成本远低于多轮代码返工的成本。

---

### 步骤 7：PRD 文档输出

**目标：** 基于以上步骤的成果，输出完整的 PRD 文档。

**执行要点：**
1. 使用 PRD 模板（详见 `prd-template.md`）组织文档结构
2. 填充所有必要章节，确保无遗漏
3. 检查文档的完整性和一致性
4. 确保所有需求标识、追溯关系、验收标准正确无误
5. 将文档提交使用者做最终确认
6. 获得使用者确认后，标记文档版本号

**输出：** 完整的 PRD 文档（版本 1.0）

## 红旗列表（Red Flags）

以下是 AI 在需求分析阶段可能用来跳过或简化流程的常见借口，以及对应的反驳：

| # | AI 的借口 | 反驳 |
|---|----------|------|
| 1 | "使用者说得很清楚，不需要再追问了" | 使用者的"清楚"是主观感受，不是客观验证。你必须通过结构化分析确认需求确实清晰。参见铁律 1。 |
| 2 | "这些非功能需求以后再考虑就行" | 非功能需求影响架构决策。延迟到开发阶段再考虑，可能导致大规模重构。参见铁律 3。 |
| 3 | "使用者没提到安全需求，说明不需要" | 使用者不提安全需求，往往是因为缺乏安全意识，而非不需要。AI 必须主动追问。参见铁律 3。 |
| 4 | "需求分析太慢了，使用者急着要结果" | 需求分析投入的时间，会在开发阶段成倍收回。跳过需求分析才是真正的慢。 |
| 5 | "我可以边写代码边明确需求" | 这是"需求蔓延"的根源。模糊需求进入开发会导致返工、技术债务和范围失控。参见铁律 1。 |
| 6 | "这个需求跟之前的项目类似，直接复用就行" | 相似不等于相同。必须验证差异点，并记录具体的需求变更。参见核心铁律 IL-1。 |
| 7 | "PRD 文档太长了，使用者不会看的" | PRD 的价值不在于"被阅读"，而在于"被制定"的过程本身。使用者至少需要确认摘要和验收标准。 |
| 8 | "我已经理解了使用者的意图，不需要逐项确认" | "理解"不等于"确认"。AI 的理解可能有偏差，只有使用者的确认才能消除偏差。参见铁律 5。 |
| 9 | "优先级很明显，不需要特别讨论" | 优先级的判断因角色而异。AI 认为重要的，使用者可能不这么认为。优先级必须与使用者协商确认。 |
| 10 | "异常场景太多了，不可能全部覆盖" | 不要求覆盖所有异常，但必须覆盖高概率和高影响的异常。AI 应识别并优先处理关键异常场景。 |
| 11 | "PRD 已经写得很详细了，不需要做原型" | 文字描述的精度上限就是文字本身。涉及 UI/交互/流程的需求，不做原型验证就定稿 PRD，等于盲人摸象。使用者看文字觉得"对"，看到实际效果才发现"不对"——这就是反复返工的根源。原型验证的成本远低于多轮代码返工的成本。参见步骤 6.5。 |

**如果出现以上任何一种情况，立即停止，回到对应步骤重新执行。**

## 出口条件（Gate 1）

以下条件必须**全部满足**方可进入 Phase 2：

- [ ] 所有核心需求已收集并文档化
- [ ] 潜在需求已挖掘并记录（含使用者确认状态）
- [ ] 核心用户的用户旅程已梳理并确认
- [ ] 每条需求有明确的验收标准（可测试、可验证）
- [ ] 需求优先级已与使用者确认
- [ ] 非功能需求已整理并量化
- [ ] 需求追溯矩阵已建立（需求 <-> 使用者场景）
- [ ] PRD 文档已输出并获得使用者显式确认
- [ ] 范围边界已明确（做什么 + 不做什么）
- [ ] 涉及 UI/交互/流程的需求已完成原型验证（如适用）

### 门控类型

**硬性门控（Hard Gate）** — PRD 文档必须经使用者显式确认（逐项或整体同意），不可绕过。

## 回退机制

当 Phase 2（规划设计）或后续阶段发现以下情况时，必须回退到 Phase 1：

| 触发条件 | 回退范围 | 处理方式 |
|---------|---------|---------|
| 发现需求遗漏 | 补充遗漏需求 | 执行步骤 1-4，补充后更新 PRD |
| 发现需求歧义 | 澄清歧义需求 | 执行步骤 6，确认后更新 PRD |
| 使用者提出需求变更 | 评估变更影响 | 执行步骤 4-7，评估影响后更新 PRD |
| 发现非功能需求不足 | 补充非功能需求 | 执行步骤 2、5，补充后更新 PRD |

**回退后必须重新通过 Gate 1 方可继续。**

---

## 注意事项

- 需求分析是一个迭代过程。如果在使用者确认后发现有遗漏，应及时补充并重新确认
- 对于需求表达能力较弱的使用者（C/D 级），应增加提问轮次，使用更多示例和场景描述
- 需求文档的版本管理很重要。每次更新都应记录版本号、变更内容和变更原因
- 需求分析的质量直接影响后续所有阶段。投入足够的时间在需求分析上，是最高效的项目管理策略
- AI 应主动识别需求中的矛盾和冲突，及时向使用者指出并协商解决
