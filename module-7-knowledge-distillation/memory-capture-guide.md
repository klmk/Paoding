# 记忆捕获指南

> **定位：** 本文件定义了 Module 7 第一层"自动记忆"的具体机制。记忆捕获是知识蒸馏的输入源——没有高质量的原始记忆，就没有高质量的蒸馏认知。
>
> **核心理念：** 参考 claude-mem 和 agent-memory 的被动记忆机制，但更进一步——不仅记录"发生了什么"，还记录"为什么发生"和"意味着什么"，为后续的主动蒸馏提供高质量素材。

---

## 1. 需要捕获的知识类型

### 1.1 知识类型全景

| 类型 | 定义 | 蒸馏目标章节 | 优先级 | 示例 |
|------|------|-------------|--------|------|
| **决策** | 在多个方案中选择了其中一个，及其理由 | 架构认知 / 编码认知 | 高 | "选择 Zustand 而非 Redux，因为 API 更简洁" |
| **模式** | 项目中反复出现的代码结构或设计方式 | 编码认知 - 项目特定模式 | 高 | "所有 Service 层使用依赖注入" |
| **错误** | 开发过程中发生的 Bug、异常或不符合预期的行为 | 避坑认知 - 高频错误 | 高 | "ERR-012: 异步函数缺少 await 导致竞态条件" |
| **偏好** | 使用者对代码风格、工具选择、工作方式的个人倾向 | AI身份锚定 | 中 | "使用者偏好函数式编程风格，避免 class" |
| **约束** | 项目必须遵守的技术限制或业务限制 | 架构认知 / 业务认知 | 高 | "API 响应时间必须 < 200ms" |
| **业务规则** | 影响系统行为的业务逻辑规则 | 业务认知 - 业务规则速查 | 高 | "订单取消后 30 分钟内不可重新下单" |
| **陷阱** | 容易踩坑的代码模式或配置 | 编码认知 - 已知陷阱 | 高 | "dayjs 的 .diff() 返回毫秒，不是秒" |
| **发现** | 开发过程中的意外发现或洞察 | 避坑认知 / 编码认知 | 中 | "发现 PostgreSQL 的 JSONB 查询比关联查询快 3 倍" |
| **变更** | 需求、架构、技术栈的变更记录 | 进度认知 | 中 | "缓存方案从本地缓存改为 Redis" |

### 1.2 各类型的捕获深度

不同类型的知识需要不同深度的捕获：

| 类型 | 必须捕获 | 建议捕获 | 可选捕获 |
|------|---------|---------|---------|
| 决策 | 决策内容、选择结果 | 候选方案、选择理由 | 放弃方案的原因 |
| 模式 | 模式描述、使用场景 | 代码示例 | 模式的优势和局限 |
| 错误 | 错误表现、触发条件 | 根因分析、修复方式 | 预防措施（蒸馏时提炼） |
| 偏好 | 偏好内容 | 偏好原因 | 偏好的边界条件 |
| 约束 | 约束内容 | 约束原因 | 违反约束的后果 |
| 业务规则 | 规则内容、触发条件 | 例外情况、关联需求 | 规则的历史变更 |
| 陷阱 | 陷阱描述、触发条件 | 正确做法 | 错误示例 |
| 发现 | 发现内容 | 发现场景 | 发现的验证方式 |
| 变更 | 变更内容 | 变更原因 | 变更的影响范围 |

## 2. 捕获时机

### 2.1 自动捕获时机

以下时机 AI 应自动触发记忆捕获，无需使用者显式要求：

| 时机 | 捕获类型 | 捕获方式 |
|------|---------|---------|
| **任务完成时** | 决策、模式、发现 | 任务完成总结中自动提取 |
| **Bug 修复时** | 错误、陷阱 | 修复完成后自动记录错误详情 |
| **设计变更时** | 决策、约束、变更 | 变更讨论中自动捕获 |
| **代码审查时** | 模式、陷阱、偏好 | 审查过程中识别并记录 |
| **需求讨论时** | 业务规则、约束 | 讨论中提取关键规则 |
| **架构决策时** | 决策、约束 | 决策确定后立即记录 |
| **环境配置时** | 陷阱、约束 | 配置过程中记录踩坑点 |
| **依赖安装时** | 陷阱、发现 | 安装过程中记录版本兼容性问题 |

### 2.2 捕获流程

```
捕获触发
  │
  ▼
判断知识类型（决策/模式/错误/...）
  │
  ▼
按对应模板记录原始记忆
  │
  ├── 包含敏感信息？ → 执行敏感信息过滤（见第 4 节）
  │
  ▼
标注来源信息（时间、Phase、关联任务）
  │
  ▼
保存到原始记忆存储
  │
  ▼
（可选）如果是高优先级知识，立即触发实时蒸馏
```

### 2.3 捕获的自动化程度

| 捕获场景 | 自动化程度 | 说明 |
|---------|-----------|------|
| Bug 修复完成 | 全自动 | AI 自动记录错误详情，无需使用者确认 |
| 代码审查发现问题 | 全自动 | AI 自动记录审查中发现的模式问题 |
| 任务完成总结 | 半自动 | AI 提取知识点，使用者确认后保存 |
| 需求讨论 | 半自动 | AI 识别业务规则，使用者确认准确性 |
| 架构决策 | 手动确认 | AI 记录决策内容，使用者确认理由和选择 |
| 使用者偏好 | 手动触发 | 使用者显式告知偏好，AI 记录 |

## 4. 敏感信息过滤

### 4.1 必须过滤的信息

以下类型的信息**绝对不可**记录到原始记忆中：

| 类型 | 示例 | 原因 |
|------|------|------|
| 密码 | 数据库密码、API 密钥、SSH 密钥 | 安全风险 |
| 密钥 | JWT Secret、加密密钥、签名密钥 | 安全风险 |
| 令牌 | OAuth Token、Session Token、API Token | 安全风险 |
| 个人信息 | 用户手机号、身份证号、银行卡号 | 隐私保护 |
| 内部 URL | 内部服务地址、管理后台地址（非公开的） | 安全风险 |
| 商业机密 | 定价策略、客户名单、合同金额 | 商业保护 |

### 4.2 过滤规则

```
记录原始记忆时，必须执行以下检查：

1. 扫描是否包含密码、密钥、令牌等敏感信息
   → 如包含，替换为 [已过滤：敏感信息]

2. 扫描是否包含个人信息（手机号、身份证号等）
   → 如包含，脱敏处理（如 138****1234）

3. 扫描是否包含内部 URL 或非公开地址
   → 如包含，替换为 [已过滤：内部地址]

4. 扫描是否包含商业机密
   → 如包含，评估是否需要记录
   → 如必须记录，使用模糊描述替代具体数值

5. 扫描是否包含使用者明确要求不记录的内容
   → 如包含，跳过记录
```

### 4.3 可以记录的信息

以下信息可以安全记录：

| 类型 | 示例 | 说明 |
|------|------|------|
| 技术决策 | "选择 PostgreSQL 而非 MySQL" | 不涉及敏感信息 |
| 代码模式 | "使用 Repository 模式封装数据访问" | 不涉及敏感信息 |
| 错误模式 | "N+1 查询导致性能问题" | 不涉及敏感信息 |
| 业务规则 | "订单超时 30 分钟自动取消" | 不涉及敏感信息 |
| 架构信息 | "系统采用微服务架构，5 个服务" | 不涉及敏感信息 |
| 依赖信息 | "使用 Redis 7.x 做缓存" | 不涉及敏感信息 |
| 编码规范 | "函数不超过 30 行" | 不涉及敏感信息 |

## 6. 与 claude-mem / agent-memory 的对比

| 维度 | claude-mem | agent-memory | Module 7 记忆捕获 |
|------|-----------|-------------|-----------------|
| 记忆方式 | 被动记录（发生了什么） | 被动记录（发生了什么） | 主动捕获（发生了什么 + 为什么 + 意味着什么） |
| 记忆结构 | 自由文本 | 键值对 | 结构化模板（按知识类型分类） |
| 知识提炼 | 无 | 无 | 主动蒸馏（提炼为专家级认知） |
| 来源追溯 | 无 | 无 | 每条记忆有完整来源标注 |
| 敏感信息过滤 | 无明确机制 | 无明确机制 | 强制敏感信息过滤 |
| 与项目框架集成 | 独立工具 | 独立工具 | 深度集成 AI-Dev-Framework |
| 知识成长 | 无 | 无 | 持续成长（错题本 -> 补强规则 -> project-brain） |
| 面向读者 | 人类 | 人类 | AI（AI 写给 AI 看） |

**Module 7 的独特价值：**

1. **从"被动记忆"到"主动蒸馏"**：不仅记录发生了什么，更提炼出专家级认知
2. **从"自由文本"到"结构化模板"**：按知识类型分类，便于检索和蒸馏
3. **从"无追溯"到"全追溯"**：每条知识都有来源，区分真实知识和幻觉
4. **从"静态积累"到"持续成长"**：通过蒸馏闭环，让知识随项目推进自动成长

## 7. 与 Phase 4 知识沉淀模块的关系

### 7.1 定位差异

| 维度 | Phase 4 知识沉淀 | Module 7 记忆捕获 |
|------|-----------------|-----------------|
| 定位 | 阶段性总结 | 持续性捕获 |
| 触发时机 | Phase 3 完成后 | 贯穿全生命周期 |
| 产出物 | 回顾报告、错题本、project-context.md | 原始记忆（.ai-memory/） |
| 知识粒度 | 宏观（项目级别） | 微观（任务级别） |
| 时间跨度 | 回顾过去一个 Phase | 实时捕获当前任务 |

### 7.2 协作关系

```
Module 7 记忆捕获（持续运行）
  │
  │  产出：原始记忆（.ai-memory/）
  │
  ├──→ 实时蒸馏 → project-brain.md（持续更新）
  │
  └──→ Phase 4 知识沉淀（阶段性）
         │
         │  输入：原始记忆 + Phase 0-3 文档
         │  产出：回顾报告、错题本、project-context.md
         │
         └──→ 反哺 Module 7
                │
                │  回顾报告中的问题模式 → 补充原始记忆
                │  错题本中的错误案例 → 补充原始记忆
                │  project-context.md → 作为 project-brain.md 的初始化素材
                │
                └──→ 下一轮蒸馏
```

### 7.3 具体协作场景

| 场景 | Phase 4 的角色 | Module 7 的角色 |
|------|--------------|----------------|
| 日常开发 | 不参与 | 持续捕获原始记忆 + 实时蒸馏 |
| Phase 完成 | 不参与 | 阶段蒸馏，更新 project-brain.md |
| 项目结束 | 全面回顾，生成知识文档 | 里程碑蒸馏，压缩 project-brain.md |
| 沙盒重置 | 保存 project-context.md | 保存 project-brain.md + 原始记忆 |
| 新项目启动 | 加载上一个项目的 project-context.md | 加载上一个项目的 project-brain.md |

## 8. 记忆捕获的质量标准

### 8.1 完整性检查

| 检查项 | 通过标准 |
|--------|---------|
| 元数据完整 | 记忆ID、类型、时间、阶段、关联任务全部填写 |
| 内容完整 | 按对应模板填写了所有"必须捕获"字段 |
| 来源清晰 | 能追溯到具体的任务、文件或对话 |

### 8.2 准确性检查

| 检查项 | 通过标准 |
|--------|---------|
| 事实准确 | 记录的内容与实际情况一致 |
| 无主观臆断 | 不包含 AI 的猜测或推断（除非明确标注为"AI 推断"） |
| 无敏感信息 | 通过敏感信息过滤检查 |

### 8.3 可用性检查

| 检查项 | 通过标准 |
|--------|---------|
| 可蒸馏 | 内容足够具体，能提炼为蒸馏知识 |
| 可检索 | 通过类型分类和索引能快速找到 |
| 可追溯 | 来源信息完整，能追溯到原始上下文 |

## 9. 注意事项

- 记忆捕获是蒸馏的前置步骤。高质量的原始记忆是高质量蒸馏的基础
- 不要过度捕获。不是所有信息都值得记录，优先捕获高优先级知识
- 捕获时追求"够用"而非"完美"。可以在蒸馏阶段补充完善
- 原始记忆是"草稿"，允许不完美。蒸馏后的 project-brain.md 才是"成品"
- 敏感信息过滤是不可妥协的底线。宁可漏记，不可泄露
- 记忆索引必须及时更新。每次新增或蒸馏记忆后，更新 index.md
