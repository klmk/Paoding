# Module 7: 项目知识蒸馏体系

## 概述

Module 7 是 AI-Dev-Framework 的原创知识蒸馏模块，独立于 Phase 0-5 的线性流程，贯穿项目全生命周期持续运行。其核心使命是解决 AI 在长期开发中的两大顽疾——**失忆**（沙盒清除、会话重启后丢失上下文）和**幻觉**（基于猜测而非真实知识生成内容），让 AI 通过持续的知识蒸馏成长为项目资深专家。

**核心问题：**

| 问题 | 表现 | 现有方案的不足 |
|------|------|--------------|
| **失忆** | 沙盒清除后会话重启，AI 丢失所有项目上下文，需要从零开始 | claude-mem（17K星）、agent-memory 只做"被动记忆"——记录发生了什么，但不会主动提炼认知 |
| **幻觉** | AI 在缺乏上下文时凭猜测生成代码或决策，产生看似合理但实际错误的输出 | BMad 的 project-context.md 是静态快照，不会随项目推进自动成长 |
| **认知断层** | 新会话的 AI 无法继承前序会话积累的经验和教训 | 缺少"错题本 -> 补强规则 -> 注入认知"的闭环机制 |

**我们的方案——三层架构：**

```
第一层：自动记忆（Capture）     捕获每次会话的关键操作和决策
         │
         ▼
第二层：主动蒸馏（Distill）     从原始记忆中提炼专家级认知，生成 project-brain.md
         │
         ▼
第三层：持续成长（Grow）        错题本 -> 补强规则 -> 注入 project-brain，能力画像跟踪
```

**与 Phase 4 知识沉淀模块的关系：**

- Phase 4 是**阶段性总结**——在项目/迭代结束时做一次性的知识沉淀
- Module 7 是**持续性蒸馏**——贯穿项目全生命周期，实时捕获、主动提炼、持续成长
- Phase 4 的产出（回顾报告、错题本、project-context.md）是 Module 7 蒸馏的**重要输入源**
- Module 7 的 project-brain.md 是 Phase 4 project-context.md 的**进化版**——从静态快照升级为持续成长的活文档

**适用场景：**

- 任何需要 AI 持续参与开发的项目
- 长期项目（跨多个会话、沙盒重置）
- 多 AI 协作场景（需要共享项目认知）
- 对 AI 输出质量有高要求的场景（需要减少幻觉）

## 核心目标

| 目标 | 描述 | 验证方式 |
|------|------|---------|
| 消除失忆 | 新会话的 AI 加载 project-brain.md 后，能在 5 分钟内恢复项目上下文 | 新会话启动测试：AI 能准确回答项目相关问题 |
| 消除幻觉 | AI 的所有输出都有知识来源追溯，不凭猜测生成 | 输出中包含来源标注，无来源标注的内容比例 < 5% |
| 专家级认知 | project-brain.md 中的知识达到资深开发者的认知水平 | 知识覆盖业务、架构、编码、避坑四个维度 |
| 持续成长 | project-brain.md 随项目推进自动成长，不依赖人工维护 | 每个蒸馏周期后，project-brain.md 有实质性更新 |
| 错误免疫 | 同样的错误不犯第二次 | 错题本中已补强的错误在后续开发中零重复 |

## 三层架构详解

### 第一层：自动记忆（Capture）

**职责：** 在每次会话中自动捕获关键操作和决策，形成原始记忆。

**捕获内容：**

| 类型 | 示例 | 捕获方式 |
|------|------|---------|
| 决策记录 | "选择 Redis 做缓存而非 Memcached，因为项目需要持久化" | 任务完成时自动记录 |
| 错误记录 | "ERR-012: 异步函数缺少 await 导致竞态条件" | Bug 修复时自动记录 |
| 模式识别 | "项目中所有 API 响应统一使用 Result<T> 包装" | 代码审查时识别 |
| 业务规则 | "订单取消后 30 分钟内不可重新下单" | 需求讨论时捕获 |
| 偏好记录 | "使用者偏好函数式编程风格" | 交互中识别 |

**详细指南：** 参见 [`memory-capture-guide.md`](./memory-capture-guide.md)

---

### 第二层：主动蒸馏（Distill）

**职责：** 从第一层的原始记忆中提炼专家级认知，生成 project-brain.md。

**蒸馏的本质：** 不是简单的信息搬运，而是从"发生了什么"提炼出"意味着什么"。如同人类专家从大量经验中提炼出直觉和判断力。

**蒸馏输入 -> 蒸馏输出 示例：**

| 原始记忆（输入） | 蒸馏认知（输出） |
|-----------------|-----------------|
| "ERR-005: 用户表联合查询超时" | **避坑规则：** 用户表与订单表联合查询在数据量 > 10万时必须走缓存，直接查询会导致 > 3s 超时 |
| "ADR-003: 选择 GraphQL 而非 REST" | **架构认知：** 项目使用 GraphQL，因为前端需要灵活的字段查询能力；所有新接口必须用 GraphQL 实现 |
| "使用者说：这个按钮颜色不对" | **业务认知：** 主按钮必须使用品牌色 #1890FF，不可使用其他颜色 |

**详细指南：** 参见 [`distillation-process-guide.md`](./distillation-process-guide.md)

---

### 第三层：持续成长（Grow）

**职责：** 基于错题本和补强规则，持续提升 AI 的能力水平。

**成长闭环：**

```
开发中犯错
    │
    ▼
记录到错题本（ERR-XXX）
    │
    ▼
提炼补强规则（RULE-XXX）
    │
    ▼
注入 project-brain.md 的"避坑认知"章节
    │
    ▼
下次遇到类似场景，AI 主动应用补强规则
    │
    ▼
验证补强规则是否有效
    │
    ▼
有效 → 保留 / 无效 → 修订
```

**能力画像跟踪：**

- 每次蒸馏后更新 AI 在各维度上的能力评分
- 识别薄弱环节，制定针对性补强策略
- 追踪能力成长趋势

### 第三层补充：形式化测试 + 错题本联动机制

**职责：** 将形式化工程测试发现的缺陷与错题本联动，实现"发现一批 → 识别模式 → 提炼规则 → 批量解决"的高效修复闭环。

**联动流程：**

```
形式化工程测试
    │
    ├── 发现 N 个隐性缺陷
    │       │
    │       ▼
    │   缺陷分类统计：
    │   - 类别 A：X 个（Y%）
    │   - 类别 B：Z 个（W%）
    │   - ...
    │       │
    │       ▼
    │   识别高频模式（占比 > 30% 的类别）
    │       │
    │       ▼
    │   提炼补强规则（RULE-XXX）
    │   "所有外部接口入参必须做空值校验"
    │       │
    │       ▼
    │   全局扫描：用补强规则扫描全部代码
    │   发现更多同类潜在问题 → 批量修复
    │       │
    │       ▼
    │   注入 project-brain.md → 后续开发自动遵循
    │
    └── 输出缺陷报告 → 使用者决策 → 按优先级修复
```

**关键原则：**
- 形式化测试负责"广度发现"（发现所有隐性问题）
- 错题本负责"深度根治"（防止同类问题再犯）
- 两者联动，不是二选一
- 同类缺陷出现 2 次以上，必须提炼为补强规则并全局扫描

## 铁律（Iron Laws）

以下铁律在 Module 7 中具有最高约束力，不可违反、不可绕过、不可降级。

### 铁律 M7-1：每个 Phase 结束后必须蒸馏

```
Phase 0 完成后 → 蒸馏：AI身份锚定 + 业务认知
Phase 1 完成后 → 蒸馏：业务认知 + 业务规则速查
Phase 2 完成后 → 蒸馏：架构认知 + 编码认知
Phase 3 完成后 → 蒸馏：避坑认知 + 进度认知
Phase 4 完成后 → 蒸馏：全面重新蒸馏压缩
Phase 5 完成后 → 蒸馏：里程碑蒸馏 + 能力画像更新
```

- 蒸馏不是"有空再做"的事，而是每个 Phase 的必要收尾动作
- 蒸馏产出必须包含：新增认知、更新认知、废弃认知（标注过时内容）
- 蒸馏完成后必须更新 project-brain.md 的"最后蒸馏时间"

### 铁律 M7-2：project-brain.md 必须 AI 写给 AI 看

```
project-brain.md 的读者是 AI，不是人类。

这意味着：
1. 使用 AI 最容易理解的结构化格式（表格、列表、代码块）
2. 每条知识都有明确的触发条件（"在什么情况下需要这条知识"）
3. 每条知识都有可验证的判断标准（"如何判断这条知识是否适用"）
4. 避免模糊描述（"注意性能" → "查询超过 1000 条记录时必须分页"）
5. 知识密度优先于可读性（精炼 > 详细）
```

- project-brain.md 不是给人看的文档，是给 AI 加载的"认知模块"
- 每条知识都应能让 AI 在 3 秒内判断"这条知识是否与当前任务相关"
- 禁止在 project-brain.md 中放置只有人类才能理解的描述

### 铁律 M7-3：知识必须有来源追溯

```
每条蒸馏知识必须标注来源：
- 来源类型：PRD / ADR / ERR-XXX / RULE-XXX / 代码审查 / 使用者反馈 / Phase X
- 来源编号：具体的文档ID或错误编号
- 蒸馏时间：提炼这条知识的时间
- 蒸馏者：AI（自动）或使用者（手动）

没有来源追溯的知识 = 幻觉风险 = 不可信。
```

- 来源追溯是区分"真实知识"和"AI幻觉"的唯一标准
- 如果知识来源不明确，标注为"来源待确认"并在下次蒸馏时补充
- 定期审查来源标注，移除无法追溯来源的知识

### 铁律 M7-4：蒸馏必须精炼

```
蒸馏的目标是压缩信息密度，而非搬运信息。

好的蒸馏：  "用户表联合查询 > 10万行时必须走缓存（来自 ERR-005）"
差的蒸馏：  "在开发过程中我们发现用户表和订单表联合查询的时候如果数据量超过10万行会导致查询超时，所以后来我们加了Redis缓存来解决这个问题（来自 ERR-005）"

好的蒸馏用一句话表达一个知识点。
差的蒸馏用一段话描述一个事件。
```

- 每条蒸馏知识的理想长度：1-3 句话
- 超过 5 句话的知识点应考虑拆分或进一步提炼
- 蒸馏不是写日记，是写"认知规则"

### 铁律 M7-5：过时知识必须及时清理

```
知识有保质期。过时的知识比没有知识更危险。

清理规则：
1. 技术栈变更 → 相关架构认知和编码认知必须更新
2. 业务规则变更 → 相关业务认知必须更新
3. 依赖升级 → 相关避坑认知必须重新验证
4. 架构重构 → 全面重新蒸馏

过时知识不清理 = AI 基于错误认知做决策 = 比失忆更严重的问题。
```

- 每次蒸馏时必须检查已有知识的时效性
- 过时知识标注为"已废弃"但保留记录（用于追溯）
- 重大变更后触发"里程碑蒸馏"进行全面重新蒸馏

### 铁律 M7-6：project-brain.md 大小必须可控

```
project-brain.md 是 AI 加载到上下文中的文件，大小直接影响 AI 的理解效率。

大小控制规则：
- 总大小上限：8000 字（约 4000 tokens）
- 单章节上限：1500 字
- 单条知识上限：100 字
- 超过上限时必须进行压缩蒸馏（提炼精华、移除冗余、合并同类项）

8000 字是 AI 能高效理解并应用的上限。超过这个限制，AI 的知识检索效率会显著下降。
```

- 当 project-brain.md 接近上限时，优先移除低价值知识
- 低价值知识归档到 project-brain-archive.md（不加载到上下文，但保留可检索）
- 里程碑蒸馏的核心目标之一就是"压缩重蒸馏"

## 执行流程

### 步骤 1：会话启动 — 加载认知

**目标：** 新会话启动时，加载 project-brain.md 恢复项目上下文。

**执行要点：**

1. 检查 project-brain.md 是否存在
2. 如果存在，加载全部内容到上下文
3. 检查"最后蒸馏时间"，判断知识的新鲜度
4. 如果知识超过 7 天未蒸馏，标记为"知识可能过时"
5. 确认 AI 身份锚定，明确当前角色和约束

**输出：** AI 已恢复项目上下文，可以开始工作

---

### 步骤 2：任务执行 — 实时记忆

**目标：** 在任务执行过程中，实时捕获关键操作和决策。

**执行要点：**

1. 任务开始前，扫描 project-brain.md 中的相关认知
2. 任务执行中，遵循已蒸馏的规则和约束
3. 任务完成时，自动捕获本次任务的关键知识点
4. 如果发生错误，记录到原始记忆（待蒸馏）

**输出：** 原始记忆记录（未蒸馏）

**详细指南：** 参见 [`memory-capture-guide.md`](./memory-capture-guide.md)

---

### 步骤 3：任务完成 — 实时蒸馏

**目标：** 每个任务完成后，立即提炼知识点。

**执行要点：**

1. 从本次任务的原始记忆中识别可蒸馏的知识点
2. 按知识类型分类（业务/架构/编码/避坑）
3. 提炼为精炼的认知规则（遵循铁律 M7-4）
4. 标注来源追溯（遵循铁律 M7-3）
5. 更新 project-brain.md 对应章节

**输出：** project-brain.md 更新（增量）

**详细指南：** 参见 [`distillation-process-guide.md`](./distillation-process-guide.md)

---

### 步骤 4：Phase 完成 — 阶段蒸馏

**目标：** 每个 Phase 结束时，进行系统化的阶段蒸馏。

**执行要点：**

1. 收集本 Phase 的全部原始记忆
2. 与 project-brain.md 现有知识进行对比
3. 识别新增认知、更新认知、废弃认知
4. 执行蒸馏（提炼、压缩、合并）
5. 更新 project-brain.md 全部相关章节
6. 更新"最后蒸馏时间"和版本号
7. 检查 project-brain.md 大小（遵循铁律 M7-6）

**输出：** project-brain.md 阶段更新版

---

### 步骤 5：里程碑到达 — 里程碑蒸馏

**目标：** 项目关键节点（如架构重构、技术栈升级、重大版本发布）进行全面重新蒸馏压缩。

**执行要点：**

1. 全面审查 project-brain.md 中的每一条知识
2. 验证知识的时效性和正确性
3. 移除过时知识，归档到 project-brain-archive.md
4. 压缩冗余知识，合并同类项
5. 重新组织知识结构（如有必要）
6. 更新能力画像
7. 生成蒸馏报告（本次蒸馏的变更摘要）

**输出：** project-brain.md 里程碑版本 + 蒸馏报告

---

### 步骤 6：会话结束 — 保存与交接

**目标：** 会话结束前，确保所有知识已保存。

**执行要点：**

1. 确认本次会话的所有原始记忆已捕获
2. 确认所有增量蒸馏已更新到 project-brain.md
3. 保存 project-brain.md 到永久存储位置
4. 如果是 Phase 结束，执行阶段蒸馏（步骤 4）
5. 生成会话知识交接摘要

**输出：** 已保存的 project-brain.md + 会话知识交接摘要

## 红旗列表

以下是 AI 在 Module 7 执行过程中可能用来跳过或简化流程的常见借口，以及对应的反驳：

| # | AI 的借口（红旗） | 反驳 |
|---|------------------|------|
| 1 | "这个任务太简单了，没什么值得蒸馏的" | 简单任务也可能暴露模式。即使是"确认了某个API的参数格式"，也是值得蒸馏的编码认知。参见铁律 M7-1。 |
| 2 | "project-brain.md 已经很完善了，不需要更新" | 项目在推进，知识在变化。即使 project-brain.md 看起来完善，每次任务后也可能有微调或验证的机会。 |
| 3 | "这条知识没有明确的来源，但我知道是对的" | "知道是对的"本身就需要来源验证。没有来源追溯的知识可能是幻觉。标注为"来源待确认"，下次蒸馏时补充。参见铁律 M7-3。 |
| 4 | "蒸馏太费时间了，直接把原始记忆存下来就行" | 原始记忆堆积 = 信息噪声。未经蒸馏的记忆在下次加载时无法高效利用。蒸馏的核心价值在于压缩信息密度。 |
| 5 | "project-brain.md 写给人看的也行，反正都是 Markdown" | 人看的文档追求可读性和完整性，AI 看的认知文件追求触发条件和判断标准。两者目标不同，格式也不同。参见铁律 M7-2。 |
| 6 | "过时的知识先留着吧，万一以后用得上" | 过时知识不清理，AI 会基于错误认知做决策。归档到 archive 文件，不要留在活跃的 project-brain.md 中。参见铁律 M7-5。 |
| 7 | "project-brain.md 超过大小限制了，加个新章节就行" | 超过大小限制必须压缩，而不是继续膨胀。优先移除低价值知识，合并同类项。参见铁律 M7-6。 |
| 8 | "这个错误已经修了，不需要记录到错题本" | 修复错误只解决了当前问题。记录错误并提炼补强规则，才能防止未来重犯。Module 7 的核心价值就在于此。 |
| 9 | "蒸馏是 Phase 4 的事，开发阶段不用管" | Phase 4 是阶段性总结，Module 7 是持续性蒸馏。开发阶段的实时蒸馏能防止知识在长会话中丢失。两者互补。 |
| 10 | "我已经记住这些知识了，不需要写到文件里" | AI 的"记住"仅限当前会话。会话结束后，所有未持久化的知识都会丢失。必须写入文件才能跨会话传承。 |

**如果出现以上任何一种情况，立即停止，回到对应步骤重新执行。**

## 注意事项

- Module 7 是持续性运行的模块，不是一次性任务。每个会话都应执行"加载 -> 记忆 -> 蒸馏 -> 保存"的循环
- project-brain.md 是 Module 7 的核心产物，也是 AI-Dev-Framework 中最重要的"活文档"
- 蒸馏的质量直接影响 AI 在后续会话中的表现。高质量的蒸馏 = 高效的上下文恢复 = 更少的幻觉
- 蒸馏是 AI 的自主行为，不需要使用者显式触发。但使用者可以随时要求查看或调整 project-brain.md
- project-brain.md 的版本管理使用简单的版本号（v1.0, v1.1, v2.0），重大变更升级主版本号，增量更新升级次版本号
