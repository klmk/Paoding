# 蒸馏流程指南

> **定位：** 本文件定义了 Module 7 知识蒸馏的具体流程、原则和质量标准。蒸馏是从原始记忆中提炼专家级认知的核心过程，是 project-brain.md 持续成长的驱动力。
>
> **核心原则：** 蒸馏不是信息搬运，是认知提炼。从"发生了什么"提炼出"意味着什么"。

---

## 1. 蒸馏的三种级别

### 1.1 实时蒸馏（Real-time Distillation）

**触发时机：** 每个任务完成后立即执行。

**目标：** 将本次任务中产生的关键知识点即时提炼并注入 project-brain.md。

**执行流程：**

```
任务完成
  │
  ▼
扫描本次任务的原始记忆
  │
  ▼
识别可蒸馏的知识点（3 分钟内完成）
  │
  ├── 有知识点 → 提炼 + 标注来源 → 更新 project-brain.md 对应章节
  │
  └── 无知识点 → 跳过（不强制蒸馏）
```

**蒸馏范围：**

| 知识类型 | 蒸馏到 | 示例 |
|---------|--------|------|
| 新发现的编码模式 | 编码认知 - 项目特定模式 | "项目中所有表单校验使用 zod schema" |
| 新遇到的错误 | 避坑认知 - 高频错误 | "ERR-015: zod schema 的 .refine() 不支持异步校验" |
| 新确认的业务规则 | 业务认知 - 业务规则速查 | "VIP 用户免运费门槛从 99 元调整为 49 元" |
| 新发现的陷阱 | 编码认知 - 已知陷阱 | "dayjs 的 .diff() 返回毫秒差，不是秒差" |

**注意事项：**

- 实时蒸馏追求速度，每个知识点提炼时间不超过 3 分钟
- 不要求全面，只提炼"最有价值"的 1-3 个知识点
- 如果任务很常规且没有新发现，可以跳过实时蒸馏
- 实时蒸馏后更新 project-brain.md 的"最后蒸馏时间"

---

### 1.2 阶段蒸馏（Phase Distillation）

**触发时机：** 每个 Phase（Phase 0-5）结束时执行。

**目标：** 系统化地蒸馏本 Phase 产生的全部知识，全面更新 project-brain.md。

**执行流程：**

```
Phase 完成
  │
  ▼
收集本 Phase 的全部原始记忆
  │
  ▼
与 project-brain.md 现有知识逐条对比
  │
  ├── 新增认知（本 Phase 新发现的知识）
  │     → 提炼 + 标注来源 → 写入对应章节
  │
  ├── 更新认知（已有知识需要修正或补充）
  │     → 修改 + 标注新来源 + 更新版本号
  │
  └── 废弃认知（已过时或被证伪的知识）
        → 标注"已废弃" + 记录废弃原因
  │
  ▼
检查 project-brain.md 大小
  │
  ├── 未超限 → 完成
  │
  └── 超过 8000 字 → 执行压缩蒸馏
        → 移除低价值知识到 archive
        → 合并同类知识
        → 重新精炼表述
  │
  ▼
更新元数据（版本号、蒸馏时间、蒸馏类型）
  │
  ▼
写入蒸馏日志
```

**各 Phase 的蒸馏重点：**

| Phase | 蒸馏重点 | 主要更新章节 |
|-------|---------|-------------|
| Phase 0 | AI 身份、使用者偏好、项目约束 | 1. AI身份锚定 |
| Phase 1 | 业务描述、业务流程、业务规则、术语 | 2. 业务认知 |
| Phase 2 | 技术栈、架构、数据模型、接口 | 3. 架构认知 |
| Phase 3 | 编码规范、项目模式、陷阱、错误 | 4. 编码认知 + 5. 避坑认知 |
| Phase 4 | 全面蒸馏压缩、错题本整合、能力画像 | 全部章节 |
| Phase 5 | 里程碑蒸馏、最终能力画像、交付知识 | 全部章节 |

**注意事项：**

- 阶段蒸馏必须在 Phase 出口条件满足后、进入下一个 Phase 前完成
- 阶段蒸馏后版本号升级次版本号（如 v1.0 → v1.1）
- 如果本 Phase 没有产生新知识，仍需检查已有知识的时效性

### 1.3 里程碑蒸馏（Milestone Distillation）

**触发时机：** 项目关键节点，包括但不限于：

- 架构重构完成后
- 技术栈升级完成后
- 重大版本发布前
- 项目阶段性交付时
- project-brain.md 大小接近上限时

**目标：** 全面重新蒸馏压缩，确保 project-brain.md 的知识完全准确、精炼、最新。

**执行流程：**

```
里程碑到达
  │
  ▼
全面审查 project-brain.md 中的每一条知识
  │
  ▼
逐条验证知识的时效性和正确性
  │
  ├── 仍然有效 → 保留（可能需要重新精炼表述）
  │
  ├── 需要更新 → 修正内容 + 更新来源
  │
  ├── 已过时 → 标注"已废弃" + 移至 archive
  │
  └── 无法验证 → 标注"来源待确认" + 保留但降权
  │
  ▼
压缩蒸馏
  │
  ├── 合并同类知识（多条相似知识合并为一条）
  ├── 移除低价值知识（使用频率低、适用范围窄）
  ├── 重新精炼表述（确保每条知识 1-3 句话）
  └── 重新组织结构（如有必要）
  │
  ▼
更新能力画像
  │
  ▼
生成蒸馏报告
  │
  ▼
版本号升级主版本号（如 v1.5 → v2.0）
```

**蒸馏报告模板：**

```markdown
# 蒸馏报告 - [项目名称] - [日期]

## 概要
- 蒸馏类型：里程碑蒸馏
- 触发原因：[如"架构重构完成"或"v2.0 发布前"]
- 版本变更：v[X.X] → v[X.X]

## 变更统计
- 新增知识：[X] 条
- 更新知识：[X] 条
- 废弃知识：[X] 条
- 合并知识：[X] 组（[X] 条合并为 [X] 条）
- 压缩率：原文 [X] 字 → 蒸馏后 [X] 字（压缩 [X]%）

## 废弃知识清单
| 知识摘要 | 废弃原因 | 替代知识 |
|---------|---------|---------|
| [待填写] | [待填写] | [待填写] |

## 能力画像更新
| 维度 | 蒸馏前 | 蒸馏后 | 变化说明 |
|------|--------|--------|---------|
| [待填写] | [待填写] | [待填写] | [待填写] |

## 待确认知识
| 知识摘要 | 待确认原因 | 建议验证方式 |
|---------|-----------|-------------|
| [待填写] | [待填写] | [待填写] |
```

## 2. 蒸馏原则

### 2.1 AI 写给 AI 看

project-brain.md 的读者是 AI，不是人类。蒸馏时必须遵循以下格式原则：

| 原则 | 说明 | 正确示例 | 错误示例 |
|------|------|---------|---------|
| 触发条件优先 | 每条知识先写"什么时候用这条" | "编写 async 函数时：所有异步调用必须 await" | "注意异步编程的正确方式" |
| 可判断性 | AI 能在 3 秒内判断知识是否与当前任务相关 | "涉及多表写入时必须使用事务" | "数据库操作要注意事务" |
| 精炼表述 | 一句话表达一个知识点 | "日期比较统一使用 UTC" | "在处理日期的时候要注意时区问题，建议统一使用 UTC 进行比较" |
| 结构化格式 | 使用表格、列表、代码块 | 表格形式列出规则 | 大段文字描述 |

### 2.2 越精炼越好

蒸馏的目标是压缩信息密度。衡量蒸馏质量的指标：

| 指标 | 目标 | 衡量方式 |
|------|------|---------|
| 单条知识长度 | 1-3 句话（< 100 字） | 统计每条知识的字数 |
| 信息密度 | 原始记忆 → 蒸馏知识，压缩比 >= 5:1 | 原始记忆字数 / 蒸馏知识字数 |
| 触发精度 | 知识的触发条件能准确匹配使用场景 | 检查是否存在"任何时候都适用"的模糊触发 |
| 可操作性 | AI 读到知识后知道具体怎么做 | 检查是否存在"注意"、"小心"等模糊指令 |

**蒸馏前 vs 蒸馏后对比：**

```
【蒸馏前 - 原始记忆】（150 字）
今天在开发订单取消功能的时候，发现了一个问题。当用户点击取消订单按钮后，
前端直接调用了后端的取消接口，但是后端没有检查订单的当前状态。
导致已经发货的订单也被取消了，这是一个很严重的 Bug。
后来我们在后端加了状态检查，只有"待支付"和"已支付"状态的订单才能取消。

【蒸馏后 - 蒸馏知识】（35 字）
订单取消接口必须校验状态：仅"待支付"和"已支付"可取消，已发货不可取消。
（来自 ERR-016）

压缩比：150 / 35 ≈ 4.3:1
```

### 2.3 可验证

每条蒸馏知识都应可验证。验证方式包括：

| 验证类型 | 说明 | 示例 |
|---------|------|------|
| 代码验证 | 通过代码检查或测试验证 | "所有 async 函数都有 await" → 运行 lint 规则验证 |
| 文档验证 | 通过查阅原始文档验证 | "订单取消规则" → 对照 PRD 第 3.2 节验证 |
| 行为验证 | 通过实际运行验证 | "缓存命中后响应 < 100ms" → 运行性能测试验证 |
| 来源验证 | 通过来源追溯验证 | "来自 ERR-005" → 查阅 ERR-005 的完整记录 |

### 2.4 有衰减意识

知识有保质期。蒸馏时必须评估知识的衰减风险：

| 衰减风险 | 说明 | 处理方式 |
|---------|------|---------|
| 高衰减 | 技术细节、API 接口、依赖版本 | 每次阶段蒸馏时重新验证 |
| 中衰减 | 编码模式、架构约定、业务规则 | 里程碑蒸馏时验证 |
| 低衰减 | 核心业务概念、项目目标、设计原则 | 仅在重大变更时验证 |

**衰减标注：**

在 project-brain.md 中，可以为知识标注衰减风险等级：

```
[高衰减] Redis 集群配置：master 节点 3 个，slave 节点 6 个（来自 ADR-004）
  → 下次阶段蒸馏时需验证配置是否变更

[低衰减] 项目核心目标：帮助中小商家统一管理多平台订单（来自 PRD-1.1）
  → 仅在项目方向变更时需重新确认
```

### 2.5 持续成长

蒸馏不是一次性任务，而是持续循环。每次蒸馏都应让 project-brain.md 比上次更好：

| 成长维度 | 衡量指标 | 目标 |
|---------|---------|------|
| 知识覆盖度 | 六大认知章节的知识条数 | 每个阶段蒸馏后增长 |
| 知识精度 | 触发条件的准确度 | 越来越精准 |
| 知识密度 | 单位字数的信息量 | 持续提升（通过压缩蒸馏） |
| 错误免疫率 | 已补强错误的零重复率 | 100% |
| 过时知识清理率 | 已废弃知识的及时移除率 | >= 95% |

## 3. 蒸馏质量检查

每次蒸馏完成后，必须通过以下质量检查：

### 3.1 来源追溯检查

| 检查项 | 通过标准 | 不通过处理 |
|--------|---------|-----------|
| 每条知识都有来源标注 | 100% 的知识有来源 | 补充来源或标注"来源待确认" |
| 来源标注格式正确 | 格式为"来自 [类型] [编号]" | 修正格式 |
| 来源可追溯 | 能从来源找到原始记录 | 如无法追溯，标注"来源待确认" |

### 3.2 冗余检查

| 检查项 | 通过标准 | 不通过处理 |
|--------|---------|-----------|
| 无重复知识 | 不存在两条知识表达相同内容 | 合并为一条 |
| 无矛盾知识 | 不存在两条知识互相矛盾 | 验证哪条正确，废弃错误条 |
| 无过度细节 | 每条知识不超过 100 字 | 进一步提炼精简 |
| 无模糊表述 | 不存在"注意"、"小心"等模糊指令 | 改为具体的可执行规则 |

### 3.3 时效性检查

| 检查项 | 通过标准 | 不通过处理 |
|--------|---------|-----------|
| 知识未过时 | 所有知识与当前项目状态一致 | 标注"已废弃"或更新 |
| 蒸馏时间新鲜 | "最后蒸馏时间"不超过 7 天（活跃开发期） | 执行实时蒸馏 |
| 版本号正确 | 版本号与实际变更匹配 | 更新版本号 |
| 衰减知识已标注 | 高衰减知识有衰减风险标注 | 补充衰减标注 |

### 3.4 结构检查

| 检查项 | 通过标准 | 不通过处理 |
|--------|---------|-----------|
| 六大章节完整 | 1-6 章节都存在 | 补充缺失章节 |
| 元数据完整 | 版本号、蒸馏时间、蒸馏类型已填写 | 补充元数据 |
| 大小在限制内 | 总大小不超过 8000 字 | 执行压缩蒸馏 |
| 蒸馏日志已更新 | 本次蒸馏有对应的日志记录 | 补充蒸馏日志 |

## 4. project-brain.md 的维护规则

### 4.1 大小控制

```
总大小上限：8000 字（约 4000 tokens）
单章节上限：1500 字
单条知识上限：100 字
```

**大小监控：**

每次蒸馏后检查 project-brain.md 的总字数：

| 大小范围 | 处理方式 |
|---------|---------|
| < 6000 字 | 正常，继续增量蒸馏 |
| 6000-8000 字 | 警告，下次蒸馏时优先考虑压缩 |
| > 8000 字 | 必须立即执行压缩蒸馏 |

**压缩策略（按优先级）：**

1. **移除低价值知识**：使用频率低、适用范围窄的知识移至 archive
2. **合并同类知识**：多条相似知识合并为一条更精炼的表述
3. **重新精炼表述**：对过长的知识重新提炼，压缩到 100 字以内
4. **归档旧进度**：将"已完成功能"中超过 3 个月的项目归档

### 4.2 版本管理

**版本号规则：**

| 变更类型 | 版本升级 | 示例 |
|---------|---------|------|
| 实时蒸馏（增量更新 1-3 条知识） | 次版本号 +0.1 | v1.0 → v1.1 |
| 阶段蒸馏（系统化更新一个章节） | 次版本号 +0.1 | v1.1 → v1.2 |
| 里程碑蒸馏（全面重新蒸馏） | 主版本号 +1.0 | v1.5 → v2.0 |

**版本历史保留：**

- 蒸馏日志中保留每次版本变更的记录
- 里程碑蒸馏时生成完整蒸馏报告
- 不保留历史版本的完整文件（通过蒸馏日志追溯变更）

### 4.3 回滚机制

当 project-brain.md 出现严重错误（如大面积知识被误删、错误知识被注入）时，需要回滚。

**回滚步骤：**

```
1. 识别问题：发现 project-brain.md 存在严重错误
2. 定位版本：通过蒸馏日志找到最后一个正确版本
3. 手动回滚：
   a. 如果有 Git 管理：git checkout [正确版本的 commit]
   b. 如果无 Git 管理：从蒸馏日志中重建正确版本
4. 验证回滚：检查回滚后的内容是否正确
5. 记录回滚：在蒸馏日志中记录回滚事件和原因
6. 重新蒸馏：基于回滚版本重新执行蒸馏
```

**回滚预防：**

- 每次蒸馏前，将当前版本备份到 project-brain-backup.md
- 里程碑蒸馏前，额外创建一个带时间戳的快照
- 使用 Git 管理时，每次蒸馏提交一个 commit，便于回滚

### 4.4 存储位置

| 文件 | 位置 | 用途 |
|------|------|------|
| project-brain.md | 项目根目录 | 活跃版本，AI 每次会话加载 |
| project-brain-archive.md | 项目根目录 | 归档知识，不加载但可检索 |
| project-brain-backup.md | 项目根目录 | 最近一次蒸馏前的备份 |
| project-brain-[timestamp].md | 项目根目录 | 里程碑蒸馏前的快照（仅里程碑时创建） |

### 4.5 加载策略

**会话启动时的加载优先级：**

```
1. 加载 project-brain.md（必须）
2. 检查"最后蒸馏时间"
   ├── < 3 天 → 知识新鲜，直接使用
   ├── 3-7 天 → 知识可能需要更新，标记提醒
   └── > 7 天 → 知识可能过时，建议执行蒸馏
3. 如果 project-brain.md 不存在
   → 从 Phase 0 开始创建初始版本
4. 如果 project-brain.md 存在但内容过少（< 500 字）
   → 标记为"认知不完整"，建议执行阶段蒸馏
```

## 5. 蒸馏的自动化触发

虽然蒸馏是 AI 的自主行为，但以下场景应自动触发蒸馏：

| 触发场景 | 蒸馏类型 | 自动化程度 |
|---------|---------|-----------|
| 任务完成（有新发现） | 实时蒸馏 | 全自动 |
| Phase 出口条件满足 | 阶段蒸馏 | 全自动 |
| Bug 修复完成 | 实时蒸馏 | 全自动 |
| 架构决策确定 | 实时蒸馏 | 全自动 |
| 代码审查发现问题 | 实时蒸馏 | 全自动 |
| 使用者显式要求蒸馏 | 任意类型 | 手动触发 |
| project-brain.md 超过大小限制 | 压缩蒸馏 | 全自动 |
| 重大技术变更 | 里程碑蒸馏 | 提醒触发（AI 提醒，使用者确认） |

## 6. 蒸馏与 Phase 4 的关系

| 维度 | Phase 4 知识沉淀 | Module 7 知识蒸馏 |
|------|-----------------|-----------------|
| 触发时机 | Phase 3 完成后，一次性执行 | 贯穿全生命周期，持续执行 |
| 产出物 | project-context.md（静态快照） | project-brain.md（持续成长的活文档） |
