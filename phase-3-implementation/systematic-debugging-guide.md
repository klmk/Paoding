# 系统化调试指南

> **定位**：本文件定义 Phase 3 中遇到 Bug 或测试失败时的系统化调试流程，参考 Superpowers 的 Systematic Debugging 方法论。
> **适用范围**：Phase 3 中所有无法快速定位根因的问题。

---

## 1. 核心原则

### 铁律：没有根因调查就不能提出修复方案

```
遇到 Bug 或测试失败时，严格禁止以下行为：
- "改一行试试" — 没有根因分析的试错
- "换个写法看看" — 没有假设验证的盲目尝试
- "先注释掉这个测试" — 逃避问题而非解决问题
- "应该修好了" — 没有验证的乐观声明
```

**正确做法：** 先调查根因（Root Cause），再基于根因提出修复方案，最后验证修复有效。

### 为什么根因分析如此重要？

| 场景 | 没有根因分析的修复 | 有根因分析的修复 |
|------|-------------------|-----------------|
| Bug 复现 | 修复了表象，根因仍在，Bug 会以不同形式复现 | 根除根因，Bug 不会复现 |
| 修复范围 | 修复了一处，其他类似问题仍然存在 | 一次性修复所有同类问题 |
| 副作用 | 修复引入新问题（因为不理解系统） | 修复精准，不影响其他功能 |
| 时间成本 | 短期快，长期反复修复 | 短期慢，长期一劳永逸 |

## 2. 四阶段根因分析流程

### 流程总览

```
问题发现
    │
    ▼
┌─────────────────────────────────┐
│  阶段 1：根因调查                │
│  (Root Cause Investigation)      │
│                                  │
│  收集信息 → 复现问题 → 检查日志   │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│  阶段 2：模式分析                │
│  (Pattern Analysis)              │
│                                  │
│  识别模式 → 缩小范围 → 假设根因   │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│  阶段 3：假设测试                │
│  (Hypothesis Testing)            │
│                                  │
│  设计实验 → 验证假设 → 排除法     │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│  阶段 4：实现修复                │
│  (Fix Implementation)            │
│                                  │
│  最小修复 → 验证修复 → 回归测试   │
└─────────────────────────────────┘
```

### 阶段 1：根因调查（Root Cause Investigation）

**目标：** 收集足够的信息来理解问题的全貌。

#### 1.1 收集信息

在动手调查之前，先收集以下信息：

| 信息类型 | 收集方式 | 示例 |
|---------|---------|------|
| **错误信息** | 读取完整的错误堆栈、错误消息 | `TypeError: Cannot read property 'id' of undefined at UserService.js:42` |
| **复现步骤** | 记录触发问题的精确步骤 | "用户登录后点击个人资料页，然后刷新页面" |
| **期望行为** | 对照 AC 确认期望的正确行为 | "应显示用户信息，不报错" |
| **实际行为** | 记录观察到的实际行为 | "页面白屏，控制台报 TypeError" |
| **环境信息** | 记录运行环境 | Node.js v18.17.0, 数据库 PostgreSQL 15 |
| **变更历史** | 检查最近的代码变更 | "最近提交了 commit abc123，修改了 UserService" |
| **相关日志** | 检查应用日志、数据库日志、系统日志 | "日志显示查询返回了 null" |

#### 1.2 复现问题

**稳定复现是调试的基础。** 如果无法稳定复现，就无法验证修复是否有效。

**复现步骤：**
1. 按照记录的复现步骤操作
2. 确认问题是否可稳定复现
3. 如果不能稳定复现，尝试以下方法：
   - 改变操作顺序
   - 改变输入数据
   - 改变环境条件（网络延迟、并发量）
   - 检查是否有时间相关的因素（缓存过期、定时任务）
4. 记录复现的条件和概率

**复现检查点：**
- [ ] 问题可以稳定复现（100% 概率）
- [ ] 或：问题可以概率性复现，且已记录影响复现的因素
- [ ] 已记录最小复现步骤

#### 1.3 检查日志

**日志是调试的眼睛。** 系统性地检查所有相关日志。

**日志检查清单：**
- [ ] 应用日志：是否有异常、警告、错误？
- [ ] 数据库日志：查询是否正确执行？返回了什么数据？
- [ ] 网络日志：请求和响应的内容是什么？状态码是什么？
- [ ] 系统日志：是否有资源不足、权限问题？
- [ ] 测试日志：测试失败的具体断言是什么？实际值和期望值的差异？

**日志分析技巧：**
- 从错误发生的时间点向前回溯，寻找第一个异常信号
- 关注"第一次出现"的异常，而非后续的连锁反应
- 检查日志中是否有被忽略的警告信息

#### 阶段 1 的输出

完成阶段 1 后，应产出以下信息：

```markdown
## 根因调查记录

**问题描述：** [一句话描述问题]
**错误信息：** [完整错误堆栈]
**复现步骤：** [最小复现步骤]
**环境信息：** [运行环境]
**相关日志：** [关键日志摘录]
**已知事实：** [从日志和观察中确认的事实列表]
```

### 阶段 2：模式分析（Pattern Analysis）

**目标：** 从收集的信息中识别规律，缩小问题范围，提出根因假设。

#### 2.1 识别模式

从已知事实中寻找模式和规律：

| 模式类型 | 检查方法 | 示例 |
|---------|---------|------|
| **时间模式** | 问题是否在特定时间出现？ | 每天凌晨 3 点（定时任务相关） |
| **数据模式** | 问题是否与特定数据相关？ | 只有大文件上传时失败 |
| **条件模式** | 问题是否在特定条件下出现？ | 只在并发请求时出现 |
| **顺序模式** | 问题是否与操作顺序相关？ | 先 A 后 B 就出错，先 B 后 A 正常 |
| **频率模式** | 问题的出现频率如何？ | 偶发（< 1%）vs 必现（100%） |

#### 2.2 缩小范围

使用**二分法**缩小问题范围：

```
1. 定位问题所在的层级：
   前端 → API 层 → 业务逻辑层 → 数据访问层 → 数据库

2. 在每个层级内，使用二分法缩小范围：
   - 注释掉一半代码，看问题是否消失
   - 如果消失，问题在被注释的一半中
   - 如果仍在，问题在未被注释的一半中
   - 重复直到定位到具体的函数或代码行

3. 排除无关因素：
   - 逐个排除环境因素（网络、缓存、配置）
   - 逐个排除依赖因素（第三方库、外部服务）
```

#### 2.3 假设根因

基于模式分析和范围缩小，提出一个或多个根因假设。

**假设格式：**

```
假设 [编号]：[根因描述]
依据：[支持该假设的证据]
预测：[如果该假设正确，应该观察到什么现象]
验证方法：[如何验证该假设]
```

**示例：**

```
假设 1：UserService.findById() 在用户不存在时返回 undefined，
       而调用方未处理 undefined 的情况。
依据：日志显示数据库查询返回了 null；错误发生在 UserService.js:42，
     该行调用了 result.id。
预测：如果传入一个不存在的用户 ID，应该能复现同样的错误。
验证方法：编写一个测试，传入不存在的用户 ID，观察是否抛出 TypeError。
```

**假设要求：**
- 每个假设必须有证据支持（不能凭空猜测）
- 每个假设必须有可验证的预测
- 优先考虑最简单的假设（奥卡姆剃刀原则）
- 同时考虑多个假设，不要过早排除

#### 阶段 2 的输出

```markdown
## 模式分析记录

**识别到的模式：** [描述发现的模式]
**问题范围：** [已排除的区域] → [疑似区域]
**根因假设：**
1. 假设 1：...
2. 假设 2：...
3. 假设 3：...
```

### 阶段 3：假设测试（Hypothesis Testing）

**目标：** 通过实验验证或排除根因假设。

#### 3.1 设计实验

为每个假设设计一个**最小实验**来验证或排除。

**实验设计原则：**
- 每次只测试一个假设
- 实验应能明确区分"假设正确"和"假设错误"
- 实验应可重复
- 实验应尽可能简单（不要设计复杂的实验）

**实验类型：**

| 实验类型 | 方法 | 适用场景 |
|---------|------|---------|
| **单元测试** | 编写测试直接验证假设 | 逻辑错误、数据处理错误 |
| **日志注入** | 在关键位置添加日志，观察数据流 | 数据传递错误、状态转换错误 |
| **条件控制** | 通过修改条件触发或避免问题 | 条件分支错误、时序问题 |
| **对比实验** | 对比正常情况和异常情况的差异 | 环境差异、配置差异 |
| **排除法** | 逐个排除变量，观察问题是否消失 | 多因素交织的问题 |

#### 3.2 验证假设

执行实验，记录结果：

```markdown
### 实验 1：验证假设 1

**实验方法：** [描述实验方法]
**实验步骤：**
1. [步骤 1]
2. [步骤 2]
3. [步骤 3]

**预期结果（假设正确）：** [如果假设正确，应该看到什么]
**实际结果：** [实际看到了什么]

**结论：** [确认 / 排除 / 不确定]
```

#### 3.3 排除法

当有多个假设时，使用排除法逐个验证：

```
假设列表：[H1, H2, H3]

第 1 轮：
  测试 H1 → 排除（实验结果与预测不符）
  测试 H2 → 确认（实验结果与预测一致）
  → 根因确定为 H2

第 2 轮（如果第 1 轮未确认）：
  测试 H3 → 排除
  → 所有假设都被排除，需要回到阶段 2 重新分析
```

**所有假设都被排除时的处理：**
1. 回到阶段 1，重新收集信息（可能遗漏了关键信息）
2. 回到阶段 2，重新分析模式（可能模式识别有误）
3. 考虑更复杂的根因（多个因素组合）
4. 考虑更底层的根因（运行时 Bug、环境问题、第三方库 Bug）

#### 阶段 3 的输出

```markdown
## 假设测试记录

### 实验 1
- 假设：...
- 方法：...
- 结果：[确认/排除/不确定]

### 实验 2
- 假设：...
- 方法：...
- 结果：[确认/排除/不确定]

**确定的根因：** [最终确认的根因描述]
```

### 阶段 4：实现修复（Fix Implementation）

**目标：** 基于确认的根因，实现最小且精确的修复。

#### 4.1 最小修复原则

**修复应尽可能小且精确，只修复根因，不做额外改动。**

| 修复方式 | 描述 | 评价 |
|---------|------|------|
| **最小修复** | 只修改根因相关的代码 | 推荐 |
| **防御性修复** | 在错误发生处添加检查，避免错误传播 | 可接受，但应同时修复根因 |
| **重写修复** | 重写整个模块 | 仅当根因是设计缺陷时使用 |
| **绕过修复** | 添加条件跳过问题代码 | 禁止 |

**最小修复的要求：**
- 修改的代码行数尽可能少
- 不引入新的依赖或复杂度
- 不改变不相关的功能
- 修复后代码的可读性不低于修复前

#### 4.2 TDD 修复流程

修复也必须遵循 TDD 流程：

```
1. RED：编写一个能复现 Bug 的测试（回归测试）
   - 这个测试在修复前应该失败
   - 这个测试描述了"正确的行为应该是什么"

2. GREEN：实现最小修复，使回归测试通过
   - 只修改根因相关的代码
   - 运行测试，确认回归测试通过

3. REFACTOR：如有必要，在测试保护下重构
   - 改善修复代码的质量
   - 确保所有测试仍然通过
```

#### 4.3 验证修复

修复后必须进行以下验证：

| 验证类型 | 说明 | 要求 |
|---------|------|------|
| **回归测试** | 运行为该 Bug 编写的回归测试 | 必须通过 |
| **全量测试** | 运行所有测试，确保无回归 | 必须全部通过 |
| **复现验证** | 按原始复现步骤操作，确认问题不再出现 | 必须确认 |
| **边界验证** | 测试与该 Bug 相关的边界条件 | 建议执行 |

#### 4.4 回归测试

修复后运行全量测试套件，确保修复没有引入新的问题。

**回归测试检查点：**
- [ ] 为该 Bug 编写的回归测试通过
- [ ] 所有已有测试仍然通过（无新的失败）
- [ ] 没有测试被跳过或忽略
- [ ] 测试覆盖率没有下降

#### 阶段 4 的输出

```markdown
## 修复记录

**根因：** [确认的根因]
**修复方案：** [修复的描述]
**修改文件：** [修改的文件和行号]
**回归测试：** [新增的测试用例]
**验证结果：**
- 回归测试：[通过/失败]
- 全量测试：[通过/失败]
- 复现验证：[通过/失败]
```

## 3. 三次修复失败升级规则

### 3.1 触发条件

当同一问题的修复尝试**连续失败 3 次**时，必须触发升级规则。

**"失败"的定义：**
- 修复后问题仍然存在
- 修复后引入了新的问题（回归）
- 修复后测试仍然失败
- 修复后问题以不同形式复现

### 3.2 升级处理流程

```
第 3 次修复失败
    │
    ├── 立即暂停当前修复工作
    │
    ├── 记录完整的调试日志
    │   ├── 问题描述和复现步骤
    │   ├── 已尝试的 3 个修复方案及失败原因
    │   ├── 观察到的所有现象
    │   └── 当前代码状态
    │
    ├── 分析失败模式
    │   ├── 3 次修复是否有共同点？
    │   ├── 是否在修复错误的东西？
    │   ├── 根因假设是否根本就是错的？
    │   └── 是否存在更深层的系统性问题？
    │
    └── 升级处理（按顺序尝试）
        ├── 1. 请求用户参与分析
        │      → 用户提供业务上下文或额外信息
        │
        ├── 2. 换一个 AI 实例/子代理独立分析
        │      → 新的视角可能发现被忽略的线索
        │
        ├── 3. 重新审视问题定义
        │      → 问题描述本身是否有误？
        │      → "Bug" 是否实际上是"特性"？
        │      → AC 是否有歧义？
        │
        └── 4. 回退到设计阶段
               → 是否是设计缺陷导致的？
               → 需要修改架构或接口设计？
```

### 3.3 升级规则的关键约束

- **不得在第 4 次尝试中继续相同的思路** — 如果 3 次修复都失败了，第 4 次用同样的方法也不会成功
- **必须引入新的视角** — 换人、换方法、换思路
- **必须重新审视问题定义** — 可能问题本身的理解就是错的
- **暂停不是放弃** — 暂停是为了更好地前进，积累的信息不会浪费

## 5. 调试与 TDD 的关系

### 5.1 调试中的 TDD

调试修复也必须遵循 TDD 流程：

```
传统调试（禁止）：
  发现 Bug → 直接改代码 → 运行看看 → 还有问题 → 再改 → ...

TDD 调试（必须）：
  发现 Bug → 编写复现 Bug 的测试（RED）
           → 确认测试失败（确认 Bug 存在）
           → 修复代码（GREEN）
           → 确认测试通过（确认 Bug 已修复）
           → 运行全量测试（确认无回归）
           → 重构（REFACTOR）
```

### 5.2 回归测试的价值

为每个 Bug 编写回归测试的价值：

- **防止复现**：即使修复不完美，回归测试也能在 Bug 复现时立即告警
- **文档化**：回归测试是 Bug 的"活文档"，记录了 Bug 的触发条件和期望行为
- **验证修复**：回归测试提供了客观的"修复完成"标准
- **信心保障**：有了回归测试，后续修改代码时不必担心重新引入该 Bug

### 5.3 调试驱动的重构

如果调试过程中发现代码结构导致 Bug 难以修复（如职责不清、耦合过紧），应在修复 Bug 后进行重构：

1. 先修复 Bug（最小修复）
2. 编写回归测试保护修复
3. 在测试保护下重构代码结构
4. 确认所有测试仍然通过

## 6. 常见调试陷阱

### 陷阱 1："确认偏差"（Confirmation Bias）

**描述：** 找到一个看似合理的解释后，只寻找支持该解释的证据，忽略反面证据。

**防范：** 为每个假设设计"证伪实验"——如果假设是错的，应该看到什么现象？如果证伪实验的结果与假设矛盾，必须放弃该假设。

### 陷阱 2："锚定效应"（Anchoring Effect）

**描述：** 第一个想到的根因假设成为"锚点"，后续分析都围绕该假设展开，忽略了其他可能性。

**防范：** 同时考虑至少 3 个不同的根因假设，用实验逐一验证，而不是只验证第一个想到的假设。

### 陷阱 3："修复即完成"（Fix and Forget）

**描述：** 修复了 Bug 后立即继续开发，不编写回归测试，不分析根因，不记录经验。

**防范：** 每个修复必须包含：回归测试、根因记录、经验教训。

### 陷阱 4："环境归因"（Environment Blaming）

**描述：** 遇到难以解释的问题时，归咎于"环境问题"、"缓存问题"、"网络问题"，而不深入调查。

**防范：** 环境问题也是问题，也需要根因分析。如果怀疑是环境问题，设计实验验证（如清除缓存后是否仍然出现）。

### 陷阱 5："无限调试"（Infinite Debugging）

**描述：** 在一个问题上花费过多时间，拒绝寻求帮助或升级处理。

**防范：** 遵守三次修复失败升级规则。超过 30 分钟的调试应评估是否需要升级。
