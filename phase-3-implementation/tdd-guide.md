# TDD 实践指南

> **定位**：本文件是 Phase 3 开发实施阶段中 TDD（测试驱动开发）的详细实践指南。
> **适用范围**：所有 Phase 3 中的编码任务，无一例外。

---

## 1. RED-GREEN-REFACTOR 循环详解

TDD 的核心是一个不断重复的三步循环。每个循环的粒度应尽可能小——一次只解决一个行为点。

### 1.1 RED 阶段：编写失败测试

**目标：** 在编写任何生产代码之前，先用测试定义"正确的行为应该是什么"。

**执行步骤：**

```
1. 阅读 AC（验收标准），明确需要实现的行为
2. 编写一个测试用例来描述该行为
3. 运行测试，确认测试失败（这是关键验证步骤）
4. 如果测试意外通过：
   a. 检查测试是否真的在验证期望行为（可能断言写错了）
   b. 检查是否已有代码实现了该行为（可能测试不够具体）
   c. 修正测试，使其真正失败
```

**RED 阶段的关键原则：**
- 测试失败的原因必须是"功能未实现"，而非"测试写错了"或"环境有问题"
- 每次只写一个失败的测试（或一组紧密相关的测试）
- 测试失败时的错误信息应该清晰指向缺失的功能

**RED 阶段的验证检查点：**
- [ ] 测试确实失败了（不是编译错误、不是环境错误）
- [ ] 失败原因是功能未实现（不是断言写错）
- [ ] 测试关联了正确的 AC ID

### 1.2 GREEN 阶段：编写最小实现

**目标：** 用尽可能少的代码使测试通过。

**执行步骤：**

```
1. 编写刚好足够使失败测试通过的生产代码
2. 不追求优雅、不追求完美，只追求"测试通过"
3. 允许"丑陋"的代码（重复、硬编码等），因为下一步会重构
4. 运行测试，确认测试通过
5. 如果测试仍然失败：
   a. 检查生产代码是否正确实现了期望行为
   b. 检查测试的断言是否合理
   c. 不要修改测试来适应实现（除非测试本身有误）
```

**GREEN 阶段的关键原则：**
- "使测试通过"是唯一目标，不要在此阶段做额外优化
- 允许的"丑陋"：重复代码、硬编码值、过长函数、不完美的命名
- 不允许的"丑陋"：跳过测试、降低断言精度、忽略边界条件

**GREEN 阶段的验证检查点：**
- [ ] 失败的测试现在通过了
- [ ] 之前通过的测试仍然通过（无回归）
- [ ] 没有引入新的测试失败

### 1.3 REFACTOR 阶段：重构优化

**目标：** 在测试保护下改善代码设计，不改变外部行为。

**执行步骤：**

```
1. 识别代码中的"坏味道"（重复、过长、命名不清等）
2. 一次只做一项重构
3. 每次重构后运行测试，确认测试仍然全部通过
4. 如果测试失败，立即撤销重构（git checkout 或 undo）
5. 重复直到代码质量达到可接受的标准
```

**REFACTOR 阶段的关键原则：**
- 重构不改变外部行为，只改变内部实现
- 每次只做一项重构，便于定位问题
- 测试是重构的安全网——测试不通过就不提交重构
- 常见的重构操作：提取函数、提取变量、重命名、移动函数、消除重复

**REFACTOR 阶段的验证检查点：**
- [ ] 所有测试仍然通过
- [ ] 代码设计有所改善（更清晰、更简洁、更易维护）
- [ ] 没有引入新的功能或改变已有行为

### 1.4 循环节奏

```
理想节奏：每个循环 5-15 分钟

RED   → 2-5 分钟（编写一个失败的测试）
GREEN → 2-5 分钟（编写最小实现）
REFACTOR → 1-5 分钟（改善设计）

如果一个循环超过 30 分钟，说明任务粒度太大，需要拆分。
```

## 2. 测试编写规范

### 2.1 测试命名规范

测试名称应清晰描述被测行为，格式为：

```
should_[期望行为]_when_[条件]
```

**示例：**

```javascript
// 好的命名
should_return_empty_array_when_no_items_exist
should_throw_validation_error_when_email_format_invalid
should_calculate_total_price_including_tax_when_cart_has_multiple_items

// 差的命名
test1()
testAdd()
testFunction()
```

**命名要求：**
- 使用 `should` 开头，描述期望行为
- 包含 `when` 子句描述前置条件
- 不使用技术术语（如 "should return null" 而非 "should return undefined because of JS coercion"）
- 测试名称本身应足以理解测试意图，无需阅读测试体

### 2.2 测试结构规范

每个测试用例应遵循 **AAA 模式**（Arrange-Act-Assert）：

```javascript
// Arrange：准备测试数据和环境
const user = { name: '张三', role: 'admin' };
const service = new UserService(mockRepository);

// Act：执行被测操作
const result = service.getPermissions(user);

// Assert：验证结果
expect(result).toContain('read');
expect(result).toContain('write');
expect(result).toContain('delete');
```

**结构要求：**
- Arrange、Act、Assert 三个部分应有清晰的分隔（注释或空行）
- 每个测试只验证一个行为点
- Assert 部分应包含正向断言和必要的反向断言

### 2.3 断言规范

**断言原则：**
- 每个测试至少有一个断言（没有断言的测试没有价值）
- 断言应验证行为，而非实现细节
- 优先使用具体的断言方法，避免过于宽泛的断言

**示例：**

```javascript
// 好的断言 —— 验证行为
expect(result.status).toBe(200);
expect(result.data.items).toHaveLength(3);

// 差的断言 —— 验证实现细节
expect(result.internalCache.has('key')).toBe(true);

// 差的断言 —— 过于宽泛
expect(result).toBeTruthy();
expect(result).not.toBeNull();
```

### 2.4 测试覆盖要求

每个功能模块的测试应覆盖以下三个维度：

| 维度 | 描述 | 示例 |
|------|------|------|
| **正常路径** | 预期输入产生预期输出 | 有效用户登录成功 |
| **边界条件** | 输入处于边界值 | 空字符串、最大长度、零值、负数 |
| **异常路径** | 非预期输入或系统异常 | 无效格式、网络超时、权限不足 |

**覆盖标准：**
- 每个公开函数/方法至少有 1 个正常路径测试
- 每个边界条件至少有 1 个测试
- 每个异常路径至少有 1 个测试
- 复杂逻辑（分支、循环）的每个分支至少有 1 个测试

### 2.5 测试独立性规范

- 测试之间不得有执行顺序依赖
- 每个测试应自行准备数据（不依赖其他测试的副作用）
- 测试执行顺序改变不应影响测试结果
- 使用 `beforeEach`/`afterEach` 清理测试环境

## 3. 测试反模式

以下是 TDD 实践中常见的反模式，必须主动识别和避免。

### AP-T01 "写完代码再补测试"

| 项目 | 内容 |
|------|------|
| **描述** | 先写生产代码，再补写测试来"覆盖"已有实现 |
| **危害** | 测试变成了代码的附庸，只验证"代码做了什么"而非"代码应该做什么"。无法发现缺失的功能。 |
| **正确做法** | 严格遵循 RED-GREEN-REFACTOR 循环。先写测试定义行为，再写代码实现行为。 |
| **违反铁律** | P3-1 没有失败的测试就没有生产代码 |

### AP-T02 "测试只验证快乐路径"

| 项目 | 内容 |
|------|------|
| **描述** | 只编写正常输入的测试，忽略边界条件和异常场景 |
| **危害** | 边界条件和异常场景是 Bug 的高发区域。只测快乐路径等于只锁了正门，后门大开。 |
| **正确做法** | 每个功能至少覆盖正常路径、边界条件、异常路径三个维度。 |
| **违反铁律** | P3-1 测试必须覆盖正常路径、边界条件和异常路径 |

### AP-T03 "测试与实现强耦合"

| 项目 | 内容 |
|------|------|
| **描述** | 测试验证的是实现细节（如内部方法调用、私有变量），而非外部行为 |
| **危害** | 重构时测试会大量失败，即使外部行为没有改变。测试变成了重构的阻碍而非助力。 |
| **正确做法** | 测试应验证公开接口的行为契约（输入→输出），不验证内部实现。 |
| **违反铁律** | P3-1 REFACTOR 阶段要求测试不阻碍重构 |

### AP-T04 "测试中的魔法数字"

| 项目 | 内容 |
|------|------|
| **描述** | 测试中直接使用硬编码的数值或字符串，没有解释其含义 |
| **危害** | 测试可读性差，维护者不理解为什么期望值是这个数字。 |
| **正确做法** | 使用有意义的常量或变量名，解释数值的业务含义。 |

```javascript
// 差的写法
expect(result).toBe(86400);

// 好的写法
const SECONDS_IN_ONE_DAY = 86400;
expect(result).toBe(SECONDS_IN_ONE_DAY);
```

### AP-T05 "巨型测试函数"

| 项目 | 内容 |
|------|------|
| **描述** | 一个测试函数验证多个不相关的行为，或包含大量 Arrange 代码 |
| **危害** | 测试意图不清晰，失败时难以定位问题，维护成本高。 |
| **正确做法** | 每个测试只验证一个行为点。复杂的 Arrange 逻辑提取到测试工厂函数或 fixture 中。 |

### AP-T06 "跳过或忽略失败的测试"

| 项目 | 内容 |
|------|------|
| **描述** | 测试失败时使用 `.skip()`、`.only()` 或注释掉测试 |
| **危害** | 跳过测试等于移除了安全网。被跳过的测试永远不会被修复。 |
| **正确做法** | 测试失败时必须立即修复——要么修复代码，要么修复测试。不得跳过。 |
| **违反铁律** | P3-1 测试失败时，必须先修复测试或生产代码，不得跳过或注释测试 |

### AP-T07 "测试依赖外部状态"

| 项目 | 内容 |
|------|------|
| **描述** | 测试依赖数据库、文件系统、网络等外部资源，且不进行隔离 |
| **危害** | 测试结果不稳定（flaky tests），在不同环境下可能产生不同结果。 |
| **正确做法** | 使用 mock/stub 隔离外部依赖。集成测试可以访问真实外部资源，但必须明确标记且可独立运行。 |

### AP-T08 "测试代码没有质量要求"

| 项目 | 内容 |
|------|------|
| **描述** | 认为"测试代码不需要好的设计"，测试代码一团糟 |
| **危害** | 难以维护的测试会被开发者厌恶，最终被忽略或删除。测试代码也是代码资产。 |
| **正确做法** | 测试代码应遵循与生产代码相同的编码规范。提取重复的测试逻辑为辅助函数。 |

## 4. 常见借口及反驳

以下是 AI 在 TDD 实践中常见的"合理化借口"及对应反驳。当 AI 内部出现类似念头时，必须立即对照本清单进行自我纠正。

### 借口 1："这个功能太简单，不值得写测试"

**反驳：** 简单代码也会出错。越"简单"的代码，越容易因为过度自信而跳过验证。Bug 往往藏在"不可能出错"的地方。TDD 的价值不在于代码复杂度，而在于确保行为正确性。一个"简单"的函数如果有 Bug，排查它的时间可能比写测试的时间长 10 倍。

### 借口 2："测试框架还没配好，先写代码"

**反驳：** 测试环境是开发环境的一部分。没有测试环境就不具备开发条件。配置测试环境是 Phase 2 的职责。如果测试环境未就绪，应回退到 Phase 2 解决，而非跳过 TDD。

### 借口 3："我会在代码写完后补测试"

**反驳：** 事后补写的测试往往只验证"代码做了什么"，而非"代码应该做什么"。这导致测试变成了代码的附庸，而非行为的定义者。更严重的是，补测试时你可能会下意识地让测试"配合"代码，而非真正验证代码。

### 借口 4："这个逻辑很难测试，跳过吧"

**反驳：** 难以测试的代码往往意味着设计有问题——职责不清、耦合过紧、依赖过多。应该重构设计使其可测试，而非跳过测试。可测试性是设计质量的指标。

### 借口 5："测试太多会拖慢开发速度"

**反驳：** 短期看测试增加了代码量，长期看测试是最高效的回归验证手段。没有测试的代码修改成本随时间指数增长。有了测试，你可以自信地重构、添加功能、修复 Bug，而不必担心"改了这个会不会影响那个"。

### 借口 6："接口还没定好，没法写测试"

**反驳：** TDD 的本质就是用测试来定义接口。先写测试就是在定义接口。如果你不知道接口应该是什么样的，说明你对需求的理解还不够，应该回到步骤 2（理解上下文）重新分析。

### 借口 7："测试代码也是代码，写测试等于写两倍代码"

**反驳：** 测试代码和生产代码的价值不同。生产代码实现功能，测试代码保障质量。测试代码是"保险费"——你不会因为要交保险费就不买车。而且，测试代码通常比生产代码简单，编写速度更快。

### 借口 8："这个 Bug 只是一个小笔误，不需要写测试"

**反驳：** "小笔误"可能是系统性问题的表象。如果你犯了这次笔误，你（或其他人）可能再次犯同样的错误。一个针对该 Bug 的回归测试可以防止问题复发。修复 Bug 时必须同时添加回归测试。

### 借口 9："测试通过了，但我知道代码有问题，先改代码再更新测试"

**反驳：** 如果你已经知道代码有问题，那当前的测试就是不充分的。正确的做法是：先写一个能暴露该问题的测试（RED），然后修复代码（GREEN），最后重构（REFACTOR）。不要跳过 RED 阶段。

### 借口 10："项目赶进度，TDD 太慢了"

**反驳：** TDD 不是在"慢"和"快"之间选择，而是在"短期慢+长期快"和"短期快+长期慢"之间选择。没有 TDD 的项目，前期看似快，但 Bug 修复、回归测试、重构成本会迅速累积，最终总时间反而更长。TDD 是对开发速度的投资，不是消耗。

### 借口 11："我已经手动测试过了，效果一样"

**反驳：** 手动测试是一次性的、不可重复的、依赖测试者记忆的。自动化测试是持久的、可重复的、可回归的。你手动测试过一次，不代表下次修改后还能记得手动测试所有场景。自动化测试是"一次性投入，永久回报"。

### 借口 12："测试覆盖率已经达标了，不用再写了"

**反驳：** 覆盖率是数量指标，不是质量指标。100% 的覆盖率如果都是无意义的断言（如 `expect(true).toBe(true)`），等于 0% 的有效覆盖。覆盖率达标不等于测试充分，应关注测试是否真正验证了关键行为。

## 5. 连续失败处理（HALT 条件）

### 5.1 HALT 触发条件

当以下任一条件满足时，必须**立即暂停**（HALT）当前任务：

| # | 条件 | 阈值 | 说明 |
|---|------|------|------|
| 1 | 连续测试失败 | **3 次** | 同一个测试或同一组测试连续失败 3 次 |
| 2 | 连续修复失败 | **3 次** | 同一个 Bug 的修复尝试连续失败 3 次 |
| 3 | 重构导致回归 | **2 次** | 重构后测试失败，且连续 2 次重构都导致回归 |
| 4 | 环境问题 | **1 次** | 测试环境异常（依赖服务不可用、配置错误等） |

### 5.2 HALT 处理流程

```
HALT 触发
    │
    ├── 记录 HALT 信息
    │   ├── 触发条件（哪个条件）
    │   ├── 失败详情（错误信息、日志）
    │   ├── 已尝试的方案
    │   └── 当前代码状态
    │
    ├── 分析 HALT 原因
    │   ├── 是测试问题？（测试写错、断言不当）
    │   ├── 是设计问题？（需求不清、设计缺陷）
    │   ├── 是环境问题？（依赖、配置、版本）
    │   └── 是能力问题？（超出当前能力范围）
    │
    ├── 采取对应措施
    │   ├── 测试问题 → 重新审视测试设计
    │   ├── 设计问题 → 回退到 Phase 2 修复设计
    │   ├── 环境问题 → 修复环境后重试
    │   └── 能力问题 → 请求用户或外部协助
    │
    └── 恢复执行
        ├── 问题已解决 → 从失败点继续
        └── 问题未解决 → 升级处理（请求用户决策）
```

### 5.3 HALT 记录模板

```markdown
## HALT 记录

**时间：** [时间戳]
**任务：** [任务编号和描述]
**关联 AC：** [AC ID]

### 触发条件
- [ ] 连续测试失败 3 次
- [ ] 连续修复失败 3 次
- [ ] 重构导致回归 2 次
- [ ] 环境问题

### 失败详情
**第 1 次失败：**
- 错误信息：...
- 尝试方案：...
- 结果：...

**第 2 次失败：**
- 错误信息：...
- 尝试方案：...
- 结果：...

**第 3 次失败：**
- 错误信息：...
- 尝试方案：...
- 结果：...

### 原因分析

```

### 5.4 HALT 后的恢复验证

HALT 解除后，恢复执行前必须完成以下验证：

1. 确认 HALT 原因已消除（不是暂时绕过）
2. 运行全量测试，确认无回归
3. 如有代码变更，确认变更已提交并有追溯标识

### 6.1 AC ID 关联规范

每个测试用例必须关联对应的 AC（验收标准）ID，确保测试可追溯到需求。

**关联方式：**

// 方式 1：在测试名称中包含 AC ID

  AC: 'AC-003',

每个任务完成后，更新追溯矩阵：

| AC ID | AC 描述 | 测试用例 | 生产代码文件 | 状态 |

## 7. 测试分层策略

> **核心原则：** 不测试"代码做了什么"，而测试"代码应该做什么"。测试用例必须对照 AIS 中的 Concrete Examples（具体示例）编写，而非对照 AI 自己的理解。

### 7.1 测试分层架构

```
Layer 4：视觉回归测试    →  截图对比，发现"界面变了"
Layer 3：E2E 交互测试    →  模拟用户操作流程（点击→输入→验证）
Layer 2：组件测试        →  单个组件的渲染和行为
Layer 1：API/接口测试    →  接口是否正常（TDD 覆盖）
```

| 层级 | 测试对象 | 必要性 | AI 参与度 | 推荐工具 |
|------|---------|--------|---------|---------|
| Layer 1 API 测试 | 后端接口 | 必须 | ✅ 完全自主 | pytest, Apifox |
| Layer 2 组件测试 | 前端组件 | 应该 | ✅ 大部分自主 | Vitest + Vue Test Utils, React Testing Library |
| Layer 3 E2E 交互测试 | 完整用户流程 | 应该 | ✅ 大部分自主 | Playwright, Cypress |
| Layer 4 视觉回归 | 页面视觉 | 可选 | ⚠️ 需要工具辅助 | Playwright 截图, Percy, Chromatic |

### 7.2 Layer 1：API/接口测试

**覆盖要求：** 所有后端接口必须有对应的 API 测试。

**AI 职责：** 完全自主——编写测试用例、执行测试、验证结果。

**测试内容：**
- 请求参数校验（必填、类型、范围）
- 正常响应格式和内容
- 错误响应格式和错误码
- 认证和授权
- 超时和重试机制

### 7.3 Layer 2：组件测试

**覆盖要求：** 核心业务组件必须有组件测试。

**AI 职责：** 大部分自主——编写组件测试脚本、执行测试、验证 DOM 输出。

**测试内容：**
- 组件是否正确渲染
- 用户交互（点击、输入、选择）是否触发正确行为
- Props 变化时组件是否正确响应
- 事件是否正确发射
- 条件渲染（loading、empty、error 状态）

**示例（Vue 项目）：**

```javascript
// 测试订单列表组件
import { mount } from '@vue/test-utils'
import OrderList from '@/components/OrderList.vue'

describe('[AC-003] 订单列表组件', () => {
  it('should render order list when orders exist', () => {
    const wrapper = mount(OrderList, {
      props: { orders: mockOrders }
    })
    expect(wrapper.findAll('.order-item')).toHaveLength(3)
  })

  it('should emit approve event when approve button clicked', async () => {
    const wrapper = mount(OrderList, {
      props: { orders: mockOrders }
    })
    await wrapper.find('.approve-btn').trigger('click')
    expect(wrapper.emitted('approve')).toBeTruthy()
    expect(wrapper.emitted('approve')[0][0]).toBe(mockOrders[0].id)
  })

  it('should show empty state when no orders', () => {
    const wrapper = mount(OrderList, {
      props: { orders: [] }
    })
    expect(wrapper.find('.empty-state').exists()).toBe(true)
  })
})
```

### 7.4 Layer 3：E2E 交互测试

**覆盖要求：** 核心用户旅程必须有 E2E 测试。

**AI 职责：** 大部分自主——编写操作脚本、执行自动化操作、验证 DOM 元素和网络请求。

**核心认知：** AI 不能"看"界面，但可以"操作"界面 + "检查" DOM 元素 + "验证"网络请求。这三件事 AI 完全可以做。

**E2E 测试用例必须包含：**
1. 操作步骤（按用户真实操作顺序）
2. 每步操作后的 DOM 检查点
3. 每步操作后的网络请求检查点
4. 异常场景的操作路径
5. 截图节点（关键步骤自动截图）

**示例（Vue 项目 + Playwright）：**

```javascript
// 测试 ERP 单据审核→推送 MES 的完整流程
const { test, expect } = require('@playwright/test')

test('[AC-005] ERP 审核推送 MES 完整流程', async ({ page }) => {
  // Step 1: 打开订单列表页面
  await page.goto('/orders')
  await expect(page.locator('.order-list')).toBeVisible()

  // Step 2: 点击审核按钮
  await page.click('.order-item:first-child .approve-btn')
  await expect(page.locator('.confirm-dialog')).toBeVisible()

  // Step 3: 确认审核
  await page.click('.confirm-dialog .confirm-btn')

  // Step 4: 验证 DOM 状态变化
  await expect(page.locator('.order-item:first-child .status')).toHaveText('已审核')

  // Step 5: 验证网络请求
  const pushRequest = await page.waitForRequest('**/api/orders/*/push-to-mes')
  expect(pushRequest.method()).toBe('POST')

  // Step 6: 验证推送结果提示
  await expect(page.locator('.push-result')).toBeVisible()
  await expect(page.locator('.push-result')).toHaveText(/推送成功/)

  // Step 7: 关键步骤截图
  await page.screenshot({ path: 'test-results/order-approved.png' })
})
```

### 7.5 Layer 4：视觉回归测试

**覆盖要求：** 关键页面建议有视觉回归测试。

**AI 职责：** 辅助——AI 触发截图，视觉对比由工具或人工完成。

**测试内容：**
- 页面布局是否正确
- 样式是否与设计稿一致
- 不同屏幕尺寸下的响应式表现

**工具推荐：**
- Playwright 截图对比（AI 可触发）
- Percy / Chromatic（自动化视觉回归平台）

## 8. 测试悖论防护

### 8.1 问题描述

AI 写的测试可能把 bug 编码为"正确行为"：

```
1. PRD 说：审核通过后推送单据
2. AI 理解偏差：审核动作发生后就推送（包括驳回）
3. AI 写的测试：test_audit_then_push() — 验证"审核后推送"，测试通过 ✅
4. 但这个测试验证的是 AI 的错误理解，不是正确的业务逻辑
```

**测试通过了 ≠ 行为正确。** AI 写的测试倾向于验证"代码做了什么"，而不是"代码应该做什么"。

### 8.2 防护规则

1. **测试用例必须对照 AIS 中的 Concrete Examples 编写**，而非对照 AI 自己的理解
2. **测试的期望值必须来自 AIS 的 I/O 契约**，而非 AI 推断的"合理值"
3. **产品经理审核测试用例**（至少审核核心用户旅程的 E2E 测试），确认测试验证的是正确的业务逻辑
4. **如果测试和 AIS 定义的行为不一致**，以 AIS 为准，修正测试
5. **新增测试时，检查是否与已有测试矛盾**——如果两个测试对同一行为有不同的期望，说明存在理解偏差

## 9. 反馈毒性防护

### 9.1 问题描述

人类纠正 AI 时使用自然语言，自然语言本身有歧义，导致纠正可能引入新的偏差：

```
第一轮：AI 理解有偏差 → 生成错误代码
第二轮：人类用自然语言纠正 → "这里应该是审核通过后才推送"
第三轮：AI 基于纠正重新理解 → 但"审核通过后"还是有歧义
第四轮：人类再次纠正 → 用了不同表述 → AI 理解又变了
...循环往复
```

### 9.2 防护规则

1. **纠正偏差时更新 AIS，而非口头描述。** 发现 AI 理解错误时，不是"告诉它哪里错了"，而是更新 AIS 中的具体条目
2. **AIS 是结构化的，没有歧义。** 用决策表、I/O 契约、状态机等结构化格式纠正，而非自然语言
3. **纠正记录必须同步到 AIS。** 每次纠正后，确认 AIS 已更新，避免下次出现同样的理解偏差
4. **如果同一问题被纠正 2 次以上仍未解决**，说明 AIS 中该条目的描述不够精确，需要重写而非微调

## 10. 规格同步检查

### 10.1 问题描述

项目越久，代码和 AIS 的脱节越严重：

```
第 1 周：AIS 和代码完全一致
第 2 周：改了一个 bug，代码变了，AIS 没更新
第 3 周：又改了两个功能，AIS 更没更新
第 4 周：AIS 已经和实际代码完全脱节
第 5 周：新会话的 AI 基于 AIS 写代码 → 和现有代码冲突
```

### 10.2 规格同步机制

1. **每次代码变更后，AI 必须检查 AIS 是否需要同步更新**
2. **检查维度：**
   - 新增/修改/删除的接口是否在 AIS 的 I/O 契约中反映
   - 业务规则变更是否在 AIS 的决策表中反映
   - 状态流转变更是否在 AIS 的状态机中反映
   - 异常处理变更是否在 AIS 的异常处理矩阵中反映
3. **如果代码行为和 AIS 定义不一致**，触发告警并记录到变更日志
4. **Phase 3 步骤 10（变更与生命周期管理）中增加 AIS 同步检查项**
5. **Gate 4.5（验收就绪预检）中增加 AIS 一致性预检**
