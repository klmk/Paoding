# 代码审查指南

> **定位**：本文件定义 Phase 3 中代码审查的完整流程和标准，结合 Superpowers 的两阶段审查机制和 BMad 的多层对抗审查模式。
> **适用范围**：Phase 3 中每个开发任务完成后自动触发。

---

## 1. 审查触发条件

### 1.1 自动触发

以下情况**自动触发**代码审查，无需人工请求：

| # | 触发条件 | 说明 |
|---|---------|------|
| 1 | 任务开发完成 | 开发者代理完成一个任务的所有 TDD 循环后 |
| 2 | 审查修复完成 | 开发者修复审查问题后，需要重新审查确认 |
| 3 | 重构完成 | 大规模重构后，需要审查确认重构质量 |
| 4 | Bug 修复完成 | Bug 修复后，需要审查确认修复方案的正确性 |

### 1.2 手动触发

以下情况可以**手动请求**代码审查：

- 开发者对某段代码不确定，主动请求审查
- 用户要求对特定模块进行审查
- 回归测试发现异常，需要审查相关代码

### 1.3 审查不触发的情况

以下情况**不触发**完整代码审查（但仍需自审）：

- 纯注释修改（不影响代码行为）
- 文档更新（不影响代码行为）
- 配置文件格式调整（不影响功能）

> 即使不触发完整审查，修改者仍需进行自审，确认修改的正确性。

## 2. 审查架构：两阶段 + 多层对抗

### 2.1 总体架构

```
任务完成
    │
    ▼
┌─────────────────────────────────────────┐
│  第一阶段：规格合规审查                    │
│  (Specification Compliance Review)       │
│                                          │
│  审查者：验收审计代理 (Acceptance Auditor) │
│  输入：代码 + PRD + 架构设计 + AC 列表     │
│  关注：代码是否匹配需求和设计               │
└──────────────┬──────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────┐
│  第二阶段：代码质量审查                    │
│  (Code Quality Review)                   │
│                                          │
│  审查层 1：盲猎手 (Blind Hunter)          │
│    → 仅看 diff，无上下文，找明显问题       │
│                                          │
│  审查层 2：边界猎手 (Edge Case Hunter)    │
│    → 看 diff + 项目代码，找边界问题        │
│                                          │
│  审查层 3：质量审计 (Quality Auditor)     │
│    → 看 diff + 上下文，审查工程质量        │
└──────────────┬──────────────────────────┘
               │
               ▼
        问题分类与分派
```

### 2.2 信息不对称原则

借鉴 BMad 的 Adversarial Review 理念，审查代理应遵循**信息不对称原则**：

- **盲猎手**：只看到代码变更（diff），不看到 PRD、架构设计或开发者的推理过程。这迫使审查者仅从代码本身判断问题。
- **边界猎手**：可以看到代码变更和项目代码，但看不到开发者的意图。这可以发现与项目其他部分的冲突。
- **验收审计**：可以看到代码变更、PRD 和架构设计。这确保实现与规格一致。
- **质量审计**：可以看到代码变更和项目上下文。这确保代码质量达标。

**关键约束：** 审查代理不得访问开发者的推理过程或中间决策记录。审查者评估的是最终产物，而非开发者的意图。

## 3. 第一阶段：规格合规审查

### 3.1 审查目标

确认代码实现**完全匹配** PRD 需求和架构设计，无遗漏、无偏离、无过度实现。

### 3.2 审查清单

| # | 检查项 | 说明 |
|---|--------|------|
| 1 | AC 覆盖完整性 | 每个 AC 是否都有对应的实现代码？ |
| 2 | AC 覆盖正确性 | 实现是否真正满足了 AC 的要求（而非形式上覆盖）？ |
| 3 | 需求无遗漏 | PRD 中的所有功能点是否都已实现？ |
| 4 | 无过度实现 | 是否实现了 PRD 中没有要求的功能？ |
| 5 | 架构一致性 | 代码的模块划分、接口定义、数据模型是否与架构设计一致？ |
| 6 | 接口契约 | 公开接口的签名、参数、返回值是否与设计文档一致？ |
| 7 | 数据模型 | 数据结构、存储方案是否与设计一致？ |
| 8 | 追溯完整性 | 每段代码是否都关联了 AC ID？ |

### 3.3 审查方法

```
1. 读取 PRD 中与当前任务相关的所有 AC
2. 逐条对照 AC，在代码中找到对应的实现
3. 验证实现的正确性（不仅"有代码"，还要"代码对"）
4. 检查是否有 AC 没有对应的实现（遗漏）
5. 检查是否有代码没有对应的 AC（过度实现或孤儿代码）
6. 读取架构设计，对照代码的模块结构和接口定义
7. 记录所有偏差
```

### 3.4 审查输出格式

```markdown
## 规格合规审查报告

**任务编号：** [任务编号]
**关联 AC：** [AC ID 列表]
**审查时间：** [时间戳]

### AC 覆盖矩阵

| AC ID | AC 描述 | 实现位置 | 状态 | 备注 |
|-------|---------|---------|------|------|
| AC-001 | ... | file.js:42 | 已满足 | |
| AC-002 | ... | - | 未实现 | 缺少异常处理 |
| AC-003 | ... | file.js:78 | 部分满足 | 缺少边界条件 |

### 架构一致性检查

| 检查项 | 状态 | 偏差描述 |
|--------|------|---------|
| 模块划分 | 一致 | |
| 接口定义 | 偏差 | 返回值类型与设计文档不一致 |
| 数据模型 | 一致 | |

### 发现的问题

1. **[Critical]** AC-002 未实现：缺少密码重置的邮件发送功能
2. **[Important]** 接口返回值类型偏差：设计文档要求返回 `Result<T>`，实际返回裸对象
```

## 4. 第二阶段：代码质量审查

### 4.1 审查层 1：盲猎手（Blind Hunter）

**角色定位：** 一个完全不了解项目背景的审查者，仅通过代码变更本身判断问题。

**输入：** 代码变更（diff），无其他上下文。

**审查重点：**
- 代码是否自解释？（不看注释能否理解代码意图）
- 是否有明显的逻辑错误？
- 是否有明显的安全隐患？（硬编码密码、SQL 拼接、未校验输入）
- 是否有明显的性能问题？（N+1 查询、无限循环风险、内存泄漏）
- 命名是否清晰？（变量名、函数名、类名是否表达意图）
- 是否有代码坏味道？（过长函数、深层嵌套、重复代码）

**输出格式：** Markdown 列表，每项包含一行标题和简要说明。

### 4.2 审查层 2：边界猎手（Edge Case Hunter）

**角色定位：** 一个了解项目代码的审查者，专注于发现边界条件和异常场景的遗漏。

**输入：** 代码变更（diff）+ 项目代码库的只读访问权限。

**审查重点：**
- 边界条件是否处理？（空值、零值、最大值、最小值）
- 并发场景是否考虑？（竞态条件、死锁、资源争用）
- 异常路径是否处理？（网络超时、文件不存在、权限不足）
- 数据一致性是否保证？（事务、幂等性、补偿机制）
- 与项目其他模块的交互是否安全？（版本兼容、接口变更影响）

**输出格式：** JSON 数组，每项包含：
```json
{
  "location": "file.js:42",
  "trigger_condition": "当用户名为空字符串时",
  "guard_snippet": "if (!username) throw new ValidationError(...)",
  "potential_consequence": "可能导致空指针异常"
}
```

### 4.3 审查层 3：质量审计（Quality Auditor）

**角色定位：** 一个经验丰富的代码质量专家，审查代码的工程质量和可维护性。

**输入：** 代码变更（diff）+ 项目上下文。

**审查重点：**

| 维度 | 检查项 |
|------|--------|
| **命名** | 变量/函数/类的名称是否清晰表达意图？是否遵循项目命名规范？ |
| **结构** | 函数长度是否合理（< 30 行）？模块职责是否单一？依赖方向是否正确？ |
| **设计模式** | 设计模式的使用是否恰当？是否过度设计或设计不足？ |
| **性能** | 是否有不必要的计算？是否有 N+1 查询？是否有内存泄漏风险？ |
| **安全** | 是否有注入风险？是否有越权风险？是否有信息泄露风险？ |
| **可维护性** | 代码是否易于理解？是否易于修改？是否易于测试？ |
| **测试质量** | 测试是否充分？测试是否独立？测试断言是否有意义？ |
| **文档** | 公开接口是否有文档？复杂逻辑是否有注释？注释是否解释"为什么"而非"是什么"？ |

**输出格式：** Markdown 列表，每项包含标题、位置、详细说明。

## 5. 问题分级

### 5.1 分级标准

所有审查发现的问题必须按以下标准分级：

| 级别 | 标签 | 定义 | 处理要求 |
|------|------|------|---------|
| **Critical** | `[C]` | 必须修复。问题会导致功能错误、安全漏洞、数据丢失或系统崩溃。 | 阻塞合并，必须立即修复。 |
| **Important** | `[I]` | 应该修复。问题会导致代码质量下降、维护困难或潜在风险。 | 强烈建议修复，修复后方可合并。 |
| **Suggestion** | `[S]` | 可以改进。问题不影响功能正确性，但改善后可以提升代码质量。 | 建议改进，可记录为后续优化项。不阻塞合并。 |

### 5.2 分级示例

**Critical 示例：**
- SQL 注入漏洞
- 用户权限校验缺失
- 数据库事务未正确处理（可能导致数据不一致）
- 核心业务逻辑错误（计算结果不正确）
- 敏感信息明文存储

**Important 示例：**
- 函数过长（> 50 行），职责不清
- 重复代码（DRY 违反）
- 缺少必要的错误处理
- 测试覆盖不足（关键路径缺少测试）
- 命名不清晰（变量名无法表达意图）
- 硬编码的配置值

**Suggestion 示例：**
- 可以使用更简洁的语法
- 注释可以更详细
- 可以提取公共辅助函数
- 魔法数字可以定义为常量
- 可以使用更现代的 API

### 5.3 分级决策规则

当不确定如何分级时，遵循以下规则：

1. **安全相关问题默认为 Critical** — 宁可误判为高优先级，不可漏判
2. **影响数据一致性的问题默认为 Critical** — 数据问题修复成本极高
3. **影响可维护性的问题默认为 Important** — 维护成本是长期成本
4. **纯风格问题归类为 Suggestion** — 风格问题不影响功能

## 6. 审查反馈格式

### 6.1 统一反馈格式

所有审查发现的问题必须使用以下统一格式：

```markdown
### [C/I/S] 问题标题

**位置：** `file/path.js:行号`
**关联 AC：** AC-XXX（如适用）
**审查层：** Blind Hunter / Edge Case Hunter / Quality Auditor / Acceptance Auditor

**问题描述：**
[清晰描述问题是什么，为什么这是一个问题]

**影响分析：**
[描述如果不修复，可能导致什么后果]

**修复建议：**
[提供具体的修复方案或方向]

**修复验证：**
[修复后应如何验证问题已解决]
```

### 6.2 审查报告模板

```markdown
# 代码审查报告

**任务编号：** TASK-XXX
**审查时间：** YYYY-MM-DD HH:mm
**审查者：** [审查代理角色]

## 审查概要

| 指标 | 数值 |
|------|------|
| Critical 问题数 | X |
| Important 问题数 | X |
| Suggestion 问题数 | X |
| 审查结论 | Approve / Changes Requested / Blocked |

## 第一阶段：规格合规审查

[规格合规审查结果，参见第 3.4 节格式]

## 第二阶段：代码质量审查

### Blind Hunter 发现

[盲猎手发现的问题列表]

### Edge Case Hunter 发现

[边界猎手发现的问题列表]

### Quality Auditor 发现

[质量审计发现的问题列表]

## 问题汇总

| # | 级别 | 标题 | 位置 | 审查层 |
|---|------|------|------|--------|
| 1 | [C] | ... | file.js:42 | Blind Hunter |
| 2 | [I] | ... | file.js:78 | Edge Case Hunter |
| 3 | [S] | ... | file.js:15 | Quality Auditor |

## 审查结论

**[Approve / Changes Requested / Blocked]**

[结论说明]
```

### 6.3 审查结论定义

| 结论 | 条件 | 后续动作 |
|------|------|---------|
| **Approve** | 无 Critical 问题，Important 问题 <= 2 | 代码可以合并，Important 问题记录为后续优化 |
| **Changes Requested** | 有 Critical 问题，或 Important 问题 > 2 | 必须修复所有 Critical 和 Important 问题后重新审查 |
| **Blocked** | 发现架构性偏差或需求理解错误 | 必须回退到设计阶段重新评估，修复后重新开发 |

## 7. 修复验证流程

### 7.1 修复流程

```
审查报告
    │
    ├── Critical 问题 → 必须修复 → 重新运行相关测试 → 确认修复
    │
    ├── Important 问题 → 应该修复 → 重新运行相关测试 → 确认修复
    │
    └── Suggestion 问题 → 记录为优化项 → 可在后续迭代中处理
```

### 7.2 修复要求

1. **每个修复必须遵循 TDD 流程**
   - 如果修复涉及新行为，先写测试（RED）
   - 再实现修复（GREEN）
   - 最后重构（REFACTOR）

2. **修复后必须运行全量测试**
   - 确认修复没有引入新的回归
   - 确认所有测试仍然通过

3. **修复必须附带说明**
   - 说明修复了什么问题
   - 说明修复方案和理由
   - 说明如何验证修复有效

### 7.3 重新审查

当存在 Critical 或 Important 问题时，修复后必须重新审查：

```
修复完成
    │
    ├── 运行全量测试 → 全部通过？
    │       │
    │       ├── 是 → 触发重新审查
    │       └── 否 → 修复回归问题，重新运行测试
    │
    └── 重新审查（仅审查修复部分 + 回归影响范围）
            │
            ├── 所有 Critical/Important 问题已解决 → Approve
            └── 仍有未解决问题 → 继续修复
```

**重新审查的范围：**
- 首次审查：审查全部代码变更
- 重新审查：仅审查修复部分及其影响范围（缩小审查范围以提高效率）

### 7.4 审查闭环确认

审查闭环的确认标准：

- [ ] 所有 Critical 问题已修复并验证
- [ ] 所有 Important 问题已修复并验证（或明确记录延后理由）
- [ ] 全量测试通过
- [ ] 无新的回归
- [ ] 审查者确认修复有效

## 8. 审查代理隔离与独立性

### 8.1 隔离原则

为确保审查的独立性和客观性，审查代理必须与开发者代理隔离：

| 隔离维度 | 要求 |
|---------|------|
| **上下文隔离** | 审查代理不访问开发者的推理过程、中间决策记录 |
| **角色隔离** | 审查代理和开发者代理使用不同的 AI 实例/会话 |
| **视角隔离** | 盲猎手不看到项目上下文，强制从代码本身判断 |
| **时间隔离** | 审查在开发完成后进行，不在开发过程中干预 |

### 8.2 审查偏见防护

借鉴 BMad 的 Adversarial Review 理念：

- **审查者必须找到问题** — "看起来不错"不是有效的审查结论。审查者必须主动寻找问题，零发现需要重新审查。
- **发现不等于确认** — 审查者找到的问题可能是误报（false positive）。最终由开发者或用户判断哪些问题是真实的。
- **多次审查递增价值** — 修复后重新审查通常能发现更多问题。第二次审查的价值高于第一次。

### 8.3 审查失败处理

如果审查代理在执行过程中失败（超时、异常、返回空结果）：

1. 记录失败的审查层
2. 使用其他审查层的结果继续
3. 向用户报告哪些审查层未能执行
4. 如果所有审查层都失败，警告用户审查不完整，建议人工审查

## 9. 审查记录与追溯

### 9.1 记录要求

每次代码审查必须记录以下信息：

| 记录项 | 说明 |
|--------|------|
| 任务编号 | 被审查的任务 |
| 审查时间 | 审查执行的时间 |
| 审查者 | 执行审查的代理角色 |
| 审查范围 | 审查覆盖的文件和代码行 |
| 发现问题数 | 按 Critical/Important/Suggestion 分类 |
| 审查结论 | Approve / Changes Requested / Blocked |
| 修复状态 | 每个问题的修复状态 |

### 9.2 追溯关联

审查记录应与以下信息关联：

- 任务编号 → 开发计划中的任务
- AC ID → PRD 中的验收标准
- 文件路径 → 代码仓库中的文件
- 审查结论 → Gate 3 门控检查的输入

### 9.3 审查指标统计

在 Phase 3 完成时，统计以下审查指标：

| 指标 | 说明 | 目标值 |
|------|------|--------|
| 审查覆盖率 | 被审查的任务数 / 总任务数 | 100% |
| 首次通过率 | 首次审查即 Approve 的任务数 / 总任务数 | >= 60% |
| Critical 问题总数 | 所有任务中 Critical 问题数量 | 0（最终状态） |
| 平均审查轮次 | 每个任务的平均审查次数 | <= 2 |
| 审查修复时间 | 从审查到修复完成的平均时间 | 越短越好 |
