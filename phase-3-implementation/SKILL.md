# Phase 3: 开发实施

## 概述

Phase 3 是将设计转化为可运行代码的核心阶段。本阶段以 Phase 2 输出的技术设计文档和实施计划为输入，通过 **TDD（测试驱动开发）+ 子代理协作 + 多层对抗审查** 三大机制，确保每一行生产代码都经过严格验证，可追溯到 PRD 需求。

**核心理念：** 代码质量不是事后检查的结果，而是开发过程的内在属性。通过 TDD 确保行为正确，通过子代理协作确保视角多元，通过对抗审查确保质量无死角。

**适用场景：**
- Phase 2（规划设计）已通过 Gate 2 门控，进入开发实施阶段
- 需要按 Story/任务粒度逐个实现功能
- 需要严格的质量保障机制（TDD + 代码审查 + 验证）

---

## 阶段目标

按照设计文档和实施计划，**高质量**地完成全部功能的编码实现。

"高质量"的定义：
- **功能正确**：所有验收标准（AC）满足，测试全部通过
- **代码可追溯**：每段代码可追溯到 PRD 需求（通过 AC ID）
- **设计合规**：代码实现与架构设计一致
- **无技术债务**：无已知的、未记录的遗留问题

---

## 入口条件

进入 Phase 3 之前，必须满足以下**全部**条件：

| # | 条件 | 验证方式 |
|---|------|---------|
| 1 | Phase 2 全部出口条件已满足 | Gate 2 门控检查通过 |
| 2 | 技术设计文档已通过评审 | 评审记录中有"通过"结论 |
| 3 | 实施计划已获用户确认 | 用户显式确认记录 |
| 4 | 开发环境已就绪 | 测试框架、构建工具、依赖项已配置 |
| 5 | 需求追溯矩阵已建立 | 每条需求有唯一标识和验收标准 |

**如果以上任何条件未满足，不得进入 Phase 3。** 必须回退到对应阶段补齐缺失项。

## 铁律（Iron Laws）

以下铁律在 Phase 3 中具有最高约束力，不可违反、不可绕过、不可降级。

### 铁律 P3-1：没有失败的测试就没有生产代码

```
严格遵循 TDD RED-GREEN-REFACTOR 循环：
RED    → 先编写一个失败的测试，定义期望行为
GREEN  → 编写最少量的生产代码使测试通过
REFACTOR → 在测试保护下重构代码，消除重复，提升设计
```

- 测试必须覆盖正常路径、边界条件和异常路径
- 测试失败时，必须先修复测试或生产代码，不得跳过或注释测试
- 不得为了"赶进度"而降低测试覆盖率要求

### 铁律 P3-2：没有根因调查就不能提出修复方案

```
遇到 Bug 或测试失败时：
1. 先调查根因（Root Cause Analysis）
2. 再提出修复方案
3. 禁止"试错式"修复（改一行试试、换个写法试试）
```

- 每个修复必须附带根因说明
- 无法确定根因时，必须使用系统化调试流程（参见 [`systematic-debugging-guide.md`](./systematic-debugging-guide.md)）
- 3 次修复失败必须触发升级规则（暂停并请求外部输入）

### 铁律 P3-3：完成声明必须有新鲜验证证据

```
声明"完成"时必须同时提供：
1. 测试结果：自动化测试的执行输出（通过/失败报告）
2. 运行演示：代码实际运行的截图、日志或输出
3. 对比验证：实现结果与 AC 的逐项对比
```

- "新鲜"意味着证据必须在本次会话中生成，不得复用历史证据
- 证据必须可重复——相同的输入必须产生相同的输出
- 禁止使用"应该可以"、"理论上没问题"等不确定表述

### 铁律 P3-4：每段代码必须可追溯到 PRD 需求

```
代码库中的每一行生产代码都必须能回答：
1. 为什么存在 → 解决了哪个需求？
2. 为谁服务 → 对应哪个用户场景？
3. 如何验证 → 预期行为是什么？
```

- 代码提交信息必须包含需求标识（如 `[AC-001]`）
- 测试用例必须关联 AC ID
- 代码注释中必须说明对应的需求或设计决策
- 无法追溯到需求的代码视为"孤儿代码"，必须标记并处理

## 执行流程

### 步骤 1：任务领取

**目标：** 从开发计划中取出下一个待实现的任务。

**执行要点：**
1. 读取实施计划，找到状态为"待开发"的下一个任务
2. 确认任务的前置依赖已完成（如有依赖任务，需先完成依赖）
3. 记录任务编号、任务描述、关联的 AC ID
4. 将任务状态更新为"开发中"

**输出：** 任务编号、关联 AC ID、任务描述

### 步骤 2：理解上下文

**目标：** 充分理解任务的技术上下文和业务背景。

**执行要点：**
1. 读取 PRD 中与该任务相关的需求描述和验收标准
2. 读取架构设计中与该任务相关的模块设计、接口定义、数据模型
3. 读取前序任务的代码，理解已有实现和代码风格
4. 识别该任务对其他模块的影响范围
5. 如有疑问，必须先澄清再继续（不得带着疑问写代码）

**输出：** 上下文理解记录（需求要点、设计约束、依赖关系、影响范围）

### 步骤 3：编写失败测试（RED）

**目标：** 为任务定义明确的期望行为。

**执行要点：**
1. 根据 AC 编写测试用例，覆盖正常路径、边界条件、异常路径
2. 每个测试用例必须关联 AC ID（在测试名称或注释中标注）
3. 运行测试，确认测试**失败**（这是 RED 阶段的验证）
4. 如果测试意外通过，说明测试编写有误，需重新审视

**输出：** 一组失败的测试用例，每个关联 AC ID

> 详细规范参见 [`tdd-guide.md`](./tdd-guide.md)

### 步骤 4：编写最小实现（GREEN）

**目标：** 用最少的代码使测试通过。

**执行要点：**
1. 编写刚好足够使测试通过的生产代码
2. 不追求完美设计，只追求测试通过
3. 运行测试，确认全部通过
4. 如果测试未通过，回到 RED 阶段检查测试是否正确
5. 代码提交信息必须包含 AC ID

**输出：** 通过全部测试的最小实现

### 步骤 5：重构优化（REFACTOR）

**目标：** 在测试保护下改善代码设计。

**执行要点：**
1. 消除重复代码（DRY 原则）
2. 改善命名（变量、函数、类的名称应清晰表达意图）
3. 优化结构（函数长度、模块职责、依赖方向）
4. 每次重构后运行测试，确保测试仍然全部通过
5. 重构不改变外部行为，只改变内部实现

**输出：** 重构后的代码，测试仍然全部通过

### 步骤 6：自我审查

**目标：** 开发者自行检查实现质量。

**执行要点：**
1. 逐行审查代码，检查：
   - 是否满足 AC 的全部条件（逐条对照）
   - 是否有未处理的边界条件
   - 是否有安全隐患（注入、越权、信息泄露）
   - 是否有性能问题（N+1 查询、内存泄漏、不必要的计算）
   - 命名是否清晰、注释是否充分
   - 是否可追溯到 AC ID
2. 对照反模式清单（[`../core/anti-patterns.md`](../core/anti-patterns.md)）检查
3. 记录自审发现的问题，立即修复

**输出：** 自审记录，所有发现的问题已修复

### 步骤 7：代码审查（两阶段）

**目标：** 通过独立审查确保代码质量和规格合规。

**执行要点：**
1. **第一阶段：规格合规审查** — 检查代码是否匹配 PRD 和架构设计
   - 每个 AC 是否都有对应的实现和测试
   - 实现是否与架构设计一致（模块划分、接口定义、数据模型）
   - 是否有需求遗漏或过度实现
2. **第二阶段：代码质量审查** — 检查代码的工程质量
   - 命名规范、代码结构、设计模式使用
   - 性能、安全、可维护性
   - 测试覆盖率和测试质量

> 详细规范参见 [`code-review-guide.md`](./code-review-guide.md)

**输出：** 审查报告（问题列表、分级、修复建议）

### 步骤 8：修复审查问题

**目标：** 解决审查中发现的所有问题。

**执行要点：**
1. 按问题分级处理：
   - **Critical（必须修复）**：阻塞合并，必须立即修复
   - **Important（应该修复）**：强烈建议修复，修复后方可合并
   - **Suggestion（可以改进）**：建议改进，可记录为后续优化项
2. 每个修复必须遵循 TDD 流程（如有新测试需求）
3. 修复后运行全量测试，确保无回归
4. 将修复结果反馈给审查者确认

**输出：** 所有 Critical 和 Important 问题已修复，测试全部通过

### 步骤 9：验证完成

**目标：** 最终确认任务完成质量。

**执行要点：**
1. 运行全量测试套件，确认无失败、无跳过
2. 逐条对照 AC 验证实现完整性
3. 检查代码追溯关系（AC ID 关联完整）
4. 更新任务状态为"已完成"
5. 生成完成报告（测试结果、AC 对比表、变更文件列表）

**输出：** 完成报告（含新鲜验证证据）

## 调试流程

当开发过程中遇到 Bug 或测试失败时，必须遵循系统化调试流程。

**核心原则：** 没有根因调查就不能提出修复方案。

### 调试触发条件

以下情况必须启动系统化调试流程：
- 测试持续失败且原因不明
- 修复后问题复现
- 同一问题出现 2 次以上
- 行为与预期不符但无法定位原因

### 调试流程概要

```
1. 根因调查 → 收集信息、复现问题、检查日志
2. 模式分析 → 识别模式、缩小范围、假设根因
3. 假设测试 → 设计实验、验证假设、排除法
4. 实现修复 → 最小修复、验证修复、回归测试
```

> 详细规范参见 [`systematic-debugging-guide.md`](./systematic-debugging-guide.md)

### 3 次修复失败升级规则

当同一问题的修复尝试连续失败 3 次时：
1. **立即暂停**当前修复工作
2. **记录调试日志**：已尝试的方案、失败原因、观察到的现象
3. **升级处理**：
   - 请求用户参与分析
   - 换一个 AI 实例/子代理进行独立分析
   - 重新审视问题定义（可能问题描述本身有误）
4. 不得在第 4 次尝试中继续相同的思路

## 红旗列表

以下是 AI 在 Phase 3 中可能用来跳过流程的常见借口，以及对应的反驳：

| # | AI 的借口（红旗） | 反驳 |
|---|------------------|------|
| 1 | "这个功能太简单，不需要 TDD" | 简单代码也会出错。TDD 的价值不在于代码复杂度，而在于确保行为正确性。参见铁律 P3-1。 |
| 2 | "我先写完功能再补测试" | 事后补写的测试只验证"代码做了什么"，而非"代码应该做什么"。这违反了 TDD 的核心精神。 |
| 3 | "这个 Bug 很明显，直接修就行" | "明显"是根因分析的替代品。没有根因调查就修复，可能导致修复了表象而根因仍在。参见铁律 P3-2。 |
| 4 | "应该可以了" | "应该"不是验证。完成声明必须有新鲜验证证据。参见铁律 P3-3。 |
| 5 | "这是基础设施代码，不需要追溯到需求" | 基础设施代码服务于具体需求。如果无法追溯到任何需求，应质疑该代码是否真的需要。参见铁律 P3-4。 |
| 6 | "改动很小，不需要代码审查" | 几行代码也可能引入严重缺陷。审查的成本与改动量无关，与缺陷的后果有关。 |
| 7 | "测试太多了会拖慢开发速度" | 短期看测试增加了代码量，长期看测试是最高效的回归验证手段。没有测试的代码修改成本随时间指数增长。 |
| 8 | "这个逻辑很难测试，跳过吧" | 难以测试的代码往往意味着设计有问题。应该重构设计使其可测试，而非跳过测试。 |
| 9 | "我已经仔细检查过了，不需要额外审查" | 自我审查存在盲区。独立审查能发现作者无法看到的问题。 |
| 10 | "之前的测试都通过了，这次改动很小" | 小改动可能引发连锁反应。每次改动都必须重新验证，不得依赖历史结果。 |
| 11 | "时间紧迫，先跳过重构" | 跳过重构意味着积累技术债务。债务不会消失，只会在未来以更高的成本偿还。 |
| 12 | "这个 TODO 以后再处理" | 没有截止日期的 TODO 等于永远不会处理。每个 TODO 必须附带问题描述、影响范围和处理期限。 |

**如果出现以上任何一种情况，立即停止，回到对应步骤重新执行。**

## 子代理协作机制

Phase 3 利用子代理（Sub-agents）实现多角色协作，确保代码质量的多视角保障。

### 子代理角色定义

| 角色 | 职责 | 触发时机 |
|------|------|---------|
| **开发者代理** | 执行 TDD 开发循环（步骤 3-5） | 每个任务开始时 |
| **审查代理** | 执行两阶段代码审查（步骤 7） | 任务完成后自动触发 |
| **调试代理** | 执行系统化调试流程 | 遇到疑难 Bug 时触发 |
| **验证代理** | 执行完成验证（步骤 9） | 审查修复后触发 |

### 子代理隔离原则

- **审查代理不得访问开发者的推理过程**，只能看到最终代码和测试（信息不对称原则，参见 BMad 的 Adversarial Review）
- **调试代理不得沿用开发者的修复思路**，必须独立分析
- 每个子代理使用独立的上下文，避免思维惯性

## 输出物清单

完成 Phase 3 后，应产出以下文件：

| # | 输出物 | 格式 | 最低要求 |
|---|--------|------|---------|
| 1 | 源代码 | 代码仓库 | 遵循编码规范，有注释和 AC ID 追溯标识 |
| 2 | 测试代码 | 代码仓库 | 覆盖正常路径、边界条件、异常路径，关联 AC ID |
| 3 | 测试报告 | 报告 | 测试覆盖率、通过/失败统计 |
| 4 | 代码审查记录 | 文档 | 每个任务的审查意见、问题分级、处理结果 |
| 5 | 变更日志 | 文档 | 所有需求/设计变更的记录 |
| 6 | 进度报告 | 文档 | 里程碑完成情况、偏差分析 |
| 7 | 调试记录（如有） | 文档 | 根因分析、修复方案、验证结果 |

## 出口条件（Gate 3）

Phase 3 完成并通过门控检查前，必须满足以下**全部**条件：

- [ ] 所有计划内的开发任务已完成
- [ ] 所有测试通过（无跳过、无忽略）
- [ ] 测试覆盖率满足预设标准
- [ ] 所有代码审查已完成且 Critical/Important 问题已解决
- [ ] 所有变更已记录并更新相关文档
- [ ] 代码可追溯到需求（AC ID 关联完整）
- [ ] 全量回归测试通过（新鲜证据）

### 门控类型

**硬性门控（Hard Gate）** — 必须有新鲜的测试通过证据，不得有未解决的 Critical/Important 审查问题。

---

## 注意事项

- Phase 3 的每个任务都应视为一个独立的"微型项目"，有明确的输入、输出和质量标准
- 任务之间的依赖关系必须严格遵守，不得并行开发有依赖的任务
- 如在开发过程中发现设计缺陷，必须回退到 Phase 2 修复设计，不得在代码层面"绕过"设计问题
- 变更管理：任何需求或设计变更必须先评估影响，再实施变更，最后更新追溯矩阵
