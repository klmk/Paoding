# Phase 4: 知识沉淀

## 概述

Phase 4 是项目开发完成后的知识沉淀阶段。其核心使命是将 Phase 0 至 Phase 3 全过程中的经验、决策、教训和成长，转化为**永久可复用的知识资产**，确保后续项目或沙盒重置后能快速恢复上下文。

**核心理念：** 没有知识沉淀的交付是不完整的交付。代码是项目的"躯体"，知识是项目的"灵魂"。躯体可能随环境变化而失效，但灵魂可以跨项目、跨会话、跨模型版本持续传承。

**原创概念 — AI错题本：** 借鉴人类学习中"错题本"的理念，为 AI 建立系统化的错误记录和补强规则库。每次犯错不是终点，而是成长的起点。错题本的核心价值在于：**同样的错误不犯第二次**。

**与BMad回顾机制的融合：** 借鉴 BMad 的 Retrospective 机制，通过结构化的回顾流程，从项目全局视角分析问题模式、成功经验和改进方向，形成闭环的知识管理体系。

**适用场景：**
- Phase 3（开发实施）已通过 Gate 3 门控，进入知识沉淀阶段
- 项目迭代结束时，需要总结阶段性经验
- 沙盒重置前，需要保存项目上下文
- AI 模型切换前，需要传递项目知识

## 阶段目标

| 目标 | 描述 | 验证方式 |
|------|------|---------|
| 项目知识沉淀 | 将开发过程中的经验、决策、教训转化为结构化文档 | 回顾报告、决策记录（ADR）、经验教训清单 |
| AI错题本更新 | 记录新发生的错误案例，生成或更新补强规则 | AI错题本（新增条目、更新补强规则） |
| AI能力画像更新 | 根据项目实际表现，调整AI在各维度上的能力评分 | 更新后的AI能力画像（含变化说明） |
| 项目知识库整理 | 生成 project-context.md，供后续项目或沙盒重置后快速恢复上下文 | project-context.md 文档 |
| 开发履历输出 | 按节点整理阶段性成果，形成完整的开发履历 | 开发履历文档 |
| 永久保存 | 所有文档同步至永久存储位置，确保不因会话结束或沙盒重置而丢失 | 文档已保存至永久位置的确认记录 |

## 入口条件

进入 Phase 4 之前，必须满足以下**全部**条件：

| # | 条件 | 验证方式 |
|---|------|---------|
| 1 | Phase 3 全部出口条件已满足 | Gate 3 门控检查通过 |
| 2 | 所有计划内开发任务已完成 | 任务状态全部为"已完成" |
| 3 | 所有测试通过（无跳过、无忽略） | 新鲜的测试通过证据 |
| 4 | 所有代码审查已完成且问题已解决 | 审查记录中有"通过"结论 |
| 5 | 代码可追溯到需求（AC ID 关联完整） | 追溯矩阵完整 |

**如果以上任何条件未满足，不得进入 Phase 4。** 必须回退到 Phase 3 补齐缺失项。

## 铁律（Iron Laws）

以下铁律在 Phase 4 中具有最高约束力，不可违反、不可绕过、不可降级。

### 铁律 P4-1：每个问题必须记录根因

```
记录问题时，必须回答：
1. 发生了什么？（错误表现）
2. 为什么发生？（根因分析）
3. 如何避免？（补强规则）

只记录表象不分析根因的问题记录，等于没有记录。
```

- 根因分析必须深入到"可操作"的层面——能据此制定预防措施
- 禁止使用"疏忽"、"不小心"等模糊归因
- 如果根因不明确，标注为"待深入分析"并制定后续调查计划

### 铁律 P4-2：每个经验必须可复用

```
记录经验时，必须包含：
1. 适用场景（在什么情况下可以使用这条经验）
2. 具体做法（具体应该怎么做）
3. 预期效果（这样做能带来什么好处）
4. 注意事项（有什么限制或风险）

无法复用的经验记录，等于日记。
```

- 经验必须剥离项目特定信息，提炼为通用规则
- 每条经验必须有明确的适用范围，避免过度泛化
- 经验应与AI错题本中的补强规则形成呼应

### 铁律 P4-3：知识必须永久保存

```
所有Phase 4产出的知识文档必须：
1. 保存至永久存储位置（不依赖当前会话或沙盒环境）
2. 使用结构化格式（Markdown），便于检索和引用
3. 包含元数据（项目名称、日期、版本、关联阶段）
4. 建立索引（便于跨项目检索）

知识只在被保存时才有价值。保存在会话记忆中的知识，会在会话结束时丢失。
```

- 永久存储位置由使用者在 Phase 0 中定义，或使用框架默认位置
- 每次保存必须确认文件已成功写入
- 保存后必须生成保存确认记录

### 铁律 P4-4：AI能力画像必须如实更新

```
基于项目实际表现更新AI能力画像时：
1. 必须基于客观证据（测试通过率、审查问题数、返工次数等）
2. 不得为了"好看"而虚报评分
3. 评分变化必须有具体的项目事件作为依据
4. 薄弱环节必须同步更新补强策略
```

- 评分上调和下调使用相同的标准，不偏不倚
- 如果项目暴露了新的薄弱环节，必须如实记录
- 能力画像的更新是持续改进的基础，诚实比好看更重要

### 铁律 P4-5：回顾必须覆盖全流程

```
项目回顾必须覆盖Phase 0至Phase 3的全部阶段：
1. 画像阶段：画像准确性、对齐有效性
2. 需求阶段：需求质量、变更频率、遗漏情况
3. 设计阶段：设计合理性、选型正确性、计划准确性
4. 开发阶段：代码质量、测试覆盖、返工情况

只回顾部分阶段的回顾，是选择性记忆，不是真正的回顾。
```

- 回顾应关注"流程本身"的效果，而非仅关注"产出物"的质量
- 每个阶段至少回答：什么做得好？什么做得不好？下次怎么改进？

## 执行流程

### 步骤 1：项目回顾

**目标：** 从全局视角分析开发过程中的问题模式、成功经验和技术债务。

**执行要点：**

1. **收集回顾素材**：
   - Phase 0-3 的全部文档（画像、PRD、架构、计划、审查记录、变更日志）
   - 开发过程中的关键事件时间线
   - 测试报告、审查报告、进度报告
   - 使用者反馈记录

2. **执行结构化回顾**（使用 `retrospective-template.md`）：
   - **项目概览**：目标 vs 实际结果，偏差分析
   - **成功经验**：识别效果好的做法，评估可复用性
   - **问题模式**：识别出现 2 次以上的共性问题，进行根因分析
   - **技术债务**：记录所有已知的捷径、原因、影响和偿还计划
   - **速度模式分析**：分析开发效率趋势，识别加速和减速因素
   - **返工循环分析**：统计测试→返工的循环轮次、每轮缺陷数量和类别分布，评估返工预算的使用效率，识别导致反复返工的根因（需求偏差/设计缺陷/AI 理解偏差）

3. **生成行动项**：
   - 每个行动项有明确的描述、负责人、截止日期和成功标准
   - 行动项按优先级排序（高/中/低）
   - 高优先级行动项必须在下一个项目开始前完成

**输出：** 填写完成的项目回顾报告（`retrospective-template.md`）

---

### 步骤 2：AI错题本更新

**目标：** 基于项目回顾中识别的问题，更新AI错题本。

**执行要点：**

1. **新增错误案例**（使用 `ai-error-notebook-template.md`）：
   - 从项目回顾的"问题模式"中提取错误案例
   - 从开发过程中的Bug修复记录中提取错误案例
   - 从代码审查记录中提取问题案例
   - 从使用者反馈中提取不满意案例

2. **为每个错误案例填写完整记录**：
   - 错误ID（ERR-001 递增）
   - 错误场景、错误表现、根因分析
   - 关联或新建补强规则
   - 触发条件、验证方式、适用范围

3. **更新补强规则**：
   - 检查已有补强规则是否需要更新（基于新经验）
   - 评估补强规则的实际效果（在项目中是否有效防止了同类错误）
   - 对无效的补强规则进行修订或替换

4. **更新统计摘要**：
   - 高频错误 TOP 10
   - 补强规则覆盖率（已有补强规则覆盖的错误占比）
   - 错误分类分布（需求理解/代码生成/测试编写/架构设计/业务逻辑）

**输出：** 更新后的AI错题本（`ai-error-notebook-template.md`）

---

### 步骤 3：AI能力画像更新

**目标：** 根据项目实际表现，更新AI在各维度上的能力评分。

**执行要点：**

1. **收集能力评估证据**：
   - 代码生成质量：审查问题数量和严重程度分布
   - 需求理解准确度：需求变更次数和原因分析
   - 测试编写能力：测试覆盖率、边界条件遗漏次数
   - 架构设计能力：设计评审问题数量、返工次数
   - 业务逻辑实现：Bug 数量和类型分布
   - 文档与沟通：文档返工次数、使用者满意度

2. **逐维度更新评分**：
   - 对照 Phase 0 的初始画像，逐项对比实际表现
   - 评分变化必须有具体的项目事件作为依据
   - 新发现的薄弱环节必须记录并制定补强策略

3. **更新历史表现记录**：
   - 在"过往项目表现"中新增本项目记录
   - 更新能力趋势分析
   - 如有模型版本变化，记录在模型更新记录中

4. **更新补强策略**：
   - 基于新识别的薄弱环节，更新补强策略
   - 评估现有补强策略的效果，淘汰无效策略
   - 新增基于项目经验的补强策略

**输出：** 更新后的AI能力画像（`ai-capability-profile-template.md`）

---

### 步骤 4：项目知识库整理

**目标：** 生成 project-context.md，供后续项目或沙盒重置后快速恢复上下文。

**执行要点：**

1. **整理项目基本信息**：
   - 项目名称、描述、技术栈、架构概要
   - 核心业务规则和约束条件
   - 编码规范和约定

2. **记录已知陷阱**：
   - 开发过程中遇到的常见错误和解决方案
   - 边界条件和性能瓶颈
   - 第三方依赖的已知问题

3. **整理关键决策记录**：
   - 技术选型决策及理由
   - 架构设计决策及理由
   - 重要的需求变更决策及理由

4. **记录文件结构说明**：
   - 核心目录和文件的职责
   - 模块间的依赖关系
   - 外部依赖清单

5. **整理待办事项**：
   - 未完成的功能
   - 已知问题
   - 技术债务偿还计划

**输出：** 填写完成的 project-context.md（`project-context-template.md`）

---

### 步骤 5：开发履历输出

**目标：** 按节点整理阶段性成果，形成完整的开发履历。

**执行要点：**

1. **按阶段整理成果**：
   - Phase 0：画像成果、对齐策略
   - Phase 1：PRD 文档、需求追溯矩阵
   - Phase 2：架构文档、开发计划、评审记录
   - Phase 3：源代码、测试报告、审查记录、变更日志

2. **整理关键里程碑**：
   - 每个阶段的开始和完成时间
   - 关键决策的时间点和内容
   - 重要变更的时间点和影响

3. **整理量化指标**：
   - 需求总数、完成率、变更率
   - 代码行数、测试覆盖率
   - 审查问题数量和分布
   - 返工次数和原因分布

4. **生成开发履历文档**：
   - 时间线形式展示项目进展
   - 量化数据支撑结论
   - 标注关键学习点

**输出：** 开发履历文档

---

### 步骤 6：永久保存

**目标：** 确保所有知识文档安全保存至永久存储位置。

**执行要点：**

1. **确认保存位置**：
   - 使用 Phase 0 中定义的永久存储位置
   - 如未定义，使用框架默认位置

2. **执行保存操作**：
   - 项目回顾报告
   - AI错题本（含新增和更新内容）
   - AI能力画像（更新版）
   - project-context.md
   - 开发履历文档
   - 所有 Phase 0-3 的文档

3. **生成保存确认记录**：
   - 文件名、保存路径、保存时间
   - 文件大小和校验信息（如适用）
   - 保存状态确认

4. **建立索引**：
   - 更新知识库索引（如有）
   - 建立跨项目引用关系
   - 标注知识的时效性

**输出：** 保存确认记录

## 红旗列表

以下是 AI 在 Phase 4 中可能用来跳过或简化流程的常见借口，以及对应的反驳：

| # | AI 的借口（红旗） | 反驳 |
|---|------------------|------|
| 1 | "项目很顺利，没什么好回顾的" | 顺利本身就是最重要的经验。为什么顺利？哪些做法值得保持？这些问题的答案对后续项目至关重要。参见铁律 P4-5。 |
| 2 | "错误已经修了，不需要再记录" | 修复错误只解决了当前问题，记录错误才能防止未来重犯。没有记录的错误，会在下一个项目中以相同的形式再次出现。参见铁律 P4-1。 |
| 3 | "AI能力画像不需要更新，变化不大" | "变化不大"不是不更新的理由。即使评分不变，项目过程中暴露的具体问题和改进点也应记录。参见铁律 P4-4。 |
| 4 | "知识沉淀可以后面再做" | 会话结束后上下文丢失，后面再做等于从零开始。知识沉淀必须在当前会话中完成。参见铁律 P4-3。 |
| 5 | "这些经验太项目特定了，没法复用" | 项目特定的经验可以提炼为通用规则。关键在于识别经验中的通用模式，而非照搬具体做法。参见铁律 P4-2。 |
| 6 | "project-context.md 跟PRD重复了" | PRD 描述"要做什么"，project-context.md 描述"做了什么、怎么做的、踩了什么坑"。两者互补，不可替代。 |
| 7 | "开发履历就是复制各阶段的文档" | 履历不是文档的堆砌，而是从时间线视角提炼关键节点、量化指标和学习点。复制文档不等于整理履历。 |
| 8 | "错题本记录太多了，看着烦" | 错题本的价值不在于"被阅读"，而在于"被检索"。当遇到类似场景时，能快速找到对应的补强规则，这才是核心价值。 |
| 9 | "这个补强规则在项目中没用上，删了吧" | 没用上不代表不需要。可能只是项目中没有触发对应的场景。补强规则应保留，并在适用范围中标注触发条件。 |
| 10 | "知识沉淀是软性门控，可以跳过" | Gate 4 确实是软性门控，但"软性"指的是允许在交付后补充，而不是允许不执行。知识沉淀的完整性直接影响下一个项目的启动效率。 |

**如果出现以上任何一种情况，立即停止，回到对应步骤重新执行。**

## 出口条件（Gate 4）

Phase 4 完成并通过门控检查前，必须满足以下**全部**条件：

- [ ] 项目回顾报告已完成，覆盖 Phase 0 至 Phase 3 全部阶段
- [ ] AI错题本已更新，新增错误案例有完整记录（含根因分析和补强规则）
- [ ] AI能力画像已基于实际表现更新，评分变化有客观依据
- [ ] project-context.md 已生成，覆盖全部必要章节
- [ ] 开发履历已整理，含关键里程碑和量化指标
- [ ] 所有文档已保存至永久存储位置
- [ ] 保存确认记录已生成

### 门控类型

**软性门控（Soft Gate）** — 知识沉淀的完整性由 AI 自评，但必须产出全部必输出物。建议完整，但允许在 Phase 5 交付后补充完善。

## 与其他阶段的关系

```
Phase 0（双向画像）
    │
    │  提供初始AI能力画像、使用者画像、对齐策略
    ▼
Phase 1（需求分析）── PRD ──→ Phase 2（规划设计）── 架构+计划 ──→ Phase 3（开发实施）
                                                                              │
                                                                              │  提供开发成果、审查记录、测试报告
                                                                              ▼
                                                                        Phase 4（知识沉淀）
                                                                              │
                                                                              │  输出：回顾报告、AI错题本、能力画像更新、project-context.md
                                                                              ▼
                                                                        Phase 5（交付标准）
                                                                              │
                                                                              │  反哺：改进建议 → 下一个项目的 Phase 0
                                                                              ▼
                                                                        下一个项目 / 沙盒重置
```

**Phase 4 对前置阶段的依赖：**
- Phase 0 的 AI 能力画像：作为能力更新的基准
- Phase 1 的 PRD：作为回顾中"目标 vs 实际"的参照
- Phase 2 的架构和计划：作为回顾中"设计合理性"的评估依据
- Phase 3 的审查记录、测试报告、变更日志：作为问题分析和经验提炼的数据源

**Phase 4 对后续阶段/项目的价值：**
- project-context.md：供沙盒重置后快速恢复上下文
- AI错题本：供下一个项目避免重复犯错
- 更新后的AI能力画像：供下一个项目的 Phase 0 使用
- 项目回顾报告中的行动项：供流程改进参考

---

## 注意事项

- Phase 4 是"软性门控"，但软性不等于可选。知识沉淀的质量直接影响后续项目的启动效率
- AI错题本是原创概念，其核心价值在于"补强规则"——不仅要记录错误，更要定义如何避免错误
- 回顾应保持客观和诚实。过度美化项目表现会导致错题本失去价值
- project-context.md 是最重要的"快速恢复"文档，沙盒重置后应首先加载此文件
- 知识沉淀是一个持续的过程。在项目执行过程中，就应逐步积累素材，而非在 Phase 4 集中补写
- 每个补强规则都应有"触发条件"，当条件满足时，AI 应主动查阅对应的补强规则
