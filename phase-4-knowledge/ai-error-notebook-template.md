# AI 错题本（AI Error Notebook）

> **定位：** AI 错题本是 AI-Dev-Framework 的原创知识管理工具。借鉴人类学习中"错题本"的理念，系统化地记录 AI 在开发过程中的错误案例，并为每个错误生成对应的"补强规则"，确保同样的错误不犯第二次。
>
> **核心理念：** 每次犯错不是终点，而是成长的起点。错题本的价值不在于记录错误本身，而在于从错误中提炼出可执行的补强规则，在未来的类似场景中主动预防。
>
> **使用方式：** 在 Phase 4 知识沉淀阶段，基于项目回顾中识别的问题，更新本错题本。在后续项目的开发过程中，当遇到类似场景时，应主动查阅对应的补强规则。

#### 错误场景

> 描述在什么情况下发生了这个错误。包括：
> - 正在执行什么任务
> - 前置条件是什么
> - 涉及哪些文件或模块

[待填写]

#### 错误表现

> 描述错误的具体表现。包括：
> - 实际发生了什么（错误信息、异常行为、不符合预期的输出）
> - 预期应该发生什么

> - 分析到"可操作"的层面（能据此制定预防措施）

> - 描述清晰、无歧义

> - 有明确的触发条件

> - 条件具体、可识别
> - 覆盖错误发生的典型场景
> - 避免过于宽泛（"任何时候"）或过于狭窄（仅限某一次）

> 描述如何验证补强规则是否有效。要求：
> - 有明确的验证标准
> - 可通过测试、审查或检查清单验证
> - 包含"通过"和"不通过"的判定标准

> - 明确适用的项目类型、技术栈、开发阶段
> - 标注不适用的场景（避免过度应用）
> - 标注适用范围是否为"通用"（跨项目适用）或"项目特定"

## 三、错误案例记录

> **说明：** 以下区域用于记录具体的错误案例。每条记录遵循上述格式。

### ERR-001：[示例] 需求理解偏差导致功能实现方向错误

| 字段 | 内容 |
|------|------|
| **错误ID** | ERR-001 |
| **错误分类** | 需求理解 |
| **严重程度** | Critical |
| **发现阶段** | Phase 3 |
| **发现时间** | [示例日期] |
| **关联项目** | [示例项目] |

#### 错误场景

在实现用户登录功能时，PRD 中描述"用户可以使用邮箱或手机号登录"。AI 将其理解为"用户注册时必须同时提供邮箱和手机号"，导致注册流程的设计与使用者期望不符。

#### 错误表现

- 注册表单要求用户同时填写邮箱和手机号
- 使用者反馈："我只想用邮箱注册，为什么还要填手机号？"
- 代码审查中发现注册逻辑与 PRD 描述不一致

#### 根因分析

1. **直接原因**：AI 将"或"（OR）关系误解为"和"（AND）关系
2. **深层原因**：AI 在理解自然语言描述的逻辑关系时存在偏差，尤其是"或/且/非"等逻辑连接词
3. **流程原因**：Phase 1 需求分析阶段未将自然语言描述转化为形式化的逻辑表达式，导致歧义未被及时发现

#### 补强规则ID

RULE-001

#### 补强规则内容

**规则名称：** 逻辑关系显式化规则

当 PRD 或需求描述中包含逻辑连接词（"或"、"且"、"非"、"必须"、"可选"等）时，必须：

1. 将自然语言描述转化为形式化的逻辑表达式
2. 使用表格或决策树明确列出所有组合情况
3. 与使用者确认逻辑表达式的正确性
4. 在测试用例中覆盖所有逻辑分支

**检查清单：**
- [ ] 需求中的"或"关系是否已转化为明确的分支逻辑？
- [ ] 需求中的"且"关系是否已转化为必要条件检查？
- [ ] 需求中的"可选"项是否已明确标注默认行为？
- [ ] 所有逻辑分支是否都有对应的测试用例？

#### 触发条件

- PRD 或需求描述中出现"或"、"且"、"非"、"必须"、"可选"等逻辑连接词
- 需求描述涉及多条件组合判断
- 使用者描述的功能涉及多种输入方式的组合

#### 验证方式

- 检查 PRD 中所有包含逻辑连接词的需求是否都有对应的形式化逻辑表达式
- 检查测试用例是否覆盖了所有逻辑分支
- 向使用者确认逻辑表达式的正确性

#### 适用范围

- **通用**：适用于所有项目
- **阶段**：Phase 1 需求分析、Phase 3 开发实施
- **不适用**：纯技术实现细节（不涉及业务逻辑的代码层面优化）

## 七、使用指南

### 7.1 何时查阅错题本

AI 应在以下时机主动查阅错题本：

1. **Phase 1 需求分析**：分析需求前，查阅"需求理解"类的补强规则
2. **Phase 2 规划设计**：设计架构前，查阅"架构设计"类的补强规则
3. **Phase 3 开发实施**：编写代码和测试前，查阅"代码生成"和"测试编写"类的补强规则
4. **Phase 4 知识沉淀**：更新错题本，新增错误案例和补强规则

### 7.2 如何新增错误记录

1. 从项目回顾、审查记录、Bug 修复记录中识别错误案例
2. 按照第二节的格式填写完整记录
3. 为错误分配唯一的错误ID（ERR-XXX 递增）
4. 如果没有对应的补强规则，创建新规则（RULE-XXX 递增）
5. 更新错误分类索引
6. 更新统计摘要

### 7.3 如何使用补强规则

1. 识别当前任务是否触发了某个补强规则的"触发条件"
2. 如果触发，阅读并应用对应的补强规则
3. 在完成任务后，验证补强规则是否有效
4. 如果补强规则无效或需要调整，记录反馈并在 Phase 4 更新

### 7.4 错题本的维护原则

- **完整性**：每条错误记录必须包含全部字段
- **可操作性**：每条补强规则必须可执行、可验证
- **时效性**：定期评估补强规则的有效性，淘汰无效规则
- **可检索性**：通过分类索引和统计摘要，确保快速检索
- **诚实性**：如实记录错误，不美化、不隐瞒
