# 框架优化闭环指南

> **定位**：本文件定义了框架优化的完整闭环流程、决策矩阵、版本管理规则，以及 Module 8 与其他模块的关系。
> **核心原则**：框架优化是持续循环的过程，不是一次性活动。每个优化决策必须有数据支撑，每个优化效果必须经过验证。

---

## 一、闭环流程

框架优化遵循以下六步闭环：

```
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  ① 数据采集 ──→ ② 聚合分析 ──→ ③ 识别问题                    │
│       ↑                                    │                │
│       │                                    ↓                │
│  ⑥ 再验证 ←── ⑤ 版本发布 ←── ④ 优化规则                      │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 步骤 1：数据采集

**做什么：** 在每个功能/模块完成后，收集约束追溯数据。

**怎么做：**
1. 使用 `constraint-trace-template.md` 模板记录约束追溯
2. 确保数据真实、完整、及时
3. 每次迭代完成后，使用 `iteration-snapshot-template.md` 生成快照
4. 将快照数据（JSON 格式）归档到统一的数据目录

**输入：** 开发过程中的实际约束触发情况
**输出：** 约束追溯记录 + 迭代快照（Markdown + JSON）

**质量标准：**
- 每个产出物都有对应的追溯记录
- 追溯数据基于真实开发过程，无编造
- JSON 数据格式规范，可被聚合分析脚本读取

### 步骤 2：聚合分析

**做什么：** 跨项目/跨迭代分析约束有效性。

**怎么做：**
1. 加载所有快照数据
2. 使用 `aggregate-analysis-template.md` 模板执行分析
3. 计算每条约束的综合触发率和平均效果评分
4. 按约束分类分别统计
5. 识别高价值约束和冗余约束候选

**输入：** 所有迭代快照的 JSON 数据
**输出：** 聚合分析报告

**触发条件：** 累积至少 3 个项目或 5 次迭代的数据后执行。建议每季度一次。

**质量标准：**
- 分析覆盖所有已归档的快照数据
- 统计方法一致，结果可复现
- 明确标注数据覆盖范围和局限性

### 步骤 3：识别问题

**做什么：** 基于聚合分析结果，识别框架中存在的问题。

**怎么做：**
1. 使用优化决策矩阵（见第二节）对每条约束进行分类
2. 识别四类问题约束：
   - **高频 + 低效**：触发率高但效果评分低，需要修订
   - **低频 + 重要**：触发率低但属于关键规则，需要加强宣导
   - **低频 + 不重要**：触发率低且效果评分低，考虑移除
   - **缺失规则**：迭代快照中多次建议新增的规则
3. 为每个问题约束制定初步处理方案

**输入：** 聚合分析报告
**输出：** 问题约束清单 + 初步处理方案

### 步骤 4：优化规则

**做什么：** 对识别出的问题约束执行具体的优化操作。

**怎么做：**
1. **修订规则**：修改约束描述、调整适用范围、补充操作指引
2. **移除规则**：从框架文件中删除冗余约束
3. **新增规则**：编写新规则并添加到对应的框架文件中
4. **加强宣导**：为重要但低频的规则增加提示和强调
5. 每个优化操作必须记录：变更内容、数据依据、预期效果

**输入：** 问题约束清单 + 初步处理方案
**输出：** 修改后的框架文件

**质量标准：**
- 每个变更都有数据依据
- 变更不影响其他规则的约束力
- 变更后的规则描述清晰、无歧义

### 步骤 5：版本发布

**做什么：** 发布框架新版本，记录变更。

**怎么做：**
1. 按照版本管理规则（见第三节）确定新版本号
2. 编写版本变更记录
3. 更新所有受影响的 Markdown 文件
4. 通知所有使用者（AI 代理和人员）新版本的变化

**输入：** 修改后的框架文件
**输出：** 框架新版本 + 版本变更记录

### 步骤 6：再验证

**做什么：** 在后续迭代中使用新版本框架，验证优化效果。

**怎么做：**
1. 在后续迭代中全面使用新版本框架
2. 按正常流程收集约束追溯数据
3. 对比优化前后的数据指标：
   - 触发率变化（修订的规则是否更常被触发？）
   - 效果评分变化（修订的规则效果是否提升？）
   - 冗余规则是否确实不再需要？
   - 新增规则是否被有效使用？
4. 将验证结果反馈到下一轮聚合分析（回到步骤 2）

**输入：** 新版本框架 + 新的约束追溯数据
**输出：** 优化效果验证报告

## 二、优化决策矩阵

根据约束的**触发率**和**效果评分**，将约束分为四个象限，分别采取不同的优化策略。

### 2.1 决策矩阵

```
                    效果评分
                 低 (< 3.0)    高 (>= 3.0)
              ┌──────────────┬──────────────┐
   高          │              │              │
(>= 60%)      │  修订        │  保留        │
              │              │              │
   触发率      ├──────────────┼──────────────┤
              │              │              │
   低          │  移除        │  加强宣导    │
(< 60%)       │              │              │
              └──────────────┴──────────────┘
```

### 2.2 四象限详细说明

#### 象限一：高频 + 高效 → 保留

**定义：** 综合触发率 >= 60% 且 平均效果评分 >= 3.0

**策略：** 保留现有规则，不做修改。

**操作：**
- 在聚合分析报告中标注为"高价值约束"
- 在框架文件中保持不变
- 可选：增加示例说明，帮助使用者更好地理解和应用

**示例：** 测试铁律（IL-2）在多个项目中触发率高、效果评分高，说明 TDD 流程被广泛接受且效果显著，应保留。

#### 象限二：高频 + 低效 → 修订

**定义：** 综合触发率 >= 60% 且 平均效果评分 < 3.0

**策略：** 规则被频繁触发但效果不佳，需要分析原因并修订。

**操作：**
- 诊断问题原因（描述模糊？过于严格？与场景脱节？缺少操作指引？）
- 修订规则描述或调整适用范围
- 补充操作指引或示例
- 在下一轮迭代中验证修订效果

**示例：** 某补强规则在 80% 的场景中被触发，但效果评分仅 2.5。经分析发现规则描述过于笼统，执行者理解不一致。修订后补充了具体的操作步骤和判断标准。

#### 象限三：低频 + 高效 → 加强宣导

**定义：** 综合触发率 < 60% 且 平均效果评分 >= 3.0

**策略：** 规则效果好但使用率低，可能是使用者不知道或不重视。

**操作：**
- 分析低频原因（场景确实少？使用者不知道？执行成本高？）
- 如果是宣导不足：在框架文件中增加强调和提示
- 如果是执行成本高：优化规则使其更易执行
- 如果是场景确实少：保留规则，但标注为"特定场景适用"

**注意：** 对于铁律类约束，即使触发率低也应保留，因为铁律的设计意图是"兜底防护"——低触发率可能恰恰说明铁律起到了预防作用。

#### 象限四：低频 + 低效 → 移除

**定义：** 综合触发率 < 60% 且 平均效果评分 < 3.0

**策略：** 规则使用率低且效果差，考虑移除。

**操作：**
- 确认数据量是否充足（至少 3 个项目或 5 次迭代）
- 排除"场景确实不适用"的情况（这种规则应保留，只是标注适用范围）
- 如果确认冗余：制定移除计划，评估风险，准备回滚方案
- 移除后在后续迭代中验证是否产生负面影响

**注意：** 移除规则应谨慎。建议先标记为"待观察"状态，经过一轮迭代验证后再正式移除。

### 2.3 特殊情况处理

| 特殊情况 | 处理方式 |
|---------|---------|
| 铁律触发率低 | 不自动移除。铁律是"兜底防护"，低触发率可能说明预防有效。需结合效果评分和具体场景分析。 |
| 规则效果评分波动大（标准差 > 1.0） | 不急于决策。分析波动原因——可能是场景差异导致，需要分场景评估。 |
| 数据量不足（< 3 个项目或 < 5 次迭代） | 标注"数据不足"，不做优化决策，继续积累数据。 |
| 规则在某类项目中高频、在另一类项目中低频 | 分析场景差异，考虑将规则细化为多个子规则，分别适用于不同场景。 |

## 三、版本管理规则

### 3.1 版本号规范

框架版本号遵循语义化版本（Semantic Versioning）规范：

```
MAJOR.MINOR.PATCH

MAJOR  — 不兼容的框架变更（如移除铁律、重构核心流程）
MINOR  — 向后兼容的功能性变更（如新增规则、修订规则描述）
PATCH  — 向后兼容的问题修复（如修正规则描述中的歧义、更新示例）
```

**版本号递增规则：**

| 变更类型 | 版本号变化 | 示例 |
|---------|-----------|------|
| 移除一条铁律 | MAJOR + 1 | v1.3.0 → v2.0.0 |
| 新增一条铁律 | MINOR + 1 | v1.3.0 → v1.4.0 |
| 修订一条规则描述 | MINOR + 1 | v1.3.0 → v1.4.0 |
| 修正规则描述歧义 | PATCH + 1 | v1.3.0 → v1.3.1 |
| 新增一条反模式 | MINOR + 1 | v1.3.0 → v1.4.0 |
| 移除一条反模式 | PATCH + 1 | v1.3.0 → v1.3.1 |
| 更新模板格式 | MINOR + 1 | v1.3.0 → v1.4.0 |

### 3.2 变更日志

每次版本发布必须编写变更日志。变更日志格式：

```markdown
# 框架变更日志

## [版本号] - 日期

### 数据依据
- 聚合分析报告编号：AA-XXXX-XX
- 覆盖项目数：X
- 覆盖迭代数：X

### 新增（Added）
- [约束编号] 约束名称：变更说明（数据依据：触发率 X%，效果评分 X.X）

### 修订（Changed）
- [约束编号] 约束名称：变更说明（数据依据：触发率 X%，效果评分 X.X）

### 移除（Removed）
- [约束编号] 约束名称：变更说明（数据依据：触发率 X%，效果评分 X.X）

### 修复（Fixed）
- [约束编号] 约束名称：变更说明

### 影响评估
- Phase 0：影响说明
- Phase 1：影响说明
- Phase 2：影响说明
- Phase 3：影响说明
- Phase 4：影响说明
- Phase 5：影响说明
```

### 3.3 回滚机制

当新版本框架在实际使用中出现严重问题时，可执行回滚。

**回滚触发条件：**
- 新增或修订的规则导致开发流程严重受阻
- 移除的规则被证明仍然有必要
- 多个项目反馈新版本存在系统性问题

**回滚流程：**
1. 立即停止使用新版本框架
2. 恢复到上一个版本的框架文件（使用版本控制系统）
3. 记录回滚原因、影响范围和受影响的项目
4. 分析回滚原因，识别优化决策中的问题
5. 在下一轮聚合分析中重新评估被回滚的变更
6. 回滚版本号格式：`{目标版本}-rollback.{序号}`（如 `v1.4.0-rollback.1`）

**回滚注意事项：**
- 回滚不是失败，是质量保障的一部分
- 每次回滚都必须记录详细原因，作为后续优化的输入
- 连续两次回滚同一类变更，说明优化方向可能有问题，需要重新审视

### 3.4 版本文件管理

| 文件 | 存放位置 | 说明 |
|------|---------|------|
| 当前版本文件 | `AI-Dev-Framework/` 各模块目录 | 正在使用的框架文件 |
| 历史版本文件 | `AI-Dev-Framework/.version-history/` | 按版本号归档的历史文件 |
| 变更日志 | `AI-Dev-Framework/CHANGELOG.md` | 所有版本的变更记录 |
| 版本元数据 | `AI-Dev-Framework/version.json` | 当前版本号、发布日期、变更摘要 |

## 四、与其他模块的关系

Module 8 是框架的"元分析"模块，其分析结果反哺整个框架体系。以下是 Module 8 与各模块的具体关系。

### 4.1 关系总览

```
                    ┌─────────────────────┐
                    │    Module 8          │
                    │  约束有效性分析       │
                    └─────────┬───────────┘
                              │
              分析结果反哺（数据驱动的规则优化）
                              │
        ┌─────────┬───────────┼───────────┬─────────┐
        ↓         ↓           ↓           ↓         ↓
   ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐
   │Phase 0 │ │Phase 1 │ │Phase 2 │ │Phase 3 │ │Phase 4 │
   │双向画像│ │需求分析│ │规划设计│ │开发实施│ │知识沉淀│
   └────────┘ └────────┘ └────────┘ └────────┘ └────────┘
        │         │           │           │         │
        └─────────┴───────────┼───────────┴─────────┘
                              ↓
                        ┌────────┐
                        │Phase 5 │
                        │交付标准│
                        └────────┘
```

### 4.2 具体影响关系

#### Module 8 → Phase 0（双向画像与对齐）

| 分析发现 | 可能的优化操作 | 影响文件 |
|---------|--------------|---------|
| 画像模板中的某些字段从未被填写 | 简化画像模板，移除冗余字段 | `phase-0-profiling/user-profile-template.md` |
| AI 能力画像的评估维度不够准确 | 增加或调整能力评估维度 | `phase-0-profiling/ai-capability-profile-template.md` |
| 对齐策略中的补强措施效果不佳 | 修订补强措施的操作指引 | `phase-0-profiling/alignment-strategy-template.md` |
| Phase 0 的铁律触发率偏低 | 分析原因，可能需要调整画像流程的触发条件 | `phase-0-profiling/SKILL.md` |

#### Module 8 → Phase 1（需求分析）

| 分析发现 | 可能的优化操作 | 影响文件 |
|---------|--------------|---------|
| 需求铁律（IL-1）效果评分高 | 保留，可能增加更多示例 | `core/iron-laws.md` |
| 需求收集指南中的某些步骤被跳过 | 简化流程或加强该步骤的必要性说明 | `phase-1-analysis/requirement-collection-guide.md` |
| PRD 模板中的某些章节从未被使用 | 移除或合并冗余章节 | `phase-1-analysis/prd-template.md` |
| 需求对齐检查表中的某些项总是通过 | 提高检查标准或移除无效检查项 | `phase-1-analysis/requirement-alignment-checklist.md` |

#### Module 8 → Phase 2（规划设计）

| 分析发现 | 可能的优化操作 | 影响文件 |
|---------|--------------|---------|
| 设计审查清单中的某些检查项从未发现问题 | 修订检查项的判断标准或移除 | `phase-2-planning/design-review-checklist.md` |
| 架构模板中的某些章节在不同项目中差异很大 | 增加模板的灵活性或拆分为多个变体 | `phase-2-planning/architecture-template.md` |
| 规划流程中的某些步骤耗时过长 | 优化步骤或提供并行执行选项 | `phase-2-planning/plan-template.md` |

#### Module 8 → Phase 3（开发实施）

| 分析发现 | 可能的优化操作 | 影响文件 |
|---------|--------------|---------|
| TDD 铁律（IL-2）在某些场景中效果评分低 | 分析原因，可能需要补充特定场景的 TDD 指引 | `phase-3-implementation/tdd-guide.md` |
| 代码审查指南中的某些规则总是被忽略 | 修订规则描述或降低优先级 | `phase-3-implementation/code-review-guide.md` |
| 系统化调试流程在某些类型的 Bug 中效果不佳 | 补充特定类型 Bug 的调试策略 | `phase-3-implementation/systematic-debugging-guide.md` |

#### Module 8 → Phase 4（知识沉淀）

| 分析发现 | 可能的优化操作 | 影响文件 |
|---------|--------------|---------|
| AI 错误笔记本中的某些分类从未被使用 | 简化分类体系 | `phase-4-knowledge/ai-error-notebook-template.md` |
| 项目上下文模板中的某些字段总是空白 | 移除冗余字段或改为可选 | `phase-4-knowledge/project-context-template.md` |
| 回顾模板中的某些问题总是得到相同的回答 | 修订问题或增加追问机制 | `phase-4-knowledge/retrospective-template.md` |

#### Module 8 → Phase 5（交付标准）

| 分析发现 | 可能的优化操作 | 影响文件 |
|---------|--------------|---------|
| 交付检查清单中的某些项从未不通过 | 提高检查标准或移除无效检查项 | `phase-5-delivery/delivery-checklist.md` |
| 交付物模板中的某些章节在不同项目中差异很大 | 增加模板灵活性 | `phase-5-delivery/deliverables-template.md` |

#### Module 8 → 核心模块（铁律、反模式、工作流）

| 分析发现 | 可能的优化操作 | 影响文件 |
|---------|--------------|---------|
| 某条铁律在所有项目中触发率都很低 | 评估是否需要保留或修订（铁律移除需 MAJOR 版本变更） | `core/iron-laws.md` |
| 某些反模式从未被触发 | 评估是否仍然 relevant，可能移除 | `core/anti-patterns.md` |
| 工作流总览中的某些阶段耗时过长 | 优化阶段定义或调整门控条件 | `core/workflow-overview.md` |

### 4.3 反哺流程

Module 8 的分析结果反哺其他模块的具体流程：

```
聚合分析报告完成
    │
    ├── 识别出需要优化的规则
    │       │
    │       ├── 确定影响的模块（Phase 0-5 或核心模块）
    │       ├── 确定影响的文件（SKILL.md、模板文件、指南文件等）
    │       └── 制定具体的修改方案
    │
    ├── 评估变更影响
    │       │
    │       ├── 该变更是否影响其他模块？
    │       ├── 该变更是否需要 MAJOR 版本变更？
    │       └── 该变更的回滚方案是什么？
    │
    ├── 执行变更
    │       │
    │       ├── 修改受影响的 Markdown 文件
    │       ├── 更新版本号和变更日志
    │       └── 归档旧版本文件
    │
    └── 验证变更效果
            │
            ├── 在后续迭代中使用新版本
            ├── 收集新的约束追溯数据
            └── 对比优化前后的数据指标
```

## 五、执行建议

### 5.1 启动建议

- **第一次使用 Module 8 时**，不需要等待历史数据。从当前迭代开始收集约束追溯数据即可。
- **前 3 个项目或 5 次迭代**，只做数据采集和迭代快照，不做聚合分析和优化决策。
- **数据积累充足后**，执行第一次聚合分析，建立基线数据。

### 5.2 常规节奏

| 活动 | 频率 | 说明 |
|------|------|------|
| 约束追溯记录 | 每个功能/模块完成后 | 即时记录，不得延迟 |
| 迭代快照 | 每次迭代/项目完成后 | 汇总本迭代数据 |
| 聚合分析 | 每季度一次 | 跨项目/跨迭代分析 |
| 框架优化 | 每半年一次 | 基于聚合分析结果优化 |
| 优化效果验证 | 优化后下一个季度 | 对比优化前后数据 |

### 5.3 注意事项

- 优化应渐进式推进，避免一次性大规模变更
- 铁律的修订和移除需要特别谨慎，建议先标记为"待观察"状态
- 每次优化都应有明确的验证指标和验证周期
- 优化效果不佳时，应及时回滚并分析原因
- 保持变更日志的完整性和可追溯性
