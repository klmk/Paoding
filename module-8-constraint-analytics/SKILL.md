# Module 8: 约束有效性分析

## 概述

框架中定义了大量规则——铁律、红旗、反模式、补强规则、流程规则、模板规则等。但这些规则到底有没有用？哪些规则在实际开发中被频繁触发并产生了正面效果？哪些规则从未被触发，可能只是增加了框架的复杂度？

Module 8 的核心使命是：**用真实项目的数据反向验证框架里每条规则的有效性**，通过数据驱动而非拍脑袋的方式，持续优化框架质量。

**核心理念：** 框架规则不是教条，而是工具。工具好不好用，要用数据说话。每条规则要么被数据证明有价值，要么被数据证明需要改进或移除。

**适用场景：**
- 每个功能/模块完成后，需要记录约束追溯数据
- 每次迭代/项目完成后，需要生成迭代快照
- 跨多个项目/迭代后，需要执行聚合分析
- 框架版本迭代时，需要基于数据决定规则的保留/修订/移除

---

## 核心概念

### 概念 1：约束追溯（Constraint Trace）

每个产出物完成后，记录该产出物触发了哪些约束、约束来自哪个文件、触发原因是什么、效果如何。

```
产出物 → 触发了哪条约束 → 约束来源文件 → 触发原因 → 效果评估
```

约束追溯是整个分析体系的数据基础。没有追溯数据，就没有分析依据。

### 概念 2：迭代快照（Iteration Snapshot）

每次迭代/项目完成后，将本迭代中所有约束追溯数据汇总为一份快照。快照包含：
- 约束调用统计（触发次数、触发率）
- 高频约束 TOP10
- 低频/未触发约束清单
- 本次迭代的发现和建议

迭代快照是聚合分析的输入数据。

### 概念 3：聚合分析（Aggregate Analysis）

跨项目/跨迭代统计约束频率，识别：
- **高价值约束**：高频触发 + 高效果评分
- **冗余约束**：低频或零触发
- **待改进约束**：高频但效果评分低

聚合分析是框架优化的决策依据。

### 概念 4：框架优化（Framework Optimization）

基于聚合分析的结果，对框架规则执行保留/修订/移除/新增操作，形成版本变更记录，并在后续迭代中再验证优化效果。

## 铁律（Iron Laws）

以下铁律在 Module 8 中具有最高约束力，不可违反、不可绕过、不可降级。

### 铁律 M8-1：每个产出物必须记录约束追溯

```
任何功能/模块完成后，必须生成约束追溯记录。
追溯记录必须包含：产出物、触发的约束、约束来源、触发原因、效果评估。
未记录追溯的产出物视为不完整。
```

- 追溯记录必须在产出物完成时立即填写，不得事后补写
- 追溯记录中的数据必须基于实际开发过程，不得编造或估算
- 追溯记录使用标准模板（参见 `constraint-trace-template.md`）

### 铁律 M8-2：追溯数据必须真实，不可编造

```
约束追溯中的每一条数据都必须来自真实的开发过程。
禁止为了"让数据好看"而编造、夸大或篡改追溯记录。
```

- 触发次数必须与实际开发过程一一对应
- 效果评估必须基于可观察的客观结果，不得主观臆断
- 如果某条约束确实未被触发，如实记录为"未触发"，不得强行关联

### 铁律 M8-3：分析必须跨项目聚合

```
单次迭代的追溯数据不足以得出可靠结论。
约束有效性分析必须基于至少 3 个项目或 5 次迭代的数据。
```

- 单次迭代的数据只能作为参考，不能作为优化决策的唯一依据
- 聚合分析时必须注明数据覆盖范围（项目数、迭代数、时间跨度）
- 数据量不足时，应标注"数据不足，建议继续积累"而非强行下结论

### 铁律 M8-4：优化决策必须有数据支撑

```
任何框架规则的保留、修订、移除决策，都必须基于聚合分析数据。
不得基于个人偏好、直觉或单一案例做出优化决策。
```

- 每个优化行动项必须关联具体的数据指标
- 优化建议必须说明：数据依据、预期效果、验证方式
- 优化后必须在后续迭代中验证效果，形成闭环

### 铁律 M8-5：框架优化必须形成闭环

```
框架优化不是一次性活动，而是持续循环：
数据采集 → 聚合分析 → 识别问题 → 优化规则 → 版本发布 → 再验证
```

- 每次优化后必须发布框架版本变更记录
- 优化效果必须在后续迭代中验证
- 验证结果反馈到下一轮分析，形成持续改进闭环

## 执行流程

### 步骤 1：约束追溯记录

**目标：** 在每个功能/模块完成后，记录约束追溯数据。

**执行要点：**
1. 功能/模块开发完成后，立即填写约束追溯记录
2. 逐条列出该产出物触发的所有约束
3. 为每条约束记录：来源文件、触发原因、效果评估
4. 效果评估从三个维度打分：是否避免了错误、是否提升了质量、是否节省了时间
5. 使用标准模板（`constraint-trace-template.md`）

**输出：** 约束追溯记录文件

### 步骤 2：迭代快照生成

**目标：** 每次迭代/项目完成后，汇总本迭代的约束调用数据。

**执行要点：**
1. 收集本迭代所有约束追溯记录
2. 统计每条约束的触发次数和触发率
3. 计算每条约束的平均效果评分
4. 生成高频约束 TOP10 和低频/未触发约束清单
5. 记录本次迭代的关键发现和建议
6. 导出 JSON 格式数据（便于后续聚合分析）
7. 使用标准模板（`iteration-snapshot-template.md`）

**输出：** 迭代快照文件（Markdown + JSON）

### 步骤 3：数据归档

**目标：** 将迭代快照归档到统一的数据存储中。

**执行要点：**
1. 将 JSON 格式的快照数据存入项目数据目录
2. 确保数据文件命名规范（如 `snapshot-{项目名}-{迭代号}-{日期}.json`）
3. 维护一个索引文件，记录所有快照的位置和基本信息
4. 确保数据可被后续聚合分析脚本读取

**输出：** 归档的快照数据文件

### 步骤 4：聚合分析

**目标：** 跨项目/跨迭代分析约束有效性。

**触发条件：** 累积至少 3 个项目或 5 次迭代的快照数据后执行。

**执行要点：**
1. 加载所有快照数据
2. 计算每条约束的跨项目触发率和平均效果评分
3. 按约束分类（铁律/红旗/反模式/补强规则/流程规则/模板规则）分别统计
4. 识别高价值约束（高频 + 高效果）和冗余约束候选（低频或零触发）
5. 分析高频但效果差的规则，提出修订建议
6. 使用标准模板（`aggregate-analysis-template.md`）

**输出：** 聚合分析报告

### 步骤 5：框架优化决策

**目标：** 基于聚合分析结果，制定框架优化方案。

**执行要点：**
1. 对每条约束做出决策：保留 / 修订 / 移除 / 新增
2. 使用优化决策矩阵（参见 `optimization-loop-guide.md`）
3. 每个决策必须关联具体的数据指标
4. 修订建议必须包含：修订内容、预期效果、验证方式
5. 移除建议必须包含：移除原因、风险评估、回滚方案
6. 新增建议必须包含：规则内容、适用场景、验证指标

**输出：** 框架优化行动项清单

### 步骤 6：版本发布与再验证

**目标：** 发布框架优化版本，并在后续迭代中验证效果。

**执行要点：**
1. 更新框架版本号（遵循语义化版本规则）
2. 编写版本变更记录（变更内容、数据依据、预期效果）
3. 更新受影响的 Markdown 文件（Phase 0-5 和 Module 7）
4. 在后续迭代中使用新版本的框架
5. 收集新版本的约束追溯数据
6. 对比优化前后的数据，验证优化效果
7. 将验证结果反馈到下一轮聚合分析

**输出：** 框架新版本 + 版本变更记录 + 优化效果验证报告

## 红旗列表（Red Flags）

以下是 AI 在执行 Module 8 时可能用来跳过流程的常见借口，以及对应的反驳：

| # | AI 的借口（红旗） | 反驳 |
|---|------------------|------|
| 1 | "追溯记录太麻烦了，影响开发效率" | 追溯记录是分析的数据基础。没有数据就没有分析，没有分析就没有优化。10 分钟的记录投入换来的是框架质量的持续提升。 |
| 2 | "这个功能没触发什么约束，没什么好记的" | "没触发约束"本身就是重要的数据。它说明这条约束在该场景下未被需要，这是识别冗余规则的关键信号。如实记录。 |
| 3 | "效果评估太主观了，随便打个分就行" | 效果评估必须基于可观察的客观结果。如果无法评估，标注为"无法评估"而非随便打分。主观评估比没有评估更危险。 |
| 4 | "只有一个项目的数据，先分析看看" | 单一项目的数据不具备统计意义。铁律 M8-3 要求至少 3 个项目或 5 次迭代。数据不足时标注"数据不足"，不要强行下结论。 |
| 5 | "这条规则我觉得没用，直接删了吧" | 优化决策必须有数据支撑。你的"觉得"不是数据。先积累追溯数据，用数据说话。 |
| 6 | "分析结果不太好看，我调整一下数据" | 铁律 M8-2 明确禁止编造或篡改数据。不好看的数据比好看的数据更有价值——它指出了真正需要改进的地方。 |
| 7 | "框架优化一次就够了，不需要反复做" | 框架优化是持续循环，不是一次性活动。业务场景在变化，规则的有效性也在变化。铁律 M8-5 要求形成闭环。 |
| 8 | "这些数据都是老项目的，没有参考价值" | 历史数据是趋势分析的基础。即使项目已经结束，其约束追溯数据仍然对框架优化有参考价值。 |
| 9 | "聚合分析太复杂了，我直接凭经验判断" | 经验判断是聚合分析的补充，不是替代。没有数据支撑的经验判断等同于拍脑袋。 |
| 10 | "优化后效果不明显，说明分析没用" | 效果不明显可能是因为：数据量不足、优化方向不对、验证周期不够。这些都是有价值的发现，应反馈到下一轮分析。 |

**如果出现以上任何一种情况，立即停止，回到对应步骤重新执行。**

## 约束分类体系

Module 8 对框架中的所有约束进行统一分类，便于聚合分析：

| 分类 | 说明 | 来源示例 |
|------|------|---------|
| **铁律** | 不可违反的最高约束力规则 | `core/iron-laws.md` 中的 IL-1 至 IL-6 |
| **红旗** | AI 跳过流程的常见借口及反驳 | 各模块 SKILL.md 中的红旗列表 |
| **反模式** | AI 开发中常见的合理化借口 | `core/anti-patterns.md` 中的 AP-R01 至 AP-F05 |
| **补强规则** | 增强某方面能力的附加规则 | 各模块中的补充约束、最佳实践 |
| **流程规则** | 定义执行步骤和顺序的规则 | 各模块的执行流程、门控条件 |
| **模板规则** | 模板中定义的填写要求 | 各模板中的必填项、格式要求 |

## 与其他模块的关系

Module 8 是框架的"元分析"模块，其分析结果反哺整个框架：

```
Module 8 分析结果
    ├── Phase 0 双向画像 → 优化画像维度和对齐策略
    ├── Phase 1 需求分析 → 优化需求分析规则和模板
    ├── Phase 2 规划设计 → 优化设计审查规则和模板
    ├── Phase 3 开发实施 → 优化 TDD 规则和代码审查标准
    ├── Phase 4 知识沉淀 → 优化知识管理规则和模板
    ├── Phase 5 交付标准 → 优化交付检查规则和模板
    └── Module 7（如有）→ 优化模块间协作规则
```

Module 8 不替代任何现有模块，而是为所有模块提供数据驱动的优化依据。

## 注意事项

- Module 8 的数据采集应尽量自动化，减少人工干预带来的偏差
- 追溯记录的粒度应在"功能/模块"级别，不需要细化到每一行代码
- 效果评估采用 1-5 分制，评分标准应保持一致
- 聚合分析应定期执行（建议每季度一次），避免数据过时
- 框架优化应渐进式推进，避免一次性大规模变更
