# Phase 5: 交付标准

## 概述

Phase 5 是整个开发流程的最终质量关卡。本阶段以 Phase 4 知识沉淀的输出为基础，通过**完整性检查、质量终审、追溯性验证、使用者验收**四道防线，确保交付物满足全部质量标准，可以安全移交。

**核心理念：** 交付不是"把东西交出去"，而是"把经得起检验的东西交出去"。不完整不交付、未验证不交付、无文档不交付——这是交付阶段的底线。

**适用场景：**
- Phase 4（知识沉淀）已通过 Gate 4 门控，进入交付阶段
- 需要对全部交付物进行系统性终审
- 需要向使用者进行正式的功能演示和验收

---

## 阶段目标

确保所有交付物**完整、达标、可移交**。

"可交付"的定义：
- **完整性**：所有计划内的代码、文档、知识资产均已产出，无遗漏
- **质量达标**：全量测试通过，无 Critical/Important 未修复问题，代码规范合规
- **文档齐全**：所有阶段文档格式规范、内容完整、可独立阅读
- **可追溯**：每段代码可追溯到 PRD 需求，形成完整的需求→AC→代码→测试链条
- **已验收**：使用者已确认功能满足需求，交付物可移交

---

## 入口条件

进入 Phase 5 之前，必须满足以下**全部**条件：

| # | 条件 | 验证方式 |
|---|------|---------|
| 1 | Phase 4 全部出口条件已满足 | Gate 4 门控检查通过 |
| 2 | 知识沉淀已完成 | 决策记录、经验教训、API 文档已产出 |
| 3 | 所有代码审查已完成 | 无未解决的 Critical/Important 问题 |
| 4 | 全量测试通过 | 新鲜的测试通过证据 |
| 5 | 需求追溯矩阵完整 | 每条需求有对应的实现和测试 |

**如果以上任何条件未满足，不得进入 Phase 5。** 必须回退到对应阶段补齐缺失项。

---

## 铁律（Iron Laws）

以下铁律在 Phase 5 中具有最高约束力，不可违反、不可绕过、不可降级。

### 铁律 P5-1：不完整不交付

```
交付清单上的每一项都必须完成。部分完成 = 未完成。
不得以"核心功能已完成，次要功能后续补充"为由交付不完整的系统。
```

- 交付清单对照 [`delivery-checklist.md`](./delivery-checklist.md) 逐项核查
- 任何一项标记为"不通过"，整个交付即暂停
- 如需部分交付，必须经使用者显式同意，并记录未交付项及计划完成时间

### 铁律 P5-2：未验证不交付

```
每一项交付物都必须有对应的验证证据。
"应该没问题"不是验证证据。"之前测过了"不是新鲜证据。
```

- 代码：必须有新鲜的测试通过报告
- 文档：必须经过格式和内容审查
- 追溯性：必须逐条核实需求→AC→代码→测试的完整链条
- 验证证据必须在本次交付流程中生成

### 铁律 P5-3：无文档不交付

```
代码没有文档，等于代码不存在。
没有文档的交付物不是交付物，是技术债务。
```

- 所有阶段文档必须齐全（PRD、架构文档、开发计划、API 文档、部署文档、用户手册）
- 文档必须格式规范、可独立阅读、与代码保持一致
- 缺失任何阶段文档，交付即暂停

### 铁律 P5-4：使用者未确认不交付

```
交付的最终决定权在使用者手中。
AI 不得单方面宣布"交付完成"，必须有使用者的显式确认。
```

- 必须向使用者演示功能，展示验证证据
- 使用者有权提出修改意见，AI 必须响应
- 使用者确认后方可进入交付打包环节

## 执行流程

### 步骤 1：交付物完整性检查

**目标：** 对照交付清单逐项核查，确认所有交付物均已产出。

**执行要点：**
1. 读取交付物清单模板（[`deliverables-template.md`](./deliverables-template.md)）
2. 逐项核查以下内容：
   - **代码文件**：源码、测试代码、配置文件、构建脚本是否齐全
   - **文档文件**：PRD、架构文档、开发计划、API 文档、部署文档、用户手册是否齐全
   - **知识文件**：project-context.md、AI 错题本、回顾报告、开发履历是否齐全
3. 对每一项标记状态：已完成 / 部分完成 / 未完成
4. 如有任何"部分完成"或"未完成"项，记录缺失内容并暂停交付
5. 生成完整性检查报告

**输出：** 交付物完整性检查报告（逐项状态）

### 步骤 2：代码质量终审

**目标：** 确认代码质量满足交付标准。

**执行要点：**
1. **全量测试**：运行完整测试套件，确认所有测试通过（无跳过、无忽略）
2. **覆盖率检查**：确认测试覆盖率满足预设标准
3. **代码规范检查**：运行 linting 工具，确认无规范违规
4. **Critical/Important 问题清零**：确认所有审查中发现的 Critical 和 Important 问题已修复
5. **敏感信息检查**：扫描代码中是否存在硬编码的密钥、密码、Token 等敏感信息
6. **构建验证**：执行完整的构建流程，确认构建成功
7. 生成代码质量终审报告

**输出：** 代码质量终审报告（测试结果、覆盖率、问题清单、构建状态）

### 步骤 3：文档完整性终审

**目标：** 确认所有阶段文档齐全且格式规范。

**执行要点：**
1. **文档齐全性检查**：确认以下文档全部存在且内容完整：
   - Phase 0：用户画像、AI 能力画像、对齐策略
   - Phase 1：PRD（需求规格文档）、需求追溯矩阵
   - Phase 2：架构文档、开发计划、设计评审记录
   - Phase 3：代码审查记录、变更日志、测试报告
   - Phase 4：决策记录（ADR）、经验教训报告、API 文档
2. **格式规范检查**：确认文档格式统一、结构清晰、可独立阅读
3. **一致性检查**：确认文档之间无矛盾（如 PRD 中的需求与架构设计一致）
4. **时效性检查**：确认文档反映的是最终状态（非过时版本）
5. 生成文档完整性终审报告

**输出：** 文档完整性终审报告（文档清单、状态、问题列表）

### 步骤 4：追溯性终审

**目标：** 确认每段代码可追溯到 PRD 需求，形成完整的追溯链条。

**执行要点：**
1. **正向追溯**（需求→代码）：
   - 从 PRD 中取出每条需求
   - 追踪到对应的 AC（验收标准）
   - 追踪到对应的实现代码（通过 AC ID）
   - 追踪到对应的测试用例（通过 AC ID）
2. **反向追溯**（代码→需求）：
   - 从代码库中取出每个模块/文件
   - 确认其对应的需求标识（AC ID）
   - 确认无"孤儿代码"（无法追溯到需求的代码）
3. **覆盖率检查**：确认每条需求都有对应的实现和测试
4. **标记异常**：对断裂的追溯链标记问题并记录
5. 生成追溯性终审报告

**输出：** 追溯性终审报告（需求→AC→代码→测试映射表、异常清单）

### 步骤 5：使用者验收

**目标：** 向使用者演示功能，确认需求满足。

**执行要点：**
1. **准备演示环境**：搭建可运行的环境，确保演示顺利进行
2. **功能演示**：按 PRD 中的用户场景逐个演示功能
3. **验证证据展示**：向使用者展示测试报告、追溯矩阵等验证证据
4. **问题收集**：记录使用者提出的问题和修改意见
5. **问题处理**：
   - 如使用者提出 Critical 问题，必须修复后重新验收
   - 如使用者提出 Suggestion，记录为后续优化项
6. **确认签字**：使用者确认交付物满足要求，记录确认状态

**输出：** 验收记录（演示记录、问题清单、使用者确认状态）

### 步骤 5.5：产品经理业务验收

**目标：** 从产品经理视角，以业务场景为驱动，系统性验证交付物在真实使用场景下的正确性和完整性。本步骤是对步骤 5（使用者验收）的补充，侧重于业务层面的深度验证。

**为什么需要本步骤：** 使用者验收往往关注"功能有没有"，而产品经理验收关注"功能对不对、场景全不全、体验好不好"。两者视角互补，缺一不可。

**执行要点：**

1. **验收准备**
   - 从 PRD 中提取所有 P0（必须有）和 P1（应该有）需求的验收标准（AC）
   - 为每条 AC 准备测试数据：正常数据 + 边界数据 + 异常数据
   - 按用户旅程编排验收顺序，模拟真实使用流程（而非孤立地逐个功能点测试）

2. **功能验收**
   - 按 AC 逐条验证，记录结果：通过 / 不通过 / 部分通过
   - 对"不通过"项，记录具体表现与期望表现的差异
   - 对"部分通过"项，明确缺失的部分
   - 特别关注：边界条件处理、异常输入处理、权限控制

3. **集成验收（跨系统场景）**
   - 验证跨系统完整链路（如 ERP 单据审核 → 推送至 MES → MES 确认接收）
   - 验证异常场景：
     - 下游系统（MES）宕机时，上游（ERP）怎么处理？是否重试？是否通知用户？
     - 网络超时怎么处理？是否有超时机制和重试策略？
     - 数据格式不匹配怎么处理？是否有数据校验和错误提示？
   - 验证数据一致性：多系统间的核心数据是否一致（抽检不少于 10 条）

4. **回归验收**
   - 新功能上线后，原有功能是否正常
   - 特别关注：接口代理/替换场景下，原有功能是否受影响（如 C# 事件代理到 Python 后，原有 C# 功能是否正常）

5. **验收结论**
   - 全部 P0 AC 通过 → 验收通过
   - 存在 P0 AC 不通过 → 打回开发，修复后重新执行本步骤
   - P1 AC 不通过 → 记录为已知问题，协商处理计划（修复/延期/接受）

**输出：** 产品经理业务验收报告（AC 验证结果表、集成验收记录、回归验证记录、验收结论）

### Gate 4.5：验收就绪预检

**目标：** 在正式进入 Phase 5 使用者验收之前，先进行内部的验收就绪预检，确保不会在正式验收时被打回。

**执行要点：**

1. **AC 完整性预检**：从 AIS 中提取所有 P0 和 P1 验收标准，确认每条 AC 有对应的实现代码和测试证据
2. **回归预检**：运行全量测试套件，确认全部通过（新鲜证据，不得复用历史结果）
3. **缺陷清零预检**：确认所有 Critical 和 Important 缺陷已修复，无遗留
4. **废弃代码预检**：确认所有废弃接口有下线计划（参见 IL-6 接口生命周期管理），无"僵尸代码"
5. **文档同步预检**：确认代码变更已同步到所有相关文档（API 文档、架构文档、AIS）
6. **形式化测试预检**：确认形式化测试中的 Critical 问题已全部修复

**预检结果处理：**
- 全部通过 → 进入步骤 6（交付打包）
- 存在不通过项 → 回退到对应阶段修复，修复后重新预检

**输出：** 验收就绪预检报告（逐项检查结果）

---

### 步骤 6：交付打包

**目标：** 将所有交付物整理为可移交的交付包。

**执行要点：**
1. **代码打包**：确保代码仓库干净（无临时文件、无调试代码）
2. **文档打包**：整理所有文档到统一的文档目录
3. **知识库打包**：整理 project-context.md、AI 错题本等知识资产
4. **环境说明**：编写运行环境、依赖版本、构建步骤说明
5. **已知限制**：整理已知的限制和待办事项
6. **交付清单**：生成最终的交付物清单（使用 [`deliverables-template.md`](./deliverables-template.md)）
7. 将交付包移交使用者

**输出：** 完整的交付包（代码 + 文档 + 知识库 + 环境说明）

## 红旗列表（Red Flags）

以下是 AI 在 Phase 5 中可能用来跳过流程的常见借口，以及对应的反驳：

| # | AI 的借口（红旗） | 反驳 |
|---|------------------|------|
| 1 | "核心功能都好了，边角功能以后再补" | 部分交付必须经使用者显式同意。未经同意的部分交付，等于未交付。参见铁律 P5-1。 |
| 2 | "之前测试都通过了，不用再跑一遍" | 交付前的全量测试是最终质量保障。环境变化、依赖更新都可能导致历史通过的测试失败。参见铁律 P5-2。 |
| 3 | "文档差不多就行了，代码才是重点" | 没有文档的代码不可维护、不可移交、不可接手。文档缺失是交付的硬性阻塞项。参见铁律 P5-3。 |
| 4 | "我觉得已经满足需求了，可以直接交付" | 交付的最终决定权在使用者。AI 的自我评估不能替代使用者的验收确认。参见铁律 P5-4。 |
| 5 | "追溯性检查太费时间，代码能跑就行" | 追溯性是质量可审计的基础。没有追溯性，就无法确认代码是否完整实现了需求。 |
| 6 | "使用者说不用演示，直接交就行" | 使用者可能不了解演示的价值。AI 应主动建议演示，并说明演示可以发现潜在问题。 |
| 7 | "这个已知问题不影响使用，可以先交付" | 已知问题必须记录在交付清单中，并说明影响范围和计划修复时间。隐瞒已知问题是不诚信行为。 |
| 8 | "交付包就是代码仓库，不需要额外整理" | 交付包应包含代码、文档、知识库、环境说明的完整集合。仅交付代码仓库是不完整的交付。 |

**如果出现以上任何一种情况，立即停止，回到对应步骤重新执行。**

## 出口条件（Gate 5）

Phase 5 完成并通过门控检查前，必须满足以下**全部**条件：

- [ ] 交付物完整性检查全部通过（无不通过项）
- [ ] 全量回归测试通过（新鲜证据）
- [ ] 代码无 Critical/Important 未修复问题
- [ ] 所有阶段文档齐全且格式规范
- [ ] 追溯性链条完整（需求→AC→代码→测试）
- [ ] 使用者已确认验收（显式确认记录）
- [ ] 产品经理业务验收通过（P0 AC 100% 通过，集成验收无阻断问题）
- [ ] 交付包已整理完成

### 门控类型

**硬性门控（Hard Gate）** — 必须有使用者的显式验收确认，不得有未解决的 Critical 问题。

---

## 注意事项

- Phase 5 是整个流程的最终关卡，发现问题时的回退成本最高。因此，前置阶段的质量把控至关重要
- 交付不是终点。交付后的维护、迭代和知识更新同样重要
- 交付包应具备"自解释"能力——一个新人拿到交付包后，应能通过文档独立理解项目
- AI 错题本和开发履历是框架的核心知识资产，必须包含在交付包中
- 如在交付阶段发现严重问题（如需求未满足、架构缺陷），必须回退到对应阶段修复，不得在交付阶段"打补丁"
